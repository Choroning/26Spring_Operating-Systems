## 1. 배경과 임계 구역 문제

### 1.1 협력 프로세스와 데이터 불일치

**협력 프로세스(cooperating process)** 란 다른 프로세스의 실행에 영향을 주거나 영향을 받을 수 있는 프로세스를 말합니다. 협력에는 **공유 상태**가 필요하고, 다음과 같은 방식으로 교환됩니다:

- **공유 메모리** — 여러 프로세스가 같은 메모리 영역을 매핑합니다.
- **메시지 전달** — 프로세스들이 명시적으로 메시지를 주고받습니다.
- **논리 주소 공간 공유** — 한 프로세스 내의 여러 스레드가 모두 같은 주소 공간을 공유합니다.

근본 문제: **공유 데이터에 대한 동시 접근은 비일관적인 결과를 만들 수 있다**는 것입니다. 접근을 정렬하는 규율이 없으면, 공유 데이터의 최종 상태는 명령어가 어떻게 인터리빙되는가에 따라 달라집니다.

> **이 장 전체의 토대가 되는 이유**: 락, 세마포어, 모니터 등 앞으로 배울 모든 동기화 도구는 단 하나의 이유로 존재합니다. 공유 데이터에 대한 동시 접근에 **질서 있는 실행**을 부여하여, 정확성이 더 이상 운에 좌우되지 않게 만들기 위해서입니다.

### 1.2 레이스 컨디션

**레이스 컨디션(race condition)** 이란 둘 이상의 프로세스(또는 스레드)가 공유 데이터에 동시에 접근할 때, **접근이 인터리빙되는 순서에 따라 최종 결과가 달라지는** 현상입니다.

**고전적인 예시.** 생산자 스레드는 `count++` 를, 소비자 스레드는 `count--` 를 실행하며 초깃값은 `count = 5`입니다. `++` 와 `--` 는 모두 **load → modify → store** 로 분해되며, 운 나쁜 인터리빙은 둘 중 하나의 갱신을 잃어버립니다:

```text
T0: producer  register1 = count        {register1 = 5}
T1: producer  register1 = register1+1  {register1 = 6}
T2: consumer  register2 = count        {register2 = 5}
T3: consumer  register2 = register2-1  {register2 = 4}
T4: producer  count = register1        {count = 6}
T5: consumer  count = register2        {count = 4}  ← 잘못된 값!
```

올바른 최종 값은 `count = 5` 입니다(증가 1회 + 감소 1회). 인터리빙에 따라 결과는 **4** (소비자가 마지막으로 씀) 또는 **6** (생산자가 마지막으로 씀)이 될 수 있습니다 — 타이밍에 의존하기 때문에 재현하기 매우 어려운 off-by-one 버그입니다.

**해결책**: 공유 데이터를 건드리는 코드 영역 — **임계 구역(critical section)** — 을 식별하고 **보호**하여 한 번에 한 스레드만 들어갈 수 있게 합니다.

> **레이스 컨디션이 실무에서 그토록 두려운 이유**: 타이밍에 의존하기 때문에 개발 중에는 모든 테스트를 통과할 수 있지만, 운영 환경의 부하 상황에서만 드러나는 경우가 많습니다. 1985–87년 Therac-25(방사선 치료기)에서 환자를 사망에 이르게 한 결함도 운영자 입력 처리에서의 레이스 컨디션이었습니다. 동기화는 단순한 스타일의 문제가 아니라 **정확성**의 문제입니다.

### 1.3 커널에서의 레이스 컨디션

커널은 모든 프로세스를 위해 광범위한 공유 상태를 다루므로, 커널의 레이스 컨디션은 시스템 전체를 망가뜨릴 수 있습니다.

- **`fork()`** 는 `next_available_pid` 카운터를 유지합니다. 두 `fork()` 호출이 동시에 같은 값을 읽으면 **두 프로세스가 같은 PID를 갖게** 됩니다.
- **열린 파일 테이블** 수정은 여러 프로세스가 동시에 파일을 열 때 충돌할 수 있습니다.
- **메모리 할당자** 의 자료 구조(free list, zone count)가 깨질 수 있습니다.
- **프로세스 리스트** 갱신(`fork()`/`exit()`)과 공유 자료 구조를 건드리는 **인터럽트 핸들러** 도 같은 위험에 노출됩니다.

커널 개발자는 이러한 레이스를 **체계적으로 방지** 해야 하며, 사용자 공간 프로그래머는 그 규율의 혜택을 누립니다.

### 1.4 임계 구역 문제

각 프로세스는 임계 구역을 중심으로 다음과 같이 구조화됩니다:

```c
while (true) {
    /* entry section */
        /* critical section */
    /* exit section */
        /* remainder section */
}
```

임계 구역 문제에 대한 올바른 해법은 **세 가지 요건**을 모두 만족해야 합니다:

1. **상호 배제(Mutual Exclusion)** — 어느 시점에도 임계 구역 안에는 한 프로세스만 있어야 합니다.
2. **진행(Progress)** — 임계 구역에 아무도 없을 때 들어가고자 하는 프로세스가 있다면, 어느 프로세스가 들어갈지 결정하는 일이 무한정 미뤄져서는 안 됩니다. (잔여 구역에 있는 프로세스는 누구의 진입도 막을 수 없습니다.)
3. **한정 대기(Bounded Waiting)** — 어떤 프로세스가 진입을 요청한 뒤, 그 요청이 받아들여지기 전까지 다른 프로세스가 임계 구역에 진입할 수 있는 횟수에 **상한**이 있어야 합니다.

> **각 요건이 막아 주는 것**:
> - **상호 배제**가 없으면 → 레이스 컨디션이 다시 등장합니다.
> - **진행**이 없으면 → 임계 구역이 비어 있는데도 시스템이 멈추는 설계상의 데드락이 가능합니다.
> - **한정 대기**가 없으면 → 기아(starvation) — 다른 프로세스들이 계속 들어가는 동안 특정 프로세스만 영영 못 들어갈 수 있습니다.

### 1.5 선점형 vs 비선점형 커널

커널은 자신의 스레드가 커널 모드에서 실행 중일 때 선점될 수 있는지 여부를 결정해야 합니다.

- **비선점형 커널** — 커널 모드 코드는 블록되거나, 종료하거나, 양보할 때까지 계속 실행됩니다. 단일 CPU에서는 다른 커널 스레드가 끼어들 수 없으므로 커널 데이터에 대한 레이스 컨디션이 **본질적으로 회피**됩니다. 단점: **응답성이 떨어집니다** (높은 우선순위의 사용자 작업이 커널이 끝날 때까지 기다려야 할 수 있습니다).
- **선점형 커널** — 커널 모드 코드 자체도 선점될 수 있습니다. 응답성이 좋고 실시간 작업에 적합하지만, 특히 **SMP** 환경에서는 레이스 컨디션을 막기 위해 신중하게 설계해야 합니다.

두 가지 실무적 포인트:
- **단일 프로세서** 시스템은 임계 구역 동안 **인터럽트를 비활성화**하는 것만으로 상호 배제를 저렴하게 구현할 수 있습니다.
- **다중 프로세서** 시스템은 그렇게 할 수 없습니다 — 모든 코어에서 인터럽트를 끄는 비용이 크기 때문에 — 다른 메커니즘(메모리 배리어, 원자 명령, 락)이 필요합니다.

> **역사적 메모**: Linux는 커널 2.6(2003)까지 비선점형이었으며, 이후 완전 선점형으로 바뀌었습니다. 현대 Linux, Windows, macOS는 모두 다수의 코어로 확장 가능하도록 설계된 선점형 커널입니다.

## 2. Peterson의 해법

### 2.1 알고리즘

Peterson의 해법은 **두 프로세스(P0, P1)** 를 위한 임계 구역 문제의 고전적인 **소프트웨어 전용** 해법입니다. 다음 두 공유 변수를 사용합니다:

```c
int     turn;       /* 누구의 차례인지 */
boolean flag[2];    /* flag[i] = true 이면 프로세스 i가 들어가고 싶어 함 */
```

각 프로세스 Pi (파트너는 Pj, `j = 1 - i`)는 다음을 실행합니다:

```c
while (true) {
    flag[i] = true;        /* 들어가고 싶다고 알림 */
    turn = j;              /* 양보: 너 먼저 */
    while (flag[j] && turn == j)
        ;                  /* 상대도 들어가려 하고, 상대 차례라면 대기 */

    /* critical section */

    flag[i] = false;       /* 끝났음을 알림 */

    /* remainder section */
}
```

직관: **`flag[i]` 는 "들어가고 싶음"** 을, **`turn` 은 "당신 먼저"** 를 의미합니다. 프로세스는 상대가 들어가고 싶지 않거나(`flag[j] == false`) 상대가 이미 양보했을 때(`turn == i`) 임계 구역에 들어갑니다.

### 2.2 정확성 증명

**1. 상호 배제.** Pi가 진입 구역의 루프를 빠져나가려면 `flag[j] == false` 이거나 `turn == i` 여야 합니다. 만약 두 프로세스가 동시에 임계 구역에 있다면 `flag[0] == flag[1] == true` 이므로 둘 다 `turn` 으로만 빠져나간 셈입니다. 그런데 `turn` 은 0 또는 1 중 하나의 값만 가지므로, `i = 0` 과 `i = 1` 둘 다일 수는 없습니다. 모순 → 상호 배제가 성립합니다.

**2. 진행.** Pj가 잔여 구역에 있다면 `flag[j] == false` 이므로 Pi는 곧장 진입합니다. Pj가 `flag[j] = true` 로 설정하고 진입 구역에 있다면 `turn` 이 결정합니다 — `turn` 은 단일 값을 가지므로 둘 중 정확히 한 명이 들어갑니다.

**3. 한정 대기.** Pi가 대기 중이라고 합시다. Pj가 임계 구역을 빠져나오면서 `flag[j] = false` 로 설정하면 Pi가 들어갑니다. Pj가 다시 들어가고 싶다면 먼저 `turn = i` 로 설정해야 하므로 Pj보다 Pi가 먼저 들어갑니다. Pj는 Pi보다 **최대 1번** 만 새치기할 수 있습니다 → **상한 = 1**.

> **실용적으로 쓰지 않는데도 Peterson의 해법이 유명한 이유**: 공유 메모리에 대한 **일반적인 load/store만으로** — 이론적으로는 특별한 하드웨어 명령 없이 — 상호 배제를 달성할 수 있음을 증명합니다. 그 이론적 명료함 덕분에 모든 OS 교재에 등장합니다.

### 2.3 현대 아키텍처에서의 한계

Peterson의 정확성 증명은 명령이 **프로그램 순서대로** 실행되고, 어떤 쓰기가 다른 CPU에 즉시 보임을 가정합니다. **현대 프로세서와 컴파일러는 두 가정을 모두 위반합니다** — 성능을 위해 독립적인 load/store를 일상적으로 **재배열**하며, 어떤 쓰기는 코어별 쓰기 버퍼에 잠시 머물다가 전역적으로 보이게 됩니다.

```text
/* Peterson 진입 구역 */
flag[i] = true;     /* ① */
turn = j;           /* ② */
```

만약 CPU가 ②를 ①보다 먼저 실행한다면, 두 프로세스 모두 진입 검사를 통과해 임계 구역에 동시에 들어갈 수 있습니다 — **상호 배제가 깨집니다**. 컴파일러의 재배열도 같은 효과를 낳을 수 있습니다(두 쓰기는 서로 무관한 메모리를 다루기 때문입니다).

*Silberschatz Figure 6.4 — Peterson 해법에서 명령어 재배열의 효과.*

**해결책**: 하드웨어와 컴파일러가 경계를 넘어 재배열하지 못하도록 **메모리 배리어**(다음 절)를 삽입합니다. 적절한 배리어가 있으면 Peterson 알고리즘은 현대 하드웨어에서도 올바르게 동작합니다 — 하지만 실무에서는 이런 복잡성을 숨기는 더 상위 수준의 기본 도구(락, 원자 변수)를 사용합니다.

> **요지**: 이상화된 메모리 모델에서 증명된 알고리즘이 실제 하드웨어에서는 처참하게 실패할 수 있습니다. 머신의 "메모리 모델" — 어떤 재배열을 허용하는가 — 은 이제 동기화 코드가 반드시 존중해야 하는 프로그래밍 모델의 일부입니다.

## 3. 동기화를 위한 하드웨어 지원

### 3.1 메모리 배리어 (Memory Fence)

시스템의 **메모리 모델** 은 메모리 연산이 여러 CPU 사이에서 어떻게 보이고 어떻게 정렬되는지에 대한 하드웨어의 보장을 기술합니다:

- **강한 정렬(strongly ordered)** — 한 프로세서의 쓰기가 다른 모든 프로세서에 **즉시**, 프로그램 순서대로 보입니다. (이해하기 쉽지만 느립니다.)
- **약한 정렬(weakly ordered)** — 쓰기가 **순서가 어긋나거나 지연된 채** 보일 수 있습니다. (빠릅니다 — 현대 멀티코어 설계의 주류, 예: ARM, 경우에 따른 x86)

**메모리 배리어(memory barrier)** 또는 **메모리 펜스(memory fence)** 는 **순서를 강제하는** 특수 명령입니다: 배리어 **이전** 의 모든 load/store는 배리어 **이후** 의 어떤 load/store보다 먼저 완료되어야 합니다(다른 프로세서의 관점에서).

```c
/* Thread 1 */                       /* Thread 2 */
while (!flag)                        x = 100;
    ;                                memory_barrier();
memory_barrier();                    flag = true;
print x;  /* 반드시 100을 출력 */
```

배리어가 없다면 Thread 2의 `x` 쓰기가 `flag` 쓰기보다 **늦게** 보일 수 있고, 그러면 Thread 1은 `flag == true` 인데 `x == 0` 을 볼 수도 있습니다. 배리어는 프로그램 순서가 전역에서도 유지되도록 강제합니다.

- 메모리 배리어는 주로 **커널과 라이브러리 작성자**가 다루는 **매우 저수준** 의 기본 도구입니다. 대부분의 응용 프로그래머는 내부적으로 필요한 배리어를 자동으로 삽입해 주는 락이나 원자 변수를 사용합니다.
- 적절한 배리어가 있으면 **Peterson의 해법도 현대 하드웨어에서 올바르게 동작합니다**.

> **유용한 멘탈 모델**: 각 CPU와 메인 메모리 사이에 "스토어 버퍼"가 있다고 생각하세요. 쓰기는 버퍼에 들어가서 어떤 순서로 빠져나갑니다. 메모리 배리어는 "다른 일을 하기 전에 내 스토어 버퍼를 비우고, 이후 연산이 이전 연산을 추월하지 못하게 하라"는 명령입니다.

### 3.2 test_and_set

`test_and_set` 은 메모리 셀의 이전 값을 반환하면서 무조건 `true` 를 쓰는 **원자(atomic)** 명령입니다 — 두 동작이 단일하고 분할 불가능한 연산으로 수행됩니다.

```c
boolean test_and_set(boolean *target) {
    boolean rv = *target;
    *target = true;
    return rv;             /* 이전 값 */
}
```

**`test_and_set` 으로 상호 배제 구현** (lock 초기화 = `false`):

```c
do {
    while (test_and_set(&lock))
        ;                  /* lock이 false였을 때까지 바쁜 대기 */

    /* critical section */

    lock = false;

    /* remainder section */
} while (true);
```

`test_and_set` 을 처음 호출한 스레드는 `false` 를 보고 `true` 로 바꾼 뒤 진입합니다. 이후의 모든 호출자는 `true` 를 보고 스핀합니다. 두 CPU가 정확히 같은 순간 `test_and_set` 을 호출하더라도 하드웨어가 이를 **직렬화** 합니다 — 단 한 명만 `false` 를 읽습니다.

### 3.3 compare_and_swap (CAS)

더 일반적인 원자 기본 연산: **compare-and-swap** 은 메모리 셀이 기대한 값을 가지고 있는지 원자적으로 확인하고, 그렇다면 새 값으로 교체합니다.

```c
int compare_and_swap(int *value, int expected, int new_value) {
    int temp = *value;
    if (*value == expected)
        *value = new_value;
    return temp;           /* 항상 원래 값을 반환 */
}
```

**CAS를 이용한 상호 배제** (lock 초기화 = 0; 0 = 자유, 1 = 점유):

```c
while (true) {
    while (compare_and_swap(&lock, 0, 1) != 0)
        ;                  /* lock이 0이 될 때까지 스핀 */

    /* critical section */

    lock = 0;

    /* remainder section */
}
```

- Intel x86에서는 **`lock cmpxchg`** 명령으로 구현됩니다 — `xchg` 변형 앞에 `lock` 접두사를 붙여 그동안 버스를 잡고 있게 합니다.
- CAS는 거의 모든 현대적인 락-프리(lock-free) 자료 구조(락-프리 큐, 위험 포인터, RCU 등)의 토대가 되는 기본 연산입니다.

> **CAS가 test_and_set보다 강력한 이유**: `test_and_set` 은 boolean에서만 동작하지만, CAS는 임의의 정수에 대해 동작하며 "내가 마지막으로 본 값과 여전히 같다면 갱신해라"라고 표현할 수 있습니다 — 한 스레드가 값을 읽고 새 값을 계산한 뒤, 그 사이 다른 누군가가 값을 바꾸지 않은 경우에만 commit하는 **낙관적 동시성(optimistic concurrency)** 의 토대입니다.

### 3.4 CAS — 한정 대기 보장

위의 단순한 CAS-스핀 락은 상호 배제는 보장하지만 **한정 대기는 보장하지 않습니다** — 운 나쁜 스레드가 계속 락을 빼앗길 수 있습니다. 다음 확장은 `waiting[]` 플래그 배열을 사용하여 모든 요청자가 다른 임계 구역 진입 횟수가 최대 `n - 1` 회 이내에 처리되도록 보장합니다:

```c
/* 공유: boolean waiting[n] = {false}, int lock = 0 */

while (true) {
    waiting[i] = true;                          /* 대기 중임을 표시 */
    key = 1;
    while (waiting[i] && key == 1)
        key = compare_and_swap(&lock, 0, 1);    /* 락 시도 */
    waiting[i] = false;

    /* critical section */

    j = (i + 1) % n;
    while ((j != i) && !waiting[j])
        j = (j + 1) % n;                        /* 다음 대기자 탐색 */

    if (j == i)
        lock = 0;                               /* 대기자 없음 */
    else
        waiting[j] = false;                     /* 다음 대기자에게 락 양도 */

    /* remainder section */
}
```

스레드가 임계 구역을 빠져나올 때 자기 다음 인덱스부터 시작해 `waiting[]` 을 **순환적으로** 스캔하여, 처음 발견된 대기자에게 락을 넘깁니다. 그래서 모든 요청자는 다른 진입을 최대 `n - 1` 회 본 뒤 반드시 처리됩니다 → **한정 대기**.

### 3.5 원자 변수

**원자 변수(atomic variable)** 는 CAS 위에 올라간 더 상위 수준의 추상으로, CAS 루프를 직접 작성하지 않고도 단일 정수의 레이스 없는 갱신을 제공합니다.

전형적인 CAS 기반 원자 증가:

```c
void increment(atomic_int *v) {
    int temp;
    do {
        temp = *v;
    } while (temp != compare_and_swap(v, temp, temp + 1));
}
```

다른 스레드가 그 사이 값을 바꾸었으면 CAS는 실패하고, 갱신된 값으로 루프가 다시 시작됩니다.

**Linux 커널 API**:

```c
atomic_t counter;
atomic_set(&counter, 5);                /* counter = 5  */
atomic_add(10, &counter);               /* counter = 15 */
atomic_sub(4, &counter);                /* counter = 11 */
atomic_inc(&counter);                   /* counter = 12 */
int val = atomic_read(&counter);        /* val     = 12 */
```

**중요한 주의**: 원자 변수는 **단일 변수** 의 갱신을 보호합니다. 여러 변수에 걸친 복합 불변식 — 예: 유한 버퍼의 `0 ≤ count ≤ n`, `in/out` 인덱스 — 은 여전히 임계 구역 전체를 보호하는 **뮤텍스** 또는 **세마포어** 가 필요합니다.

> **유용한 어림 규칙**: "카운터를 증가" 하거나 "플래그를 세팅" 하는 것이 전부라면 원자 변수를 쓰세요. 두 개 이상의 관련 필드를 **함께** 갱신하여 어떤 관찰자도 둘이 어긋나는 상태를 보지 못하게 해야 한다면 락을 쓰세요.

## 4. 뮤텍스 락

### 4.1 뮤텍스 락 — 기본

**뮤텍스 락**(**mut**ual **ex**clusion)은 가장 단순한 동기화 도구로, Boolean `available` 플래그와 두 연산으로 구성됩니다.

```c
acquire() {
    while (!available)
        ;                  /* 바쁜 대기 */
    available = false;
}

release() {
    available = true;
}
```

사용 패턴:

```c
acquire(lock);
    /* critical section */
release(lock);
```

교재 정의가 숨기는 두 가지 요건:

- `acquire()` 와 `release()` 자체가 **원자적** 이어야 합니다. 보통 `test_and_set` 또는 CAS 위에 구현됩니다.
- 락 변수는 **Boolean** 입니다 — 정확히 두 상태(점유 / 자유).

뮤텍스의 역할은 임계 구역이 마치 **고립되어 실행된 것처럼** 보이게 만드는 것입니다.

### 4.2 스핀락과 바쁜 대기

**스핀락(spinlock)** 은 `acquire()` 가 락이 풀릴 때까지 **바쁘게 대기(loop)** 하는 뮤텍스입니다 — 스레드를 잠재우지 않습니다.

| 속성 | 스핀락 | Sleeping 락 |
|----------|----------|---------------|
| 대기 방식 | **스핀** (루프) | **블록** (sleep) |
| 깨어나는 비용 | 없음 — 이미 실행 중 | 컨텍스트 전환 2회 (sleep + wake) |
| 적합한 경우 | 락을 **매우 짧게** 잡는 경우 | 락을 **오래** 잡는 경우 |
| 위험 | 단일 프로세서에서 CPU 낭비; 한 코어만 생산적으로 스핀 가능 | 락 획득 지연 |

**스핀락의 장점**:
- **컨텍스트 전환 오버헤드가 없습니다** — 락이 풀리면 스피닝 중인 스레드는 다음 루프 반복에서 곧장 봅니다.
- **멀티코어 시스템에서 특히 유용합니다**: 락을 코어 A가 잡고 있는 동안 코어 B는 생산적으로 스핀할 수 있고, A가 풀자마자 스케줄러 개입 없이 B가 잡을 수 있습니다.
- **어림 규칙**: 예상 점유 시간이 **컨텍스트 전환 2회보다 짧을 때** 스핀락을 사용합니다.

**스핀락의 단점**:
- CPU 사이클을 낭비합니다. 다른 유용한 일이 없는 **단일 프로세서**에서는 순수한 낭비입니다(스피너가 CPU를 점유하고 있는 동안 락 점유자는 풀 수 없습니다).
- 높은 **락 경합(lock contention)** — 같은 락을 동시에 차지하려는 스레드가 많을 때 — 은 성능을 파괴합니다.

**락 경합 용어**:
- **무경합(uncontended) 락** — 한 스레드가 시도해 곧바로 획득하는 경우.
- **경합(contended, blocked) 락** — 여러 스레드가 동시에 시도해 일부가 기다려야 하는 경우.

> **더 깊은 그림**: 스핀락 vs sleeping 락은 *시간-비용* 의 트레이드오프입니다. Sleep은 컨텍스트 전환 2회의 고정 비용을 지불하고, 스핀은 대기 시간에 비례하는 비용을 지불합니다. 더 작은 쪽이 이깁니다 — 그래서 커널은 매우 짧은 커널 데이터 임계 구역에는 스핀락을, 더 긴 사용자 작업에는 sleeping 락(뮤텍스)을 쓰는 경우가 많습니다.

## 5. 세마포어

### 5.1 정의 (wait / signal)

**세마포어(semaphore)** S는 두 **원자** 연산으로만 접근되는 정수 변수이며, Edsger Dijkstra가 처음 제안했습니다:

**`wait(S)`** — **P** (네덜란드어 *proberen*, "테스트하다") 라고도 합니다:

```c
wait(S) {
    while (S <= 0)
        ;          /* 바쁜 대기 */
    S--;
}
```

**`signal(S)`** — **V** (네덜란드어 *verhogen*, "증가시키다") 라고도 합니다:

```c
signal(S) {
    S++;
}
```

놓치기 쉬운 두 가지 원자성 요건:
- `S` 의 변경(`S--` 와 `S++`)은 read-modify-write로 분해되므로 원자적이어야 합니다.
- `wait` 에서는 **`S` 의 테스트와 감소가 함께 분할 불가능** 해야 합니다 — 그렇지 않으면 두 스레드가 모두 `S = 1` 을 보고 둘 다 감소시켜 `S = -1` 이 되어 한 명만 들어갈 수 있는 상황에 두 명이 들어갑니다.

### 5.2 세마포어 종류와 활용

**계수형 세마포어(counting semaphore).**
- 범위: 음이 아닌 임의의 정수.
- **N개의 동일 자원으로 이루어진 자원 풀** 을 모델링합니다.
- `wait(S)` 는 자원 하나를 획득(S 감소), `signal(S)` 는 자원 하나를 반환(S 증가)합니다.
- `S = 0` 이면 모든 자원이 사용 중 → 이후 `wait()` 호출은 블록됩니다.

**이진 세마포어(binary semaphore).**
- 범위: 0 또는 1.
- **뮤텍스 락과 거의 동일하게** 동작합니다 — 1로 초기화, `wait` = lock, `signal` = unlock.

**세마포어로 실행 순서 제어.** 세마포어는 한 문장이 다른 문장보다 먼저 실행되도록 강제할 수도 있습니다. `synch` 를 0으로 초기화하면:

```text
P1:                           P2:
    S1;                           wait(synch);
    signal(synch);                S2;
```

`P2` 는 `P1` 이 `S1` 실행 후 `signal(synch)` 를 호출할 때까지 `wait(synch)` 에서 블록됩니다. 따라서 P1과 P2가 어떤 순서로 스케줄되든 **S2는 반드시 S1 이후에 실행됩니다**.

> **뮤텍스 대비 핵심 일반화**: 계수형 세마포어는 **N개의 구별 불가능한 자원** (DB 커넥션, 스레드풀 슬롯, 빈 버퍼 슬롯 등)이 있을 때 자연스러운 도구입니다. 뮤텍스 = "한 번에 하나만 접근"; 계수형 세마포어 = "한 번에 최대 N개까지 접근".

### 5.3 바쁜 대기 제거 (대기 큐)

교재의 `wait()` 정의는 바쁜 대기를 하므로 임계 구역이 길면 낭비입니다. 실제 구현은 각 세마포어에 **대기 큐(waiting queue)** 를 붙입니다:

```c
typedef struct {
    int value;
    struct process *list;     /* 대기 큐 */
} semaphore;

wait(semaphore *S) {
    S->value--;
    if (S->value < 0) {
        /* 이 프로세스를 S->list에 enqueue */
        sleep();              /* 깨워질 때까지 블록 */
    }
}

signal(semaphore *S) {
    S->value++;
    if (S->value <= 0) {
        /* S->list에서 어떤 프로세스 P를 dequeue */
        wakeup(P);
    }
}
```

기억해 둘 두 가지 결과:

- `S->value` 가 **음수** 일 때, 그 **절댓값은 이 세마포어에 블록된 프로세스 수와 같습니다**.
- **FIFO** 대기 큐는 **한정 대기**를 보장합니다(블록된 프로세스가 순서대로 깨워집니다).

`wait` 가 먼저 감소시킨 뒤 검사하는 것에 주의하세요 — 위의 음수 값 부기와 일치합니다.

### 5.4 세마포어 구현의 원자성

`wait()` 와 `signal()` 이 올바르게 동작하려면 그 본체가 **원자적으로** 실행되어야 합니다.

- **단일 프로세서**: wait/signal 본체 동안 **인터럽트를 비활성화** 하면 됩니다. 저렴하고 효과적입니다.
- **다중 프로세서**: 모든 CPU에서 인터럽트를 끄는 것은 비싸고 지연에 좋지 않습니다. 구현은 **CAS** 또는 **스핀락** 을 내부에서 사용해 코어 간에 wait/signal 본체를 원자적으로 만듭니다.

이는 바쁜 대기를 **완전히 제거하지는 않습니다** — 응용의 잠재적으로 긴 임계 구역을 wait/signal 내부의 짧은 몇 명령으로 줄일 뿐입니다. 응용은 여전히 sleep/wakeup의 이점을 누립니다 — *자기* 임계 구역이 더 이상 바쁜 대기를 요구하지 않기 때문입니다.

> **머릿속에 둘 그림**: 동기화 기본 도구는 **스택**을 이룹니다. 맨 아래에 하드웨어(CAS, 펜스), 그 위에 작은 내부 스핀락, 그 위에 sleep/wake 인터페이스를 노출하는 세마포어. 각 층은 그 아래 층의 바쁜 대기를 숨깁니다.

## 6. 모니터

### 6.1 세마포어 사용의 어려움

세마포어는 강력하지만 **오류에 취약**합니다. 잘못 쓰면 특정 인터리빙에서만 드러나는 타이밍 버그가 생겨 디버깅이 끔찍해집니다. 흔한 실수:

- **순서 뒤바꿈** — `signal(mutex); CS; wait(mutex);` → 여러 프로세스가 동시에 임계 구역에 들어갈 수 있음 → **상호 배제 위반**.
- **wait 중복** — `wait(mutex); CS; wait(mutex);` → 두 번째 `wait` 에서 영원히 블록 → **데드락**.
- **wait 또는 signal 누락** — 임계 구역이 보호되지 않거나 아무도 깨우지 않음.

이 버그들은 세 가지 공통점이 있습니다: (1) 코드가 그럴싸해 보입니다, (2) 보통 테스트는 통과합니다, (3) 실패는 드물고 타이밍에 의존합니다. 해결책은 명시적 `wait`/`signal` 호출을 숨기는 **상위 수준 구문** — **모니터(monitor)** — 입니다.

### 6.2 모니터 — 구조

**모니터** 는 자신의 연산들에 대해 자동으로 상호 배제를 제공하는 **추상 자료형(ADT)** 입니다: 어느 시점에도 모니터 안에는 최대 한 스레드만 활성화됩니다.

```text
monitor monitor_name {
    /* 공유 변수 선언 (private) */

    function P1(...) { ... }
    function P2(...) { ... }
    ...
    function Pn(...) { ... }

    initialization_code(...) { ... }
}
```

*Silberschatz Figure 6.12 — 모니터의 도식적 구조.*

핵심 속성:
- **한 번에 한 스레드** 만 모니터 안에서 활성화될 수 있습니다 — 각 함수에 락/언락을 삽입하는 일은 컴파일러/런타임의 몫이며 프로그래머가 아닙니다.
- 내부 변수에는 **모니터의 함수를 통해서만** 접근할 수 있습니다 — 외부에서 손상시킬 방법이 없습니다.
- 프로그래머는 모니터 락을 위한 `wait(mutex)` / `signal(mutex)` 를 결코 작성하지 않습니다.

> **이것이 큰 도약인 이유**: 세마포어에서 상호 배제는 프로그래머의 **약속** 입니다("X를 만지는 모두는 먼저 wait를 호출해야 한다"). 모니터에서 상호 배제는 ADT의 **구조적** 속성입니다 — 모니터 함수를 통하지 않고는 X에 접근할 길이 없고, 모니터 함수는 자동으로 보호됩니다.

### 6.3 조건 변수

상호 배제만으로는 충분하지 않은 경우가 있습니다 — 모니터 안의 스레드가 다른 모니터 함수가 만들어 줄 **조건**을 기다려야 할 때가 있습니다. **조건 변수(condition variable)** 가 그 역할을 합니다:

```text
condition x, y;

x.wait();      /* 호출한 스레드가 x에서 대기 */
x.signal();    /* x에서 대기 중인 스레드 하나를 깨움 */
```

세마포어와의 두 가지 본질적인 차이:

- `x.signal()` 은 `x` 에서 대기 중인 스레드가 없으면 **아무 일도 하지 않습니다**. 세마포어의 `signal` 은 항상 값을 바꿉니다.
- `x.wait()` 를 호출한 스레드는 다른 스레드가 들어올 수 있도록 **모니터를 놓습니다**; 그렇지 않으면 모니터가 데드락에 빠집니다.

**`x.signal()` 이후 신호자 P와 깨어난 Q가 모두 모니터 안에 있고 싶을 때 어떻게 하는가?** 세 가지 정책이 있습니다:

- **Signal-and-wait** — P가 기다리고, Q가 즉시 실행됩니다. (원래 Hoare의 정의.)
- **Signal-and-continue** — P가 모니터를 떠날 때까지 계속 실행되고, 그 다음에 Q가 실행됩니다.
- **타협안** — P는 신호 직후 모니터를 떠나고, Q가 재개됩니다.

*Silberschatz Figure 6.13 — 조건 변수가 있는 모니터.*

> **조건 변수가 *왜* 있는가**: 조건 변수는 어떤 상태 변화(예: "버퍼가 비지 않음")를 기다리는 스레드가 **모니터를 정중히 양보** 할 수 있게 하고, 다른 스레드가 그 상태를 바꿨을 때 그 대기자를 **깨울** 수 있게 합니다. 조건 변수가 없다면 대기 스레드는 스핀(CPU 낭비)하거나 모니터 락을 잡은 채로 블록(데드락) 해야 합니다.

### 6.4 세마포어로 모니터 구현

모니터는 세마포어 위에 만들 수 있습니다. 다음은 **signal-and-wait** 버전입니다. 모든 외부 모니터 함수 `F` 는 다음과 같이 감싸집니다:

```text
semaphore mutex      = 1;     /* 모니터 진입을 보호 */
semaphore next       = 0;     /* 신호자가 여기서 대기 */
int       next_count = 0;     /* next에 블록된 신호자 수 */

/* 모든 외부 함수 F의 본체 */
wait(mutex);
    ... F의 본체 ...
if (next_count > 0)
    signal(next);             /* 대기 중인 신호자 깨우기 */
else
    signal(mutex);            /* 아니면 모니터 락 해제 */
```

각 조건 변수 `x` 는 자기만의 세마포어 `x_sem` 과 카운터 `x_count` 가 필요합니다:

```text
/* x.wait() */                          /* x.signal() */
x_count++;                              if (x_count > 0) {
if (next_count > 0)                         next_count++;
    signal(next);                            signal(x_sem);
else                                         wait(next);
    signal(mutex);                           next_count--;
wait(x_sem);                            }
x_count--;
```

부기는 복잡합니다 — 그것이 바로 *요점* 입니다: 모니터 추상은 이 복잡성을 프로그래머에게서 숨기고, 컴파일러/런타임이 한 번 정확히 구현하면 됩니다.

### 6.5 ResourceAllocator 모니터 예제

가장 **짧은** 시간을 요구하는 요청자에게 우선권을 주는 단일 자원 할당 모니터(`condition.wait(time)` 우선순위 형식 사용):

```text
monitor ResourceAllocator {
    boolean   busy;
    condition x;

    void acquire(int time) {
        if (busy)
            x.wait(time);     /* time = 우선순위; 작을수록 먼저 */
        busy = true;
    }

    void release() {
        busy = false;
        x.signal();           /* 가장 작은 time을 가진 대기자를 깨움 */
    }

    initialization_code() {
        busy = false;
    }
}
```

사용:

```text
R.acquire(t);
    ... 자원 사용 ...
R.release();
```

`x.wait(c)` 는 호출자에게 우선순위 값 `c` 를 저장합니다; `x.signal()` 은 저장된 값이 **가장 작은** 대기자를 재개합니다. 이것은 일부 모니터 구현이 지원하는 **conditional-wait** 변형입니다.

## 7. Liveness

**Liveness** 실패란 개별 프로세스가 살아 있지만 시스템이 그들이 원하는 일을 결코 진행하지 못하는 상태입니다. 상호 배제는 **안전성(safety)** 에 관한 것이고, liveness는 **결국 자기 차례를 얻는 것** 에 관한 것입니다.

### 7.1 데드락

**데드락(deadlock)** 은 어떤 프로세스 집합의 모든 프로세스가 **그 집합 안의 다른 프로세스만이** 일으킬 수 있는 사건을 기다리는 상태입니다. 누구도 진행할 수 없습니다.

두 세마포어 S, Q (둘 다 1로 초기화) 를 사용한 고전적인 두-프로세스 데드락:

```text
P0                          P1
wait(S);                    wait(Q);
wait(Q);                    wait(S);
...                         ...
signal(S);                  signal(Q);
signal(Q);                  signal(S);
```

P0이 S를 획득하고 `wait(Q)` 직전에 선점되며, P1이 Q를 획득하고 `wait(S)` 직전에 선점된다면 — **각자가 상대가 필요로 하는 것을 들고 있고**, 누구도 진행할 수 없습니다.

- 데드락은 주로 **자원 획득/해제** 주변에서 발생합니다.
- 데드락의 진단, 예방, 회피, 검출, 복구는 **8장(다음 주)** 에서 자세히 다룹니다.

> **데드락의 네 가지 필요 조건** (8장 미리보기): 상호 배제, 점유 대기, 비선점, 순환 대기. 이 중 어느 하나만 제거해도 데드락을 막을 수 있습니다 — 무엇을 봐야 하는지만 알면 유용한 틀이 됩니다.

### 7.2 우선순위 역전과 우선순위 상속

**우선순위 역전(priority inversion)** 은 미묘한 liveness 버그로, **낮은 우선순위** 프로세스가 간접적으로 **높은 우선순위** 프로세스를 **블록**시키는 현상입니다.

상황 — 우선순위 **L** (낮음) < **M** (중간) < **H** (높음):

1. **L** 이 세마포어 S를 획득합니다.
2. **H** 가 실행 가능 상태가 되어 S를 획득하려 시도 → 블록, L을 기다림.
3. **M** 이 실행 가능 상태가 됩니다. M의 우선순위가 L보다 높으므로 M이 **L을 선점합니다**.
4. M이 실행되는 동안 L은 S를 풀 수 없고, H는 진행할 수 없습니다.

순효과: H가 M보다 우선순위가 높음에도 불구하고 **H는 M에 의해 지연됩니다**.

**우선순위 상속 프로토콜(Priority Inheritance Protocol)**:
- L이 H가 기다리는 락을 들고 있는 동안 **L의 우선순위를 일시적으로 H 수준으로 끌어올립니다**.
- M은 더 이상 L을 선점할 수 없습니다 → L이 임계 구역을 빠르게 끝내고 S를 풉니다.
- 락을 풀고 나면 L은 원래 우선순위로 되돌아갑니다.

> **유명한 실제 사건 — Mars Pathfinder, 1997.** 화성 착륙선이 화성에서 알 수 없는 시스템 리셋을 반복했습니다. 원인은 **VxWorks RTOS** 의 우선순위 역전 버그였습니다: 낮은 우선순위의 기상 작업이 높은 우선순위의 버스 관리 작업이 필요로 하는 뮤텍스를 들고 있었고, 중간 우선순위의 통신 작업이 낮은 우선순위 작업을 계속 선점했습니다. 워치독 타이머가 만료되어 시스템을 리셋시켰습니다. JPL 엔지니어들은 문제의 뮤텍스에 **우선순위 상속을 활성화** 함으로써 원격으로 문제를 해결했습니다. 동기화는 실제 결과를 가져오는 진짜 공학입니다.

### 7.3 동기화 도구 평가

동기화 도구는 **락 경합** 정도에 따라 **성능**이 달라집니다:

| 경합 수준 | CAS 기반 (낙관적) | 락 기반 (비관적) |
|------------------|------------------------|-----------------------|
| **무경합** | 빠름 (약간 우세) | 빠름 |
| **중간** | **훨씬 빠름** — 컨텍스트 전환 없음 | 느림 — 컨텍스트 전환 비용 |
| **높음** | 느림 — CAS 재시도 다수 | 빠름 — 큐 기반 정렬 |

추상 수준별 분류:

| 수준 | 도구 | 주 사용자 |
|-------|-------|--------------|
| **저수준** | 하드웨어 (CAS, test_and_set, 펜스) | 주로 더 상위 도구의 *기반* 으로 사용됨 |
| **중간** | 뮤텍스 락, 세마포어 | 커널/응용에서 가장 흔히 사용 |
| **고수준** | 모니터, 조건 변수 | 언어 수준 지원 (Java, C# `lock` 등) |

> **실용적 시사점**: "최고의" 동기화 기본 도구는 없습니다. 올바른 선택은 **예상 경합**, **임계 구역의 길이**, 그리고 **다중 자원** 인지 **단일 플래그** 인지의 의미에 따라 다릅니다.

## 8. 동기화의 고전 문제들

세 가지 고전 문제는 시스템 소프트웨어 곳곳에서 반복적으로 나타납니다. 각각 독특한 동기화 패턴을 보여 줍니다.

### 8.1 유한 버퍼 문제

**크기 n의 유한 버퍼** 를 항목을 넣는 **생산자(producer)** 와 빼는 **소비자(consumer)** 가 공유합니다. 세 개의 세마포어:

| 세마포어 | 초깃값 | 역할 |
|-----------|------|------|
| `mutex` | 1 | 버퍼 보호 (이진). |
| `empty` | n | 빈 슬롯 수 (계수형). |
| `full`  | 0 | 채워진 슬롯 수 (계수형). |

공유 선언:

```c
int n;
semaphore mutex = 1;
semaphore empty = n;
semaphore full  = 0;
```

`empty` 와 `full` 은 **계수형** 세마포어이고, `mutex` 는 버퍼 접근을 위한 **이진** 세마포어입니다.

### 8.2 유한 버퍼 — 생산자/소비자 코드

```c
/* Producer */                       /* Consumer */
while (true) {                       while (true) {
    /* 항목 생산 */                       wait(full);
                                         wait(mutex);
    wait(empty);
    wait(mutex);                         /* 버퍼에서 next_consumed로 */
                                         /* 항목 제거                */
    /* 버퍼에 항목 추가 */
                                         signal(mutex);
    signal(mutex);                       signal(empty);
    signal(full);
}                                        /* next_consumed 소비 */
                                     }
```

명백하지 않은 두 가지 정확성 포인트:

- **wait 호출 순서가 중요합니다**: 항상 `wait(mutex)` **이전에** `wait(empty)` 또는 `wait(full)` 을 호출해야 합니다. 생산자가 `wait(mutex); wait(empty);` 로 순서를 바꾸면, 버퍼가 가득 찼을 때 `mutex` 를 들고 `empty` 에 블록되고 — 소비자가 슬롯을 비우려 해도 `mutex` 를 획득할 수 없습니다. **데드락**.
- 생산자와 소비자는 **구조적으로 대칭적** 입니다 (`empty` ↔ `full`). 이 패턴을 보면 대칭을 확인하세요 — 가장 쉬운 정확성 검사인 경우가 많습니다.

> **왜 단일 뮤텍스가 아니라 세 세마포어인가**: 단일 뮤텍스는 버퍼에 대한 상호 배제는 보장하지만, 생산자가 빈 슬롯이 생길 때까지 **기다리게** 만들 수 없습니다. 두 계수형 세마포어가 슬롯의 *가용성*을 표현하고, 뮤텍스는 삽입/제거 *행위*를 직렬화합니다.

### 8.3 Readers-Writers 문제

공유 데이터셋에 두 종류의 프로세스가 접근합니다:

- **읽기자(readers)** — 데이터를 읽기만 합니다.
- **쓰기자(writers)** — 읽고 수정합니다.

동기화 규칙:

- **여러 읽기자는 동시에 읽을 수 있습니다** (서로 충돌하지 않습니다).
- **쓰기자는 배타적 접근이 필요합니다** — 다른 쓰기자도, 어떤 읽기자도 안 됩니다.

문제의 두 변형은 누구의 선호가 이기는가에 따라 달라집니다:

- **첫 번째 Readers-Writers 문제(읽기자 우선).** 쓰기자가 대기 중이라도 읽기자는 들어갈 수 있습니다. 쓰기자는 **기아**할 수 있습니다.
- **두 번째 Readers-Writers 문제(쓰기자 우선).** 쓰기자가 대기 중이면 새로운 읽기자는 들어갈 수 없습니다. 읽기자는 **기아**할 수 있습니다.

두 버전 모두 **기아**가 가능합니다 — 피해자만 다를 뿐입니다. **Reader-Writer 락** 은 뮤텍스를 **읽기 모드(공유)** 와 **쓰기 모드(배타)** 로 구분할 수 있도록 일반화한 것이며, 우선순위 정책은 매개변수로 남깁니다.

### 8.4 Readers-Writers — 첫 번째 문제 해법

공유 변수:

```c
semaphore rw_mutex = 1;     /* writer-vs-readers 배제 */
semaphore mutex    = 1;     /* read_count 보호       */
int       read_count = 0;
```

코드:

```c
/* Writer */                            /* Reader */
while (true) {                          while (true) {
    wait(rw_mutex);                         wait(mutex);
                                            read_count++;
    /* 쓰기 수행 */                          if (read_count == 1)
                                                wait(rw_mutex);
    signal(rw_mutex);                       signal(mutex);
}
                                            /* 읽기 수행 */

                                            wait(mutex);
                                            read_count--;
                                            if (read_count == 0)
                                                signal(rw_mutex);
                                            signal(mutex);
                                        }
```

프로토콜 읽기:

- **첫 번째 읽기자** 가 `rw_mutex` 를 획득(쓰기자 차단); 이후 읽기자들은 그러지 않습니다.
- **마지막 읽기자** 가 `rw_mutex` 를 풀어(대기 중인 쓰기자가 들어올 수 있게 함).
- `mutex` 는 `read_count` 카운터만 보호하고 짧게만 잡힙니다.

n명의 읽기자와 1명의 쓰기자가 모두 대기 중일 때: 1명의 읽기자가 `rw_mutex` 를 들고 있고, n − 1명의 읽기자가 `mutex` 에서 대기하며, 쓰기자는 `rw_mutex` 에서 대기합니다.

### 8.5 Reader-Writer 락

**Reader-Writer 락** 은 읽기/쓰기 구분을 1급 시민으로 만든 뮤텍스의 일반화입니다.

- **읽기 모드** — 여러 스레드가 동시에 락을 잡을 수 있습니다.
- **쓰기 모드** — 한 스레드만 락을 잡을 수 있고, 어떤 읽기자도 동시에 잡을 수 없습니다.

Reader-Writer 락이 보상받는 경우:
- 읽기와 쓰기 연산이 코드에서 명확히 구분되는 경우.
- **읽기자가 쓰기자보다 압도적으로 많은** 경우 — 더 복잡한 락의 비용이 읽기자들 간의 더 높은 동시성으로 갚아집니다.

API 예시:
- POSIX: `pthread_rwlock_t`.
- Java: `java.util.concurrent.locks.ReentrantReadWriteLock`.
- Windows: **Slim Reader-Writer Lock** (SRWL).

### 8.6 식사하는 철학자들 문제

다섯 명의 철학자가 원형 식탁에 앉아 있습니다. 인접한 두 철학자 사이에는 젓가락이 하나씩 있어 — 총 다섯 개입니다. 각 철학자는 **사색** 과 **식사** 를 번갈아 하지만, 식사를 하려면 **양쪽 젓가락을 모두 들어야** 합니다.

규칙:
- 한 번에 하나의 젓가락만 집을 수 있습니다.
- 식사를 시작하려면 왼쪽과 오른쪽 젓가락이 모두 사용 가능해야 합니다.
- 식사를 마치면 두 젓가락을 모두 내려놓습니다.

*Silberschatz Figure 7.5 — 식사하는 철학자들.*

식사하는 철학자들 문제는 **여러 자원을 데드락이나 기아 없이 여러 프로세스에 동시에 할당하는 어려움** 의 정전급 예시입니다.

### 8.7 식사하는 철학자들 — 세마포어 해법과 데드락

각 젓가락을 1로 초기화된 이진 세마포어로 표현합니다:

```c
semaphore chopstick[5];   /* 모두 1로 초기화 */

/* 철학자 i */
while (true) {
    wait(chopstick[i]);              /* 왼쪽 */
    wait(chopstick[(i + 1) % 5]);    /* 오른쪽 */

    /* 잠시 식사 */

    signal(chopstick[i]);
    signal(chopstick[(i + 1) % 5]);

    /* 잠시 사색 */
}
```

이 해법은 **올바르지 않습니다**. 다섯 철학자가 동시에 `wait(chopstick[i])` 를 실행하면(각자 왼쪽을 집음) 모든 젓가락 값이 0이 되고, 각 철학자는 `wait(chopstick[(i+1)%5])` 에서 블록됩니다. **순환 대기 → 데드락**, 그리고 누구도 영영 먹을 수 없습니다.

### 8.8 식사하는 철학자들 — 데드락 회피 전략

세 가지 고전적인 해결책:

- **방법 1 — 동시에 식탁에 앉는 사람을 최대 4명으로 제한.** 4로 초기화된 추가 세마포어 `seats` 를 둡니다. 5개의 젓가락을 두고 최대 4명이 경쟁하므로 항상 적어도 한 명은 양쪽을 들 수 있습니다.
- **방법 2 — 양쪽이 모두 사용 가능할 때만 젓가락을 집기.** 검사와 집기를 한 임계 구역에서 수행해 어떤 철학자도 한쪽만 들고 있는 일이 없게 합니다. 모니터 해법이 이 방식입니다.
- **방법 3 — 비대칭 프로토콜.** 홀수 번 철학자는 **왼쪽 먼저**, 짝수 번 철학자는 **오른쪽 먼저** 집습니다. 이는 **순환 대기** 조건을 깹니다(데드락의 네 가지 조건은 다음 주 8장에서 다룹니다).

주의: **데드락이 없다고 해서 기아가 없는 것은 아닙니다**. 양쪽 이웃이 매우 배고픈 철학자는 원리적으로 영영 먹지 못할 수도 있습니다. 기아 제거는 추가적인 공정성 메커니즘이 필요합니다.

### 8.9 식사하는 철학자들 — 모니터 해법

모니터 해법은 "양쪽 젓가락이 모두 사용 가능할 때만 집기" 를 구조적으로 강제되는 불변식으로 만듭니다.

```text
monitor DiningPhilosophers {
    enum {THINKING, HUNGRY, EATING} state[5];
    condition self[5];

    void pickup(int i) {
        state[i] = HUNGRY;
        test(i);
        if (state[i] != EATING)
            self[i].wait();
    }

    void putdown(int i) {
        state[i] = THINKING;
        test((i + 4) % 5);    /* 왼쪽 이웃 재확인 */
        test((i + 1) % 5);    /* 오른쪽 이웃 재확인 */
    }

    void test(int i) {
        if (state[(i + 4) % 5] != EATING &&
            state[i] == HUNGRY &&
            state[(i + 1) % 5] != EATING) {
            state[i] = EATING;
            self[i].signal();
        }
    }

    initialization_code() {
        for (int i = 0; i < 5; i++)
            state[i] = THINKING;
    }
}
```

각 철학자 i는 다음과 같이 모니터를 호출합니다:

```text
DiningPhilosophers.pickup(i);
    ...
    /* 식사 */
    ...
DiningPhilosophers.putdown(i);
```

동작 방식:

- `pickup(i)` 는 상태를 HUNGRY로 바꾸고 `test(i)` 를 호출합니다. 두 이웃이 모두 EATING이 아니면 상태가 EATING이 되고 철학자가 진행합니다. 그렇지 않으면 `self[i].wait()` 가 그를 블록시킵니다.
- `putdown(i)` 는 상태를 THINKING으로 바꾸고 두 이웃에게 다시 `test()` 를 호출합니다 — 이웃 중 누군가가 배고프고 그 이웃의 이웃들이 식사 중이 아니면 그 이웃의 wait가 신호받습니다.
- **데드락 없음**: 철학자는 **양쪽** 젓가락이 모두 사용 가능할 때만 식사하므로, 누구도 한쪽만 들고 있지 않습니다.
- **기아는 가능**: 이웃이 계속 번갈아 식사하는 운 나쁜 철학자는 영원히 건너뛰어질 수 있습니다(교재의 연습 문제로 남겨져 있습니다).

## 9. 실제 시스템에서의 동기화

### 9.1 Windows 동기화

Windows는 단일 프로세서와 다중 프로세서 경우를 구분하며, 커널 측과 사용자 측 기본 도구를 모두 제공합니다.

- **단일 프로세서** — 커널에서 **인터럽트 마스킹** 으로 전역 상태를 보호합니다.
- **다중 프로세서** — 매우 짧은 커널 임계 구역에는 **스핀락**을 사용합니다. 스핀락을 들고 있는 동안에는 **절대 선점되지 않습니다** (그렇지 않으면 점유자가 잠든 사이 스피너가 영원히 CPU를 태우게 됩니다).

**Dispatcher Objects** — 사용자 스레드에 노출되는 동기화용 커널 객체.
- 예: **Mutex Lock**, **Semaphore**, **Event**, **Timer**.
- 각자 두 상태를 가집니다: **signaled** (사용 가능) 와 **nonsignaled** (사용 중).
- nonsignaled 객체를 요청한 스레드는 **블록**되고 객체의 대기 큐에 놓입니다.
- 객체가 signaled가 되면 대기자가 **ready** 상태로 옮겨집니다.

**Critical-Section Object** — 흔한 경우에 최적화된 사용자 모드 뮤텍스.
- 먼저 **스핀락** 으로 시도(커널 호출 회피).
- 스핀이 너무 길어지면 **커널 뮤텍스** 를 할당하고 블록합니다.
- **무경합** 인 경우 획득/해제가 전적으로 사용자 모드에서 일어납니다 → 매우 빠릅니다.

### 9.2 Linux 동기화

Linux는 커널 **2.6** 이후 **완전 선점형** 입니다(이전에는 비선점형).

| 도구 | 목적 | 비고 |
|------|---------|-------|
| `atomic_t`     | 원자 정수 연산 | 카운터, 시퀀스 생성기 |
| `spinlock_t`   | 짧은 커널 임계 구역 (SMP) | 단일 프로세서에서는 **선점 비활성화** 로 대체 |
| `mutex`        | 더 긴 임계 구역 | **Sleeping** 락 — 스핀하지 않고 블록 |
| `semaphore`    | 계수형 세마포어 | 긴 대기에 적합 |

스핀락 기본 동작의 단일 CPU vs SMP 차이:

```c
/* 단일 프로세서                | 다중 프로세서          */
/* 커널 선점 비활성화           | 스핀락 획득            */
/* ... 임계 구역 ...                                   */
/* 커널 선점 활성화             | 스핀락 해제            */
```

Linux 스핀락과 뮤텍스의 두 가지 중요한 속성:

- **비재귀(nonrecursive)** — 이미 락을 잡고 있는 스레드가 같은 락을 다시 잡으려 하면 **데드락** 입니다.
- 커널은 작업당 `preempt_count` 를 유지합니다 — `preempt_count == 0` 일 때만 선점이 허용됩니다. 락을 잡으면 이 카운터가 증가합니다.

### 9.3 POSIX 뮤텍스 락

POSIX (UNIX, Linux, macOS) 는 `pthread` 를 통해 **사용자 수준** 의 동기화 기본 도구를 제공합니다.

```c
#include <pthread.h>

pthread_mutex_t mutex;
pthread_mutex_init(&mutex, NULL);

pthread_mutex_lock(&mutex);
    /* critical section */
pthread_mutex_unlock(&mutex);
```

이는 어떤 pthread 프로그램에서도 보게 될 표준 뮤텍스 관용구입니다.

### 9.4 POSIX 세마포어 (Named/Unnamed)

POSIX는 공유 범위가 다른 두 종류의 세마포어를 제공합니다.

**Named 세마포어** — 문자열 이름으로 식별되며, **무관한 프로세스들** 사이에 공유될 수 있습니다:

```c
#include <semaphore.h>

sem_t *sem;
sem = sem_open("SEM", O_CREAT, 0666, 1);

sem_wait(sem);
    /* critical section */
sem_post(sem);
```

**Unnamed 세마포어** — `sem_init(&sem, 0, 1)` 로 초기화되며, **같은 프로세스 내 스레드들** (또는 공유 메모리를 통한 관련 프로세스들) 사이에 공유됩니다.

| 종류 | API | 공유 범위 |
|------|-----|---------------|
| Named   | `sem_open` / `sem_wait` / `sem_post` | **무관한 프로세스들** 간 |
| Unnamed | `sem_init` / `sem_wait` / `sem_post` | **단일 프로세스 내** (또는 공유 메모리 경유) |

### 9.5 POSIX 조건 변수

C에는 모니터 내장 기능이 없으므로, POSIX의 조건 변수는 뮤텍스와 명시적으로 짝지어 사용됩니다.

설정:

```c
pthread_mutex_t  mutex;
pthread_cond_t   cond_var;
pthread_mutex_init(&mutex, NULL);
pthread_cond_init(&cond_var, NULL);
```

대기 스레드:

```c
pthread_mutex_lock(&mutex);
while (a != b)
    pthread_cond_wait(&cond_var, &mutex);   /* 원자적: 뮤텍스 해제 + sleep */
pthread_mutex_unlock(&mutex);
```

신호 스레드:

```c
pthread_mutex_lock(&mutex);
a = b;
pthread_cond_signal(&cond_var);             /* 한 명의 대기자 깨움 */
pthread_mutex_unlock(&mutex);
```

모든 pthread 프로그래머가 알아야 할 세 가지 규칙:

- `pthread_cond_wait()` 는 **원자적으로 뮤텍스를 해제하고 블록**합니다; 깨어날 때 뮤텍스를 **다시 획득** 한 뒤 반환합니다.
- `pthread_cond_signal()` 은 뮤텍스를 해제하지 **않습니다** — 신호자가 직접 `unlock()` 을 호출해야 합니다.
- 조건 검사는 항상 **`while`** 루프 안에서 해야 하며, `if` 가 아닙니다. 표준은 spurious wakeup(가짜로 깨어남)을 허용하며, 깨어난 뒤 뮤텍스를 다시 잡는 사이 다른 스레드가 상태를 바꿨을 수 있습니다.

> **"항상 while, 절대 if" 규칙**: `if (a != b) pthread_cond_wait(...)` 라고 쓰는 것은 고전적인 버그입니다. spurious wakeup이 결코 일어나지 않더라도, 다른 대기자가 깨어난 당신보다 먼저 자원을 가져가버릴 수 있고, 당신은 잘못된 전제로 진행하게 됩니다. `while` 은 깨어날 때마다 술어를 다시 검사합니다.

## 10. Java 동기화

Java는 모니터 같은 동기화를 **언어 수준** 에서 제공하며, `synchronized` 키워드가 주된 도구이고 `java.util.concurrent` 에 더 풍부한 기본 도구들이 있습니다.

### 10.1 Java 모니터 — `synchronized`, `wait()`/`notify()`

Java의 내장 모니터를 사용한 유한 버퍼:

```java
public class BoundedBuffer<E> {
    private static final int BUFFER_SIZE = 5;
    private int count, in, out;
    private E[] buffer;

    public synchronized void insert(E item) {
        while (count == BUFFER_SIZE) {
            try { wait(); }
            catch (InterruptedException ie) { }
        }
        buffer[in] = item;
        in = (in + 1) % BUFFER_SIZE;
        count++;
        notify();
    }

    public synchronized E remove() {
        while (count == 0) {
            try { wait(); }
            catch (InterruptedException ie) { }
        }
        E item = buffer[out];
        out = (out + 1) % BUFFER_SIZE;
        count--;
        notify();
        return item;
    }
}
```

`synchronized` 키워드는 메소드 진입 시 수신자 객체의 모니터 락을 **자동으로** 획득하고, 종료 시(예외 발생 시에도) 해제합니다.

### 10.2 Java 모니터의 내부 동작

- 모든 Java 객체는 **하나의 락**, **entry set** (락을 기다리는 스레드들), **wait set** (`wait()` 에 블록된 스레드들)을 가집니다.

**스레드가 `wait()` 를 호출하면**:
1. 객체의 락을 해제합니다.
2. **blocked** 상태로 전이됩니다.
3. **wait set** 에 합류합니다.

**스레드가 `notify()` 를 호출하면**:
1. wait set에서 **임의의** 스레드 T를 선택합니다.
2. T를 wait set에서 **entry set** 으로 옮깁니다.
3. T를 runnable로 표시합니다.
4. T가 결국 락을 다시 획득하면 그 `wait()` 호출이 반환됩니다.

`notifyAll()` 은 `notify()` 와 같지만 wait set의 **모든** 스레드에 대해 동작합니다. 같은 락을 공유하는 대기 조건이 두 종류 이상일 때는 `notifyAll()` 을 사용하세요 — 조건이 실제로 만족된 스레드들만 `while` 루프를 빠져나가게 됩니다.

### 10.3 ReentrantLock과 Semaphore

`java.util.concurrent.locks.ReentrantLock` 은 `synchronized` 보다 유연한 락입니다:

```java
Lock key = new ReentrantLock();
key.lock();
try {
    /* critical section */
} finally {
    key.unlock();    /* finally에서 반드시 해제 */
}
```

특이한 기능:
- **공정성 매개변수** (`new ReentrantLock(true)`) 는 락이 가장 오래 기다린 스레드를 선호하게 합니다.
- **재진입 가능(reentrant)**: 이미 락을 잡고 있는 스레드는 데드락 없이 락을 다시 잡을 수 있습니다(락이 깊이를 카운트합니다).

`java.util.concurrent.Semaphore` 는 표준 세마포어입니다:

```java
Semaphore sem = new Semaphore(1);
try {
    sem.acquire();
    /* critical section */
} catch (InterruptedException ie) { }
finally {
    sem.release();
}
```

### 10.4 Java 조건 변수

`Condition` 객체(`Lock` 과 연관)는 **이름 있는** 조건에서 대기할 수 있게 해, `notify()` 로 무작위로 깨우는 대신 *특정* 대기자를 깨울 수 있게 해 줍니다.

```java
Lock lock = new ReentrantLock();
Condition[] condVars = new Condition[5];
for (int i = 0; i < 5; i++)
    condVars[i] = lock.newCondition();

public void doWork(int threadNumber) {
    lock.lock();
    try {
        if (threadNumber != turn)
            condVars[threadNumber].await();

        /* 작업 수행 */

        turn = (turn + 1) % 5;
        condVars[turn].signal();
    } catch (InterruptedException ie) { }
    finally {
        lock.unlock();
    }
}
```

- `await()` 는 락을 해제하고 블록합니다(`pthread_cond_wait()` 와 Java `Object.wait()` 의 대응).
- `signal()` 은 그 이름 있는 조건에서 **특정** 대기자를 깨웁니다 — 내장 모니터의 단일 익명 `notify()` 대비 핵심 장점입니다.

> **이름 있는 조건 변수가 중요한 경우**: 유한 버퍼에서는 "버퍼가 가득 차지 않음" 과 "버퍼가 비지 않음" 이 별개의 조건입니다. 단일 익명 `notify()` 로는, 마지막 슬롯을 채운 생산자가 빈 슬롯을 기다리는 또 다른 *생산자* 를 깨울 수 있고, 그 생산자는 즉시 다시 블록됩니다 — 헛된 깨움입니다. 이름 있는 조건 변수를 쓰면 생산자가 소비자 조건을 정확히 신호할 수 있어 헛된 왕복이 사라집니다.

## 11. 실습 — 식사하는 철학자들 구현

이번 실습은 POSIX 스레드를 사용해 식사하는 철학자들 문제로 이 장의 개념을 연습합니다.

### 11.1 실습: 단순 pthread 구현

각 젓가락은 `pthread_mutex_t` 입니다. 각 철학자는 "사색 → 왼쪽 잡기 → 오른쪽 잡기 → 식사 → 내려놓기" 를 반복하는 스레드입니다.

```c
#include <pthread.h>
#include <stdio.h>
#include <unistd.h>
#include <stdlib.h>
#define N 5

pthread_mutex_t chopstick[N];

void *philosopher(void *arg) {
    int id = *(int *)arg;
    while (1) {
        printf("Philosopher %d is thinking\n", id);
        usleep(rand() % 1000000);

        pthread_mutex_lock(&chopstick[id]);
        pthread_mutex_lock(&chopstick[(id + 1) % N]);

        printf("Philosopher %d is eating\n", id);
        usleep(rand() % 1000000);

        pthread_mutex_unlock(&chopstick[id]);
        pthread_mutex_unlock(&chopstick[(id + 1) % N]);
    }
    return NULL;
}
```

실습 과제:
- 프로그램을 실행해 어떻게 **데드락**에 빠질 수 있는지 관찰합니다(모든 철학자가 왼쪽 젓가락을 들고 있는 상황).
- 고전적인 해결책 중 하나(비대칭 프로토콜, 최대 4명 제한, 모니터)를 구현하고 데드락이 사라짐을 확인합니다.

### 11.2 실습: 비대칭(Asymmetric) 해법

짝수 번 철학자는 **오른쪽** 을 먼저, 홀수 번 철학자는 **왼쪽** 을 먼저 집습니다 — 순환 대기 조건이 깨집니다.

```c
void *philosopher_asymmetric(void *arg) {
    int id = *(int *)arg;
    int first, second;

    if (id % 2 == 0) {
        first  = (id + 1) % N;     /* 오른쪽 먼저 */
        second = id;
    } else {
        first  = id;               /* 왼쪽 먼저 */
        second = (id + 1) % N;
    }

    while (1) {
        printf("Philosopher %d is thinking\n", id);
        usleep(rand() % 1000000);

        pthread_mutex_lock(&chopstick[first]);
        pthread_mutex_lock(&chopstick[second]);

        printf("Philosopher %d is eating\n", id);
        usleep(rand() % 1000000);

        pthread_mutex_unlock(&chopstick[first]);
        pthread_mutex_unlock(&chopstick[second]);
    }
    return NULL;
}
```

> **왜 동작하는가**: 데드락은 "P0이 P1의 젓가락을 기다리고, P1이 P2의 젓가락을 기다리고, ..., Pn이 P0의 젓가락을 기다리는" *사이클* 이 필요합니다. 비대칭 프로토콜에서는 인접한 두 철학자가 공유하는 젓가락을 획득하는 순서가 뒤집히기 때문에 사이클이 닫힐 수 없습니다.

### 11.3 실습: POSIX 조건 변수 기반 모니터 해법

§ 8.9의 교재 모니터를 POSIX로 직접 옮긴 형태:

```c
#include <pthread.h>
#define N 5

enum {THINKING, HUNGRY, EATING} state[N];
pthread_mutex_t monitor_lock = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t  self[N];

void test(int i) {
    if (state[(i + 4) % N] != EATING &&
        state[i] == HUNGRY &&
        state[(i + 1) % N] != EATING) {
        state[i] = EATING;
        pthread_cond_signal(&self[i]);
    }
}

void pickup(int i) {
    pthread_mutex_lock(&monitor_lock);
    state[i] = HUNGRY;
    test(i);
    while (state[i] != EATING)
        pthread_cond_wait(&self[i], &monitor_lock);
    pthread_mutex_unlock(&monitor_lock);
}

void putdown(int i) {
    pthread_mutex_lock(&monitor_lock);
    state[i] = THINKING;
    test((i + 4) % N);
    test((i + 1) % N);
    pthread_mutex_unlock(&monitor_lock);
}
```

`pthread_cond_wait` 주변에 `while (state[i] != EATING)` 을 사용한 점에 주목하세요 — § 9.5의 패턴 그대로입니다(spurious wakeup 방어).

## 점검 문제

1. 레이스 컨디션을 정의하고 생산자/소비자의 `count++` / `count--` 예시로 설명하세요. 왜 증상이 타이밍에 의존하나요?

   > **답:** 레이스 컨디션은 둘 이상의 스레드가 공유 데이터에 동시에 접근하고 **결과가 인터리빙 순서에 따라 달라지는** 경우입니다. `count = 5` 일 때 `count++` 와 `count--` 는 각각 load → modify → store로 분해되며, 한 스레드의 store가 다른 스레드의 store를 덮어쓰는 인터리빙은 한 갱신을 잃어 `count` 가 `5` 대신 `4` 또는 `6` 이 됩니다. 나쁜 인터리빙이 일어나는지는 스케줄링에 달려 있어 버그가 간헐적으로 나타나며 — 테스트에서 놓치기 쉽습니다.

2. 임계 구역 문제의 세 요건을 말하세요. 각각이 빠지면 어떤 일이 생기나요?

   > **답:** **상호 배제** — 임계 구역 안에 한 번에 한 프로세스만 (없으면 레이스 컨디션 재등장). **진행** — 임계 구역에 아무도 없을 때 대기 중인 프로세스를 무한정 미루지 않고 선택해야 함 (없으면 임계 구역이 비어 있어도 시스템이 데드락에 빠질 수 있음). **한정 대기** — 대기 프로세스가 진입을 받기 전에 다른 프로세스가 들어갈 수 있는 횟수에 유한한 상한 (없으면 기아).

3. 선점형 커널과 비선점형 커널을 비교하세요. 왜 다중 프로세서 선점형 커널이 단일 CPU 비선점형 커널보다 더 신중한 동기화를 필요로 합니까?

   > **답:** **단일 CPU 비선점형** 커널에서는 다른 커널 스레드가 현재 스레드가 양보할 때까지 실행될 수 없으므로 커널 데이터 레이스가 본질적으로 없습니다. **선점형** 커널은 어느 순간에도 다른 스레드가(다른 **코어** 에서) 끼어들 수 있으므로 — 모든 공유 커널 구조에 명시적 보호가 필요합니다. 인터럽트 비활성화는 단일 프로세서에서는 충분하지만 SMP의 코어 간에는 그렇지 않습니다.

4. Peterson 해법을 살펴보세요. 왜 상호 배제를 만족하며, 왜 현대 하드웨어에서 깨질 수 있나요?

   > **답:** 각 프로세스 Pi는 `flag[i] = true`, `turn = j` 를 설정한 뒤 `flag[j] && turn == j` 인 동안 대기합니다. **상호 배제**: 두 프로세스가 모두 임계 구역 안에 있다면 `flag[0] = flag[1] = true` 이므로, 둘 다 빠져나간 유일한 길은 `turn == self` 입니다 — 그러나 `turn` 은 단일 값이므로 모순. **현대 하드웨어** 는 `flag[i] = true` 와 `turn = j` 의 쓰기를 재배열할 수 있으며, `turn = j` 가 먼저 보이면 두 프로세스 모두 진입 검사를 통과할 수 있습니다. 해결책: 두 쓰기 사이에 메모리 배리어를 삽입.

5. `test_and_set` 과 `compare_and_swap` 의 의사코드를 쓰고, 각각이 락을 어떻게 구현하는지 보이세요. 왜 CAS가 더 강력한가요?

   > **답:**
   > - `test_and_set(target)` 은 `*target` 을 원자적으로 읽고 `true` 로 설정한 뒤 이전 값을 반환합니다. 락: `test_and_set(&lock)` 이 `true` 를 반환하는 동안 스핀.
   > - `compare_and_swap(value, expected, new_value)` 는 `*value == expected` 일 때만 원자적으로 `new_value` 를 쓰고 원래 값을 반환합니다. 락: `compare_and_swap(&lock, 0, 1) != 0` 인 동안 스핀.
   > - CAS는 **임의의 정수** (boolean뿐 아니라)에서 동작하며 "내가 마지막으로 본 이후 값이 바뀌지 않았다면 갱신" 을 표현할 수 있어 — 낙관적 동시성과 락-프리 자료 구조의 토대가 되기 때문에 더 강력합니다.

6. 원자 변수란 무엇이며, 어떤 용도에는 **부적합**한가요?

   > **답:** 원자 변수는 보통 CAS 위에 만들어진 **단일 변수** 의 레이스 없는 갱신(예: `atomic_inc(&counter)`)을 제공합니다. 여러 변수에 걸친 불변식 — 예: 유한 버퍼의 `count`, `in`, `out` 트리플은 함께 갱신되어야 함 — 에는 **부적합** 하며, 단일 뮤텍스/세마포어로 보호하지 않으면 관찰자가 비일관 상태를 볼 수 있습니다.

7. 스핀락과 sleeping 뮤텍스를 비교하세요. 각각이 더 적합한 경우와 어림 규칙은 무엇인가요?

   > **답:** **스핀락** 은 바쁜 루프; **sleeping 뮤텍스** 는 블록(컨텍스트 전환 2회). 스핀락은 락이 매우 짧게 잡힐 때 — 특히 한 코어가 임계 구역을 끝내는 동안 다른 코어가 스핀할 수 있는 멀티코어 시스템에서 — 더 적합합니다. Sleeping 락은 자는 대기자가 자기 CPU를 풀어 주므로 오래 잡히는 락에 유리합니다. **어림 규칙**: 예상 점유 시간이 **컨텍스트 전환 2회보다 짧을 때** 스핀락 사용.

8. 세마포어 `wait` 와 `signal` 연산을 말하세요. 왜 원자적이어야 하나요?

   > **답:** `wait(S)`: `S <= 0` 이면 스핀; 이후 `S--`. `signal(S)`: `S++`. `S` 의 테스트와 갱신이 분할 불가능한 한 단계로 일어나야 하므로 원자적이어야 합니다 — 그렇지 않으면 두 스레드가 모두 `S = 1` 을 읽고 둘 다 감소시켜 둘 다 진입했다고 생각하게 되며 `S = -1` 이 되어 두 스레드가 임계 구역에 동시에 있게 됩니다.

9. 대기 큐로 구현된 세마포어에서 `S->value` 가 **음수** 라는 것은 무엇을 뜻하나요?

   > **답:** 절댓값이 **세마포어에 블록된 프로세스 수와 같습니다**. 각 `wait` 는 먼저 감소시키고 결과가 음수면 호출자를 enqueue 합니다; 각 `signal` 은 증가시키고 결과가 ≤ 0 이면 dequeue 하여 한 명을 깨웁니다.

10. 흔한 세마포어 오용 버그 세 가지를 나열하세요. 왜 디버깅이 그토록 어렵나요?

    > **답:** **(1) 순서 뒤바꿈** — `signal(mutex); CS; wait(mutex)` 는 여러 스레드를 임계 구역에 들입니다. **(2) wait 중복** — `wait(mutex); CS; wait(mutex)` 는 두 번째 wait에서 데드락. **(3) wait 또는 signal 누락** — 임계 구역이 보호되지 않거나 아무도 깨우지 않음. 특정 인터리빙에서만 드러나고 일반 테스트는 통과하기 때문에 디버깅이 어렵습니다; 정답은 **상위 수준 구문(모니터)** 사용.

11. 모니터란 무엇이며, 세마포어와 어떻게 다른가요? 왜 조건 변수가 추가로 필요한가요?

    > **답:** 모니터는 자기 연산들에 대한 **상호 배제를 자동으로 강제하는 ADT** 입니다 — 한 번에 한 스레드만 안에 있을 수 있고, 락/언락은 런타임이 (프로그래머가 아니라) 삽입합니다. **조건 변수** (`x.wait()`, `x.signal()`) 는 상호 배제만으로 "버퍼가 비지 않을 때까지 대기" 를 표현할 수 없기에 필요합니다; `wait` 는 모니터를 풀어 다른 스레드가 상태를 바꿀 수 있게 하고, `signal` 은 그들이 상태를 바꿨을 때 대기자를 깨웁니다.

12. 모니터에서 **signal-and-wait** 와 **signal-and-continue** 정책을 구분하세요. 무슨 문제를 푸나요?

    > **답:** 모니터 안의 스레드 P가 `x.signal()` 을 호출하고 다른 스레드 Q가 `x` 에서 대기 중이면, 두 스레드 모두 모니터 안의 활성 스레드가 되고 싶지만 한 명만 가능합니다. **Signal-and-wait**: P가 일시정지되고 Q가 즉시 실행. **Signal-and-continue**: P가 모니터를 떠날 때까지 계속 실행되고 그 다음 Q. 언어마다 정책이 다르며, 두 경우 모두 조건 술어는 `while` 루프 안에서 재검사해야 합니다.

13. **데드락**을 정의하고 두 세마포어 예시를 드세요.

    > **답:** **데드락** 은 어떤 프로세스 집합의 모든 프로세스가 그 집합 안의 다른 프로세스만이 일으킬 수 있는 사건을 기다리는 상태입니다. 세마포어 S, Q (둘 다 1) 예시: P0은 `wait(S); wait(Q);`, P1은 `wait(Q); wait(S);`. P0이 S를 획득하고 선점되면 P1은 Q를 획득하고 `wait(S)` 를 시도하며, P0은 `wait(Q)` 를 시도 — 각자가 상대가 필요한 것을 들고 있습니다.

14. 세 우선순위 L < M < H로 우선순위 역전을 설명하세요. 우선순위 상속 프로토콜이 어떻게 해결하며, 어떤 실제 사건이 더 넓은 채택을 촉발했나요?

    > **답:** L이 락을 들고; H가 락이 필요해 블록; M이 실행 가능해져 L을 선점(M > L) — 이제 H가 M에 의해 지연됩니다. **우선순위 상속** 은 L이 H가 필요로 하는 락을 들고 있는 동안 L의 우선순위를 H 수준으로 일시적으로 끌어올려 M이 L을 선점할 수 없게 하고; L이 끝내고 락을 풀면; L은 원래대로 돌아옵니다. **Mars Pathfinder (1997)**: 시스템 리셋이 반복되어 추적된 결과 VxWorks의 우선순위 역전 버그였고; JPL은 우선순위 상속을 활성화하여 원격으로 해결했습니다.

15. 세 세마포어로 유한 버퍼 문제를 푸세요. 왜 `wait` 호출 순서가 중요한가요?

    > **답:** `mutex = 1` (이진), `empty = n`, `full = 0` (계수형) 사용. 생산자: `wait(empty); wait(mutex); insert; signal(mutex); signal(full)`. 소비자: `wait(full); wait(mutex); remove; signal(mutex); signal(empty)`. **순서 중요**: 생산자가 `wait(mutex); wait(empty)` 로 뒤집으면 버퍼가 가득 찼을 때 생산자는 `mutex` 를 *들고* `empty` 에 블록되고 — 소비자가 슬롯을 비우러 들어올 수 없습니다 → **데드락**.

16. 첫 번째와 두 번째 Readers-Writers 문제를 구분하세요. 기아를 제거할 수 있나요?

    > **답:** **첫 번째**: 읽기자 우선 — 쓰기자 대기 중에도 읽기자가 들어갈 수 있음 → **쓰기자 기아**. **두 번째**: 쓰기자 우선 — 쓰기자가 대기 중이면 새 읽기자가 들어갈 수 없음 → **읽기자 기아**. 두 변형 모두 비선호 측의 기아를 허용합니다; 제거하려면 추가 공정성 메커니즘(예: 읽기자와 쓰기자를 인터리빙하는 큐) 필요.

17. 왜 단순 세마포어 식사하는 철학자들 해법은 데드락에 빠지나요? 세 가지 해결책을 나열하세요.

    > **답:** 모든 철학자가 동시에 `wait(chopstick[i])` (왼쪽)을 실행하면 다섯 젓가락 모두 0이 되고; 각자 오른쪽 젓가락에 블록 → **순환 대기 → 데드락**. 해결책: **(1) 동시에 4명** 만 식탁에 (사이클 제한); **(2) 양쪽 젓가락을 임계 구역에서 원자적으로 집기**, 예: 모니터 해법; **(3) 비대칭 프로토콜** — 홀수 철학자는 왼쪽 먼저, 짝수는 오른쪽 먼저 — 순환 대기를 깸.

18. Windows Dispatcher Objects와 Critical-Section Objects를 비교하세요. 각각 언제 사용되나요?

    > **답:** **Dispatcher Objects** (Mutex, Semaphore, Event, Timer)는 signaled/nonsignaled 상태를 가진 커널 객체이며; nonsignaled 객체를 획득하면 스레드가 커널에서 블록됩니다. **Critical-Section Objects** 는 **먼저 사용자 공간에서 스핀락을 시도** 하고, 경합이 심할 때만 커널 뮤텍스로 폴백하는 사용자 모드 뮤텍스입니다. 흔한 무경합 경우에 커널을 피해 훨씬 빠릅니다.

19. Linux의 `spinlock_t` 와 `mutex` 를 비교하세요. **비재귀(nonrecursive)** 라는 것은 무엇을 뜻하나요?

    > **답:** `spinlock_t` 는 바쁜 대기를 하며 매우 짧은 커널 임계 구역에 적합; `mutex` 는 더 긴 대기를 위한 **sleeping** 락입니다. **비재귀** 는 이미 락을 잡은 스레드가 같은 락을 다시 획득하려 하면 데드락이 된다는 뜻입니다 — 내부 "이미 들고 있음" 카운터가 없습니다. 재귀 락이 필요하면 같은 락을 재진입하지 않도록 코드를 구조화해야 합니다.

20. 모든 POSIX 조건 변수 사용자가 따라야 할 세 가지 규칙을 말하세요.

    > **답:**
    > **(1)** `pthread_cond_wait()` 는 원자적으로 뮤텍스를 해제하고 블록하며; 깨어날 때 뮤텍스를 다시 획득합니다.
    > **(2)** `pthread_cond_signal()` 은 뮤텍스를 해제하지 않으므로 — 신호자가 직접 `pthread_mutex_unlock()` 을 호출해야 합니다.
    > **(3)** 술어 검사는 항상 **`while`** 루프 안에서 — `if` 가 아닙니다. 표준은 spurious wakeup을 허용하며, 깨어난 뒤 뮤텍스를 다시 잡는 사이 다른 스레드가 상태를 바꿨을 수 있습니다.

21. 왜 Java 이름 있는 `Condition` 객체(`ReentrantLock.newCondition()` 으로 생성)는 단일 익명 `notify()` 보다 유한 버퍼 패턴에서 도움이 되나요?

    > **답:** 유한 버퍼는 두 가지 다른 조건을 가집니다: "가득 차지 않음" (생산자가 대기) 과 "비지 않음" (소비자가 대기). 단일 익명 `notify()` 로는 사실 소비자의 조건이 만족되었을 때 런타임이 생산자를 깨울 수도 있고(혹은 그 반대), 깨어난 스레드는 자기 `while` 을 검사하고 다시 잠들어 깨움이 헛되게 됩니다. **이름 있는 조건** 은 신호자가 만족된 **특정** 조건을 정확히 겨냥할 수 있게 하여 헛된 왕복을 제거합니다.
