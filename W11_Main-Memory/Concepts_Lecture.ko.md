## 1. 배경과 기본 하드웨어

### 1.1 메모리 관리가 필요한 이유

CPU 사용률을 높이려면 OS는 **여러 프로세스를 동시에 메모리에 적재**해야 합니다. 하나 이상의 프로세스가 물리 메모리를 공유하는 순간 세 가지 문제가 발생합니다:

- **상호 보호** — 한 프로세스가 다른 프로세스의 메모리를 읽거나 덮어쓸 수 없어야 함.
- **주소 변환** — 모든 프로세스가 0부터 시작하는 깨끗한 주소 공간을 원하지만, 실제로 모두가 물리 주소 0에 살 수는 없음.
- **효율적 공유** — 코드(라이브러리, 커널)는 공유해야 하지만 개인 데이터는 절대 공유되면 안 됨.

> **이 토대가 중요한 이유**: 이후의 모든 메모리 주제(페이징, 가상 메모리, 페이지 교체, copy-on-write)는 이 세 가지 성질을 대규모로 강제하기 위한 서로 다른 방법들입니다.

### 1.2 CPU가 직접 접근할 수 있는 저장소

CPU가 직접 접근 가능한 저장소는 두 종류입니다:

```text
   +-----------+
   | Register  |   <- 1 cycle
   +-----------+
   |   Cache   |   <- 몇 cycle (ISA에 투명)
   +-----------+
   |   Main    |   <- 수십~수백 cycle
   |  Memory   |
   +-----------+
   |   Disk    |   <- 수백만 cycle, 직접 접근 불가
   +-----------+
```

> **선수 지식 — 메모리 계층 구조(memory hierarchy)** (컴퓨터구조): 각 계층은 용량과 지연시간을 맞바꿉니다. **"ISA에 투명(transparent)"** 이란 캐시가 전적으로 하드웨어에 의해 관리됨을 뜻합니다 — OS도 실행 중인 프로그램도 캐시에 명시적 load/store를 발행하지 않음; CPU가 수요 발생 시 자동으로 캐시 라인을 채우고 자체 정책으로 추방. 명령어 집합의 load/store에 노출되는 것은 **레지스터 파일**과 **메인 메모리**뿐이라서 OS 교과서들은 캐시가 없는 듯이 "직접 접근 가능한" 저장소를 이야기합니다.

디스크의 모든 것은 CPU가 건드리기 전에 **메모리로 적재**되어야 합니다. 메모리가 희소하고 경쟁이 심한 자원인 이유가 바로 이것입니다.

### 1.3 Base / Limit 레지스터

가장 단순한 하드웨어 보호 방식: 각 프로세스에 **base**와 **limit** 레지스터를 부여.

- **Base 레지스터** — 프로세스가 합법적으로 접근 가능한 최저 물리 주소.
- **Limit 레지스터** — 합법 범위의 *크기* (상한 주소가 아님).
- CPU가 발생시킨 주소는 `base ≤ addr < base + limit`을 만족해야 함. 위반 시 **OS로 트랩**(사용자 프로그램에서는 segmentation fault).

구체적 예: Base = 300040, Limit = 120900 이면 접근 가능 범위는 `300040 ~ 420939`.

이 레지스터는 **커널 모드**에서 특권 명령어로만 적재 가능 — 그렇지 않으면 악의적 사용자 프로세스가 자기 한계를 올려버릴 수 있음.

## 2. 주소 바인딩과 MMU

### 2.1 세 가지 바인딩 시점

프로그램의 메모리 주소는 마법처럼 정해지지 않고, 어느 시점에 물리 위치로 *바인딩(binding)* 됩니다. 바인딩 시점이 런타임에 OS가 해야 할 일을 결정합니다.

| 바인딩 시점 | 설명 | 특성 |
|------------|------|------|
| **컴파일 시점** | 메모리 시작 주소가 컴파일 시 고정 | **절대 코드(absolute code)** 생성; 시작 주소 변경 시 재컴파일 필요 (MS-DOS .COM) |
| **적재 시점** | 로더가 프로그램을 메모리에 배치할 때 시작 주소 고정 | **재배치 가능(relocatable) 코드** 생성; 실행 후 이동 불가 |
| **실행 시점** | 프로그램 실행 *중* 바인딩 | **MMU** 필요; 거의 모든 현대 OS가 사용 |

### 2.2 사용자 프로그램의 다단계 처리

소스 코드는 여러 단계를 거치며 각 단계마다 부분 바인딩을 수행합니다:

```text
  source.c   --(compile)-->  object module   (재배치 가능, 심볼명 보유)
             --(link)----->  load module
             --(load)----->  in-memory binary
             --(run)------>  physical reference (MMU)
```

- **심볼릭** 주소(변수명) → **재배치 가능** (모듈 시작 기준 오프셋) → **절대** (물리 주소).

### 2.3 논리 주소 vs 물리 주소

런타임에는 두 주소 공간이 공존합니다.

- **논리 주소** (= *가상 주소*)
  - CPU가 명령어를 실행하면서 생성하는 주소.
  - 프로세스는 0부터 어떤 최대치까지의 독립적 공간을 봄.
- **물리 주소**
  - DRAM에 실제로 전달되는 주소 (Memory Address Register에 적재됨).
  - 실제 공유 범위 `R + 0 … R + max`에 존재.

**컴파일/적재 시점** 바인딩에서는 논리 = 물리. **실행 시점** 바인딩에서는 논리 ≠ 물리이며 **모든 메모리 참조마다 변환이 필요**합니다.

### 2.4 MMU — 메모리 관리 장치

**MMU**는 논리→물리 변환을 수행하는 하드웨어입니다. 가장 단순한 동적 재배치(dynamic relocation) 방식에서는 모든 논리 주소에 **재배치 레지스터(relocation register)** 를 더할 뿐입니다. *(참고: 이 가장 단순한 설계에서 재배치 레지스터는 §1.3에서 소개한 **base 레지스터**와 같은 물리 레지스터입니다 — 두 이름은 서로 다른 역할을 강조하는 표현일 뿐. "base"는 limit 검사의 하한을, "relocation"은 사용자가 보는 0…max를 실제 물리 범위로 끌어올리는 가산값을 의미합니다.)*

```text
  logical addr  -->  +-----+
                     | MMU |  + relocation_register  -->  physical addr  -->  DRAM
                     +-----+
```

사용자 프로그램은 *오직* 논리 주소 0…max만 봅니다. MMU가 CPU와 버스 사이의 경로에 있기 때문에 물리 주소를 위조할 방법이 없습니다.

실제 시스템은 훨씬 정교한 MMU(페이징, 세그멘테이션)를 쓰지만 원리는 같습니다 — 프로그램이 쓰는 것과 DRAM에 도달하는 것 사이에 하드웨어가 강제하는 간접층(indirection).

### 2.5 동적 적재 (Dynamic Loading)

**동적 적재**: 루틴은 **호출될 때만** 메모리에 적재됩니다. 메인 프로그램은 메모리에 있고, 드물게 쓰이는 루틴(예: 잘 일어나지 않는 에러 핸들러)은 필요할 때까지 디스크에 머무릅니다.

```text
  메인 프로그램 (메모리)
   ├── routine_A()        이미 적재
   ├── routine_B()        호출 시 적재
   └── error_handler()    에러 발생 시에만 적재  (디스크)
```

흥미롭게도 **OS 지원 없이** 프로그래머가 구현 가능합니다. 전체 프로그램은 크지만 *상주* 크기는 작게 유지됩니다.

### 2.6 동적 연결과 공유 라이브러리

**정적 연결(static linking)** 은 링크 시점에 라이브러리 코드를 실행 파일에 복사 — 모든 프로그램이 `libc` 사본을 들고 다님. **동적 연결(dynamic linking)** 은 바인딩을 *실행 시점*까지 미루고 메모리상의 한 사본을 모든 프로세스가 공유.

| 측면 | DLL / 공유 라이브러리 이점 |
|------|--------------------------|
| 메모리 절약 | 물리 사본 하나가 모든 프로세스에 서비스 |
| 디스크 절약 | 모든 실행 파일에 라이브러리 코드 중복 없음 |
| 쉬운 업데이트 | 라이브러리 파일만 교체; 사용자 측 재빌드 불필요 |
| 버전 관리 | 여러 버전 공존; 프로그램은 호환되는 것을 선택 |

각 프로그램의 작은 **stub**이 첫 호출에서 라이브러리 루틴을 찾고, 이후 호출 사이트를 직접 점프하도록 패치합니다. 동적 연결은 라이브러리가 프로세스의 보호된 주소 공간에 매핑되어야 하므로 OS 지원이 필요합니다.

```text
  Static Linking                    Dynamic Linking
  +-----------------+               +-----------------+
  | Program A       |               | Program A       |
  | +-------------+ |               | (stub -> libc)  |
  | | libc copy 1 | |               +-----------------+
  | +-------------+ |                        |
  +-----------------+               +-----------------+
  | Program B       |               | Program B       |
  | +-------------+ |               | (stub -> libc)  |
  | | libc copy 2 | |               +-----------------+
  | +-------------+ |                        |
  +-----------------+               +-----------------+
                                    | libc (1 copy)   | <- 공유
                                    +-----------------+

  libc 2MB × 40 프로세스           libc 2MB × 1 사본
  = 80MB 메모리                    = 2MB 메모리
```

## 3. 연속 메모리 할당

### 3.1 개요와 메모리 보호

역사적으로 가장 단순한 방식: 각 프로세스는 **연속된 하나의 물리 메모리 블록**을 받음.

- 메모리는 **OS 영역**(보통 인터럽트 벡터 근처의 높은 주소)과 **사용자 영역**으로 나뉨.
- **홀(hole)** 은 빈 블록. 다양한 크기의 홀이 사용자 영역에 흩어짐.
- 프로세스가 도착하면 OS는 충분한 크기의 홀을 선택.

보호는 **재배치 레지스터 + Limit 레지스터**로 강제: 모든 논리 주소에 재배치 레지스터를 더한 뒤 Limit 레지스터로 검사. OS 디스패처가 컨텍스트 전환 시 두 레지스터 값을 적재.

### 3.2 가변 분할 방식

프로세스마다 필요에 맞는 **가변 크기 분할**을 부여. 프로세스가 종료되면 홀이 생기고, **인접 홀은 합쳐(coalesce)** 더 큰 빈 블록을 형성.

문제: 현재 홀 목록이 주어졌을 때, 크기 n의 새 요청을 **어떤 홀에** 할당할 것인가? — **동적 저장소 할당 문제(Dynamic Storage-Allocation Problem)**.

### 3.3 First-fit / Best-fit / Worst-fit

| 전략 | 방법 | 특성 |
|------|------|------|
| **First-fit** | 충분한 크기의 첫 홀에 할당 | 빠름(첫 일치에서 중단); 일반적으로 좋은 사용률 |
| **Best-fit** | 충분한 크기 중 가장 작은 홀에 할당 | 잔여 최소, 전체 스캔 필요; 작은 홀이 많이 생성됨 |
| **Worst-fit** | 가장 큰 홀에 할당 | 가장 큰 잔여 홀을 남김; 직관과 달리 *최악*인 경우가 많음 |

**경험적 결과**: First-fit과 Best-fit이 Worst-fit을 이김; First-fit이 비슷한 사용률에서 Best-fit보다 빠름.

### 3.4 할당 예제

홀(순서대로): 300KB, 600KB, 350KB, 200KB, 750KB, 125KB.
요청 순서: P1 = 115KB, P2 = 500KB, P3 = 358KB, P4 = 200KB, P5 = 375KB.

```text
First-fit:
  P1(115) -> 300KB hole       P2(500) -> 600KB hole
  P3(358) -> 750KB hole       P4(200) -> 200KB hole
  P5(375) -> 잔여 홀 (185, 100, 392, 125) 중 392에 들어감

Best-fit:
  P1(115) -> 125KB hole       P2(500) -> 600KB hole
  P3(358) -> 750KB hole       P4(200) -> 200KB hole
  P5(375) -> 392KB hole

Worst-fit:
  P1(115) -> 750KB hole       P2(500) -> 635KB hole
  P3(358) -> 실패
```

요점은 정확한 수치가 아니라 **할당 전략이 어떤 요청 시퀀스가 성공하는지를 의미 있게 바꿈**.

### 3.5 외부 단편화 vs 내부 단편화

서로 다른 두 형태의 낭비:

- **외부 단편화** — 총 사용 가능 메모리는 충분하지만 **단일 홀이 부족** — 연속성 문제.
- **내부 단편화** — 시스템이 고정 *단위*로 할당하고, 요청이 단위보다 작으면 할당 *내부*의 잔여 공간이 낭비됨.

```text
외부 단편화                         내부 단편화
+--------+                         +---------+
|  P1    |                         |Allocated|  <- 요청: 3.5KB
+--------+                         | (3.5KB) |
|  hole  | 50KB                    |---------|
+--------+                         |  낭비   |  <- 0.5KB
|  P2    |                         | (0.5KB) |
+--------+                         +---------+  <- 할당 단위: 4KB
|  hole  | 30KB
+--------+
|  P3    |  요청: 70KB  총 free: 80KB  최대 연속: 50KB → 실패
```

> **50% 규칙 (Knuth)**: first-fit으로 N개의 블록이 할당되었을 때, 통계적으로 약 0.5N 블록이 단편화로 잃어버린다는 분석. 실제로 장기 가동되는 가변 분할 시스템에서 **약 1/3** 의 메모리가 외부 단편화로 사용 불가능해짐. 이것이 페이징으로의 전환을 촉발한 경험적 고통.

### 3.6 압축 (Compaction)

외부 단편화의 무차별 해결책: **점유된 메모리를 한쪽 끝으로 옮기고** 모든 빈 공간을 반대 끝으로 모아 하나의 큰 홀을 만듦.

```text
Before                  After
+--------+              +--------+
|  P1    |              |  P1    |
+--------+              +--------+
|  hole  |              |  P2    |
+--------+              +--------+
|  P2    |  -- 이동 -> |  P3    |
+--------+              +--------+
|  hole  |              |        |
+--------+              |  hole  |  <- 하나의 큰 연속 블록
|  P3    |              |        |
+--------+              +--------+
```

압축은 **실행 시점 바인딩**에서만 가능(프로세스의 base 레지스터를 즉석에서 변경 가능). 매우 비싸기도 함 — 모든 프로세스를 복사하고 base 레지스터를 갱신하는 동안 시스템이 멈춤.

더 나은 답은 **페이징** — 애초에 외부 단편화가 생길 수 없도록 비연속 할당을 함.

## 4. 페이징 — 기본 방법

### 4.1 페이징 개요

**페이징**은 논리 메모리와 물리 메모리를 분리합니다:

- 물리 메모리는 고정 크기 **프레임(frame)** 으로 분할.
- 논리 메모리는 같은 크기의 **페이지(page)** 로 분할.
- **페이지 테이블**이 페이지 번호 → 프레임 번호를 (프로세스별로) 매핑.

결과:

- **외부 단편화 제거** — 임의의 빈 프레임이 임의의 페이지를 담을 수 있음.
- **내부 단편화는 남음** — 각 프로세스의 마지막 페이지가 거의 100% 채워지지 않음; 평균 반 페이지 낭비.

### 4.2 주소 변환 원리

페이지 크기가 2^n, 논리 주소 공간이 2^m 바이트라 하자(32비트 아키텍처에서는 m = 32, x86-64에서는 m = 48 — §9.3 참조). 주소를 분할:

```text
Logical Address (m bits)
+-------------------+------------------+
| Page number (p)   | Page offset (d)  |
|    (m-n bits)     |    (n bits)      |
+-------------------+------------------+

변환
1. p를 페이지 테이블의 인덱스로 사용.
2. PT[p]에서 프레임 번호 f 읽기.
3. 물리 주소 = (f << n) | d.

Physical Address
+-------------------+------------------+
| Frame number (f)  | Page offset (d)  |
+-------------------+------------------+
```

**핵심 불변**: 페이지 오프셋 `d`는 그대로 — 페이지와 프레임의 크기가 같으므로 내부 위치가 동일.

> **`(f << n) | d` 가 `f × page_size + d` 와 같은 이유** (컴퓨터구조의 비트 연산 복습): 정수를 왼쪽으로 `n` 비트 시프트하면 2ⁿ을 곱한 것과 같음. 페이지 크기 = 2ⁿ 이므로 `f << n` 은 정확히 `f × page_size`. `f << n` 의 하위 `n` 비트는 모두 0이므로 `n`-비트 오프셋 `d` 를 OR로 끼워 넣는 것은 더하기와 동치. 결국 주소를 (페이지 #, 오프셋)으로 분할하는 것은 페이지 크기로 나눈 *몫과 나머지*를 산술 없이 배선만으로 얻는 것과 같습니다.

### 4.3 페이징 하드웨어와 프리 프레임 관리

모든 CPU 메모리 참조마다 MMU가 **현재 프로세스의 페이지 테이블**을 참조해 변환. 각 프로세스는 자기 테이블을 유지 — 컨텍스트 전환은 페이지 테이블 베이스 주소(PTBR) 적재를 포함.

OS는 **프리 프레임 리스트**를 유지. 프로세스가 생성되거나 확장될 때 리스트에서 프레임을 꺼내 페이지 테이블에 기록.

### 4.4 내부 단편화와 페이지 크기 트레이드오프

페이지 크기 2,048 bytes, 프로세스 크기 72,766 bytes:

- 72,766 / 2,048 = 35 페이지 + 1,086 bytes
- OS는 **36 프레임** 할당 → 내부 단편화 = 2,048 − 1,086 = **962 bytes**
- 최악: 크기 *(n × page_size) + 1 byte* 인 프로세스는 n+1 프레임 필요 → 거의 한 프레임 낭비.

페이지 크기 트레이드오프:

| 작은 페이지 | 큰 페이지 |
|-------------|----------|
| 내부 단편화 적음 | 내부 단편화 많음 |
| 페이지 테이블 큼 | 페이지 테이블 작음 |
| I/O 횟수 많음 | 대량 I/O에 효율적 |

현대 시스템은 보통 기본 **4KB** 또는 **8KB**, 메모리 집약 워크로드를 위한 옵션으로 **2MB**·**1GB** 거대 페이지(huge page).

## 5. 페이징 하드웨어 지원 — TLB

### 5.1 페이지 테이블을 어디에 둘 것인가

두 가지 구현 전략:

**방법 1 — 전용 레지스터.** 페이지 테이블 전체를 특수 하드웨어 레지스터에 저장.
- 장점: 매우 빠른 조회.
- 단점: 페이지 테이블 항목이 수백 개일 때만 가능 — 확장성 없음.

**방법 2 — PTBR (Page-Table Base Register).** 페이지 테이블을 메인 메모리에 두고, PTBR이 시작 주소 지정.
- 장점: 컨텍스트 전환은 PTBR만 갱신.
- 단점: **모든 메모리 참조가 메모리 접근 2회를 비용으로** — PTE 1회 + 실제 데이터 1회. 실효 메모리 지연이 두 배.

### 5.2 TLB — Translation Look-aside Buffer

**TLB**는 최근 변환을 캐시하는 작고 매우 빠른 **연관 메모리(associative memory)**.

> **"연관 메모리"가 무엇인가** (컴퓨터구조 — **Content-Addressable Memory / CAM** 이라고도 부름): 일반 RAM은 *주소*를 받아 그 주소의 *값*을 돌려줍니다. CAM은 반대 — *값*(여기서는 페이지 번호)을 주면 전용 비교기 하드웨어가 **저장된 모든 키를 한 사이클에 병렬 비교**, 일치하는 항목의 슬롯 인덱스(와 저장된 데이터)를 반환. 같은 하드웨어 발상이 캐시 태그 배열의 기반입니다. 병렬 비교 덕분에 TLB 조회가 실질 무비용이지만, 항목당 트랜지스터 비용도 큼 — 그래서 TLB 크기가 수십~수천 항목에 그칩니다.

- 크기: 보통 32~1,024 항목.
- 각 항목: **(페이지 번호, 프레임 번호)** + 보호 비트.
- 조회는 입력 페이지 번호를 **모든 키와 병렬 비교** — 단일 사이클.
- CPU 파이프라인 내부에 위치 → TLB 히트는 관측 가능한 지연이 0.

```text
TLB Hit / Miss 과정:

CPU --p,d--> TLB에서 p 검색
              |
          +---+---+
          |       |
       hit       miss
          |       |
        f 획득  메모리의 페이지 테이블 워크,
          |    (p,f)를 TLB에 삽입
          |       |
          +---+---+
              |
          f + d --> 물리 주소 --> DRAM
```

TLB가 가득 차면 항목을 추방(LRU, round-robin, random). 일부 항목은 **wired down**(추방 불가) — 보통 커널 코드/데이터, OS 응답성 유지 목적.

### 5.3 ASID — Address-Space Identifier

기본적으로 컨텍스트 전환은 **TLB 전체를 플러시**해야 함. P1의 TLB 항목이 P2의 주소를 잘못 변환하기 때문. 비용: 스케줄러가 프로세스를 전환할 때마다 TLB 미스 폭주.

> **규모 감각**: 1,024 항목 TLB에 미스 1회당 ~100 ns라면 플러시 후 재워밍업에 약 **100 µs** — 바쁜 서버의 Linux 타임 슬라이스에 맞먹는 시간. 여기에 컨텍스트 전환률(흔히 초당 수천 회)을 곱하면 완화되지 않은 TLB 플러시가 만만찮은 오버헤드가 됩니다. 이 간극을 메우려고 ASID가 설계되었습니다.

**ASID**는 각 TLB 항목에 프로세스 식별자를 태그:

```text
+------+-----+-----+
| ASID |  p  |  f  |
+------+-----+-----+
|  P1  |  3  |  7  |   프로세스 1의 페이지 3 -> 프레임 7
|  P2  |  3  | 12  |   프로세스 2의 페이지 3 -> 프레임 12
+------+-----+-----+
```

조회 시 페이지 번호 **와** 현재 프로세스 ASID가 모두 일치해야 함. 여러 프로세스의 항목이 TLB에 공존 가능 → 컨텍스트 전환 비용이 훨씬 저렴.

### 5.4 유효 접근 시간 (EAT)

TLB 성능을 정량화. 정의:

- α = TLB **히트율**
- M = 메인 메모리 접근 시간
- T = TLB 접근 시간

```text
EAT = α · (T + M) + (1 − α) · (T + 2M)
```

> **식의 유도** (두 사건의 기댓값):
> - **히트**(확률 α): T (TLB 조회 자체, 항상 지불) + M (실제 데이터 메모리 접근 1회) → `T + M`.
> - **미스**(확률 1 − α): T (실패한 TLB 조회) + M (페이지 테이블 항목 메모리 읽기) + M (이어서 실제 데이터 읽기) → `T + 2M`.
> - 각 사례에 확률을 곱해 더함. 이산확률의 기댓값(expected value) 표준 기법입니다.

**수치 예** (M = 100ns, T = 10ns):

```text
α = 0.80:  EAT = 0.80·110 + 0.20·210 = 88 + 42  = 130ns  (raw 대비 30% 저하)
α = 0.98:  EAT = 0.98·110 + 0.02·210 = 107.8 + 4.2 = 112ns  (~ 2% 저하)
α = 0.99:  EAT = 0.99·110 + 0.01·210 = 108.9 + 2.1 = 111ns  (~ 1% 저하)
```

교훈: 높은 TLB 히트율(≥98%, *공간*·*시간* 지역성 덕분에 실제 워크로드에서 쉽게 달성)은 페이징을 실효 무비용에 가깝게 만듦.

## 6. 페이징 — 보호와 공유 페이지

### 6.1 보호 비트와 유효/무효 비트

페이지 테이블 항목마다 추가 비트:

**권한 비트**:
- **R/W** — 읽기 전용 vs 읽기/쓰기.
- **X** — 실행 가능.
- 보호 위반 참조는 하드웨어 트랩 발생; OS는 보통 `SIGSEGV` 전달.

**유효/무효 비트**:
- *유효(Valid)* — 페이지가 프로세스의 논리 주소 공간에 속함.
- *무효(Invalid)* — 이 프로세스가 사용하지 않는 페이지; 참조 시 트랩.

이 비트는 프로세스의 실제 주소 공간 사용량이 이론적 최대보다 훨씬 작은 일반적 상황을 처리. 페이지 테이블을 실제 사용 부분으로 축소하기 위해 하드웨어는 **PTLR (Page-Table Length Register)** 를 노출해 실제 길이를 제한.

### 6.2 공유 페이지

같은 코드를 실행하는 두 프로세스는 읽기 전용 텍스트 페이지의 프레임을 **공유** 가능, 단 코드가 **재진입(reentrant)** 이어야 함(자기 수정 명령어 없음, 프로세스별 전역 상태 없음).

```text
예 — libc text 2MB를 쓰는 40명 사용자의 텍스트 에디터:
  공유 안 함: 40 × 2MB = 80MB
  공유:      1 × 2MB =  2MB + 40 × (개인 데이터)
```

일반적인 공유 대상:
- **시스템 라이브러리** (libc, 동적 링커, 커널 모드 공유 코드).
- **컴파일러, 에디터, 윈도우 시스템** — 시분할 서버에서 다수 사용자가 동시 사용.
- **데이터베이스 엔진** — 많은 클라이언트 연결.
- **공유 메모리** — 명시적 IPC.

OS는 보호 비트로 읽기 전용 속성을 *강제*. 각 프로세스의 개인 데이터/스택 페이지는 별도 유지.

## 7. 페이지 테이블 구조

### 7.1 단일 페이지 테이블의 확장성 문제

가상 페이지당 한 항목의 평탄한 페이지 테이블은 작은 주소 공간에서는 좋지만 빠르게 부풀어 오릅니다.

```text
32-bit 주소, 4 KB (= 2^12) 페이지:
  항목 = 2^32 / 2^12 = 2^20 ≈ 1,048,576
  항목당 4 bytes → 페이지 테이블 = 프로세스당 4 MB

64-bit 주소, 4 KB 페이지:
  항목 = 2^64 / 2^12 = 2^52 ≈ 4·10^15
  항목당 8 bytes → 페이지 테이블 ≈ 32 PB (!)
```

한 프로세스의 페이지 테이블 4 MB를 연속 할당하기도 어렵고, 32 PB는 말할 것도 없습니다. 세 가지 고전적 해결책:

1. **계층적 페이징(hierarchical paging)** (다단계).
2. **해시 페이지 테이블(hashed page tables)**.
3. **역 페이지 테이블(inverted page tables)**.

### 7.2 계층적 페이징

페이지 테이블 자체를 페이징.

```text
Logical address (32-bit, 4KB 페이지, 2단계):
+----------+----------+----------+
|   p1     |   p2     |    d     |
| 10 bits  | 10 bits  |  12 bits |
+----------+----------+----------+

  p1  →  외부 페이지 테이블 (1024 항목)
  p2  →  내부 페이지 테이블 (1024 항목)
  d   →  페이지 오프셋
```

이것이 고전적 **forward-mapped page table**. 큰 이득: 사용하지 않는 주소 공간 영역의 내부 페이지 테이블은 **메모리에 적재되지 않음**. 자료구조 관점에서 본 메커니즘: 외부 테이블은 1024개 포인터의 배열이고, 각 포인터는 내부 페이지 테이블(4 KB, 1024 항목)을 가리키거나 **null/invalid**. null 외부 항목을 워크하면 OS로 트랩("매핑 없음"); 해당 4 MB 가상 주소 공간은 페이지 테이블 자체에 물리 메모리를 *전혀* 쓰지 않음. 보통 프로세스는 4 GB 가상 공간의 작은 부분만 사용하므로 대부분의 외부 항목이 null로 남고, 실제 쓰이는 가지에만 비용을 지불합니다.

### 7.3 64비트에서 계층적 페이징의 한계

순진한 2단계를 64비트 주소 공간에 적용:

```text
+----------+----------+----------+
|    p1    |    p2    |    d     |
|  42-bit  |  10-bit  |  12-bit  |
+----------+----------+----------+

외부 테이블 = 2^42 항목 = 2^44 bytes = 16 TB  → 여전히 불가능
```

단계를 더 추가하면 도움은 되지만 빠르게 비현실적:

```text
3단계: 외부 ≈ 16 GB → 여전히 큼
~ 7단계 필요 → 참조당 비용 급증.
```

따라서 실제 64비트 시스템은 32비트와 같은 균일 다단계 페이지 테이블을 쓰지 않음. **희소 주소 공간 사용을 활용한 4단계 페이징**(x86-64) 또는 아래의 대안을 사용.

### 7.4 해시 페이지 테이블

테이블을 **가상 페이지 번호의 해시**로 인덱싱:

```text
hash(p) → (p, f, next) 항목 체인
체인 워크하면서 p 일치 찾기
```

각 버킷은 충돌을 연결 리스트로 보관 — 자료구조의 표준 **분리 연쇄(separate chaining)** 충돌 해소 방식: 버킷 헤드에 저장되는 항목 형태가 `(가상 페이지, 프레임, next)` 이고 `next` 가 같은 버킷의 다음 충돌을 연결. 페이지 `p` 변환 시 `hash(p)` 를 계산하고 그 버킷의 연결 리스트를 워크하여 저장된 가상 페이지가 `p` 와 일치하는 노드에서 멈춤. 적재율(load factor)을 낮게 유지하면 평균 조회 O(1), 최악은 O(체인 길이). **희소** 주소 공간(64비트 범위 대부분이 미사용)에 적합 — 테이블 크기가 실제 사용 페이지에 비례.

**클러스터 페이지 테이블(clustered page table)** 은 인접 페이지 여러 개의 매핑을 한 항목에 저장해 해시 오버헤드를 분산. 64비트 희소 주소 공간에 특히 잘 맞음.

### 7.5 역(Inverted) 페이지 테이블

자료 구조를 뒤집어, **프레임 번호로 인덱싱되는 단일 전역 테이블** 유지(물리 프레임당 한 항목):

```text
Inverted PT:
  frame_id | (pid, virtual_page)
  -------- | ---------------------
     0     | (P3, 0x4012)
     1     | (P1, 0x000A)
     ...
```

- **장점**: 전체 크기가 물리 메모리에 비례, 가상 메모리가 아님 — 크게 절약.
- **단점**: 조회는 *가상* 주소로 하므로 OS가 전체 테이블 검색해야 함. 해법은 역 테이블 *위에 얹는* **보조 해시 테이블**: `(pid, 가상 페이지)` 를 해시해서 **프레임 인덱스**(= 역 테이블의 행)를 얻음. §7.4의 해시 페이지 테이블과는 구조적으로 다름 — 거기서는 해시가 페이지 테이블을 *대체*하지만, 여기서는 프레임으로 인덱싱되는 별개 구조의 조회를 가속할 뿐입니다.
- **한계**: 각 프레임이 정확히 하나의 (pid, vpage)로 매핑. 같은 프레임이 여러 프로세스 페이지 테이블에 나타나는 **공유 페이지**를 표현하기 어려움.

### 7.6 비교

| 구조 | 장점 | 단점 | 적합 |
|------|------|------|------|
| 단일 | 단순, 빠름 | 프로세스당 거대 | 작은 주소 공간 |
| 계층 | 미사용 영역 스킵 | 64비트에서 단계 폭주 | 32비트 시스템, x86-64 (4단계) |
| 해시 | 크고 희소한 VA 처리 | 해시 충돌 오버헤드 | 64비트, 희소 VA |
| 역 | 크기 = O(물리 메모리) | 검색 느림, 공유 어려움 | 큰 물리 RAM |

## 8. 스와핑

### 8.1 표준 스와핑

전체 프로세스를 메모리에서 **백킹 스토어**(이 목적으로 디스크에 예약된 영역)로 **swap out**; 다시 스케줄될 때 **swap in**.

- 시스템 과부하 시 다른 프로세스를 위한 메모리 확보.
- **매우 비쌈**: 전체 프로세스 디스크 전송에 초 단위 소요, 타임 슬라이스보다 훨씬 큼.
- 현대 OS에서는 거의 쓰지 않음.

### 8.2 페이지 단위 스와핑

현대의 대응물은 **페이지 단위**로 작동:

- **Page out**: 특정 페이지를 메모리에서 백킹 스토어로.
- **Page in**: 특정 페이지를 백킹 스토어에서 요청 시 적재.
- *필요한* 페이지만 이동, 전체 프로세스 아님. 더 빠르고 정밀하고 효율적.

이것이 **가상 메모리**의 기반 — [[concepts_lecture_w12]]에서 자세히 다룸.

### 8.3 모바일 시스템의 스와핑

대부분의 모바일 OS는 스와핑을 **지원하지 않음**. 이유:

- **플래시는 쓰기 수명 제한** — 스와핑이 마모시킴.
- 플래시와 RAM 간 **처리량이 데스크탑 SSD보다 훨씬 낮음**.
- **저장 용량**이 애초에 작음.

대안:

- **iOS** — 메모리 부족 시 시그널로 앱에 메모리 자발적 반납 요청; 비협조 앱은 종료.
- **Android** — 프로세스 종료 전 앱의 직렬화된 상태를 플래시에 저장; 새 프로세스이지만 사용자는 빠른 재시작으로 인지.

## 9. 실제 아키텍처

### 9.1 IA-32

32비트 x86. **세그멘테이션 + 페이징** 조합 사용.

- **논리 주소**는 (세그먼트 셀렉터, 오프셋) 쌍.
- 세그멘테이션 하드웨어가 **선형 주소(linear address)** 로 매핑.
- 페이징이 선형 주소를 물리 주소로 매핑.
- 페이지 크기: **4 KB** (PSE로 **4 MB** 거대 페이지).

### 9.2 IA-32 PAE

**PAE (Page Address Extension)** 로 32비트 프로세서가 4GB 이상의 물리 RAM 접근 가능.

- 페이지 테이블 항목 **32비트 → 64비트** 확장.
- 물리 프레임 번호 20→**24비트** (+ 12비트 오프셋 = 36비트 물리 주소 ⇒ 64GB).
- Linux와 macOS는 PAE 지원; 32비트 일반 사용자 Windows는 어차피 4GB로 제한.

### 9.3 x86-64

64비트 확장 아키텍처 (원래 AMD64, Intel도 채택).

- **48비트 가상 주소 공간** = 256 TB. (상위 미사용 비트는 비트 47의 부호 확장이어야 함 — "canonical addresses". *의미*: 64비트 포인터 중 정보를 담는 것은 하위 48비트뿐이며, 비트 48–63은 모두 비트 47과 일치해야 함 — 즉 **전부 0 또는 전부 1**. 비-canonical 포인터를 역참조하면 #GP 일반 보호 폴트 발생. 이 설계는 미래의 CPU가 주소 공간을 확장해도 상위 비트를 쓰레기로 채우던 소프트웨어가 깨지지 않도록 미리 비트를 예약한 것입니다.)
- **52비트 물리 주소**(PAE형 확장) = 4 PB.
- 페이지 크기: **4 KB**, **2 MB**, **1 GB**.
- 4단계 페이지 테이블 (PML4 → PDPT → PD → PT). 최신 칩은 57비트 VA를 위해 5단계 추가.
- 세그멘테이션은 사실상 비활성 (평탄 64비트 주소 공간).

### 9.4 ARMv8

모바일/임베디드 분야의 지배적 64비트 아키텍처.

- 최대 **4단계** 계층적 페이징 (translation granule 4 KB, 16 KB, 64 KB).
- **2단계 TLB**: 명령어/데이터 별도 micro-TLB + 더 큰 main TLB.
- **ASID** 하드웨어 지원 — 컨텍스트 전환 시 플러시 회피.

### 9.5 비교

| 특성 | IA-32 | x86-64 | ARMv8 |
|------|-------|--------|-------|
| 주소 크기 | 32-bit | 48-bit 가상 | 48-bit 가상 |
| 페이징 단계 | 2 (PAE 시 3) | 4 | 최대 4 |
| 페이지 크기 | 4 KB, 4 MB | 4 KB, 2 MB, 1 GB | 4 KB, 16 KB, 64 KB |
| 세그멘테이션 | 사용 | 사실상 비활성 | 없음 |
| 물리 메모리 | 4 GB (PAE 시 64 GB) | 4 PB (52-bit) | 최대 48-bit |
| TLB | 있음 | 다단계 | 2단계 |

## 10. 실습 — 메모리 할당 시뮬레이터

### 10.1 실습 목표와 자료 구조

**목표**: 메모리 블록 리스트를 관리하고 First-fit, Best-fit, Worst-fit을 같은 요청 시퀀스로 비교.

요구 사항:
- 메모리를 블록 연결 리스트로 표현 (start, size, status, owner).
- 세 전략으로 할당.
- 해제: 인접 free 블록 합치기.
- 출력: 레이아웃 스냅샷, 할당 성공/실패, 외부 단편화 크기.

```c
typedef struct Block {
    int start;        // 시작 주소
    int size;         // 블록 크기
    int allocated;    // 0 = free, 1 = allocated
    char process[10]; // 소유 프로세스 이름
    struct Block* next;
} Block;
```

> **왜 단일 연결 리스트인가?** (자료구조 관점): 주소순 정렬 연결 리스트는 (1) 홀을 찾으려 리스트를 워크 — 순차 접근, O(n); (2) 블록 분할 — 포인터 수정, O(1); (3) 해제 후 인접 합치기 — 이웃 포인터 확인, O(1) 의 자연스러운 선택. **First-fit 은 첫 일치에서 워크를 단락 종료**하므로 단편화된 리스트에서 기댓값이 n보다 훨씬 작음; **Best-fit과 Worst-fit** 은 가장 작은/큰 충분한 홀을 확실히 찾으려면 매번 전체 리스트를 스캔 — 항상 O(n). 이 비대칭이, 단편화 품질과 별개로 First-fit이 다른 둘보다 보통 빠른 알고리즘적 이유입니다.

### 10.2 실습 함수와 입출력 예

```c
Block* first_fit(Block* head, int size);   // 충분한 첫 홀
Block* best_fit(Block* head, int size);    // 충분한 가장 작은 홀
Block* worst_fit(Block* head, int size);   // 가장 큰 홀
void   release(Block* head, char* process);// 해제 + 인접 합치기
void   print_memory(Block* head);
```

예시 트레이스:

```text
Input
  Memory size: 1024
  Allocate P1: 200
  Allocate P2: 350
  Allocate P3: 150
  Release P2
  Allocate P4: 300

Output (First-fit)
  [  0- 199] P1 (200)
  [200- 549] FREE (350)      <- P2 해제 후
  [550- 699] P3 (150)
  [700-1023] FREE (324)

  Allocate P4(300) → First-fit → [200-499] P4
  [  0- 199] P1 (200)
  [200- 499] P4 (300)
  [500- 549] FREE (50)       <- 잔여 홀
  [550- 699] P3 (150)
  [700-1023] FREE (324)

  외부 단편화 = 50 + 324 = 374 bytes
```

확장: 무작위 요청 스트림을 공급, 전략별 평균 외부 단편화 비교, 50% 규칙 경험적으로 검증.

| MMU | 하드웨어 변환; base/limit으로 보호, 재배치 가능 |
| 동적 적재/연결 | 루틴 지연 적재; 공유 라이브러리는 메모리 절약·업데이트 용이 |
| 연속 할당 | First/Best/Worst-fit; 실전에서 First-fit이 보통 승 |
| 단편화 | 외부 (연속성 부족) vs 내부 (단위 > 요청); 약 1/3 손실 (50% 규칙) |
| 페이징 | 고정 크기 페이지 → 프레임; 외부 단편화 제거 |
| TLB | 변환 캐시; α≥98%면 페이징 거의 무비용 |
| ASID | 컨텍스트 전환 시 TLB 플러시 회피 |
| 페이지 테이블 구조 | 단일 / 계층 / 해시 / 역 |
| 64비트 | 계층만으로 불충분; 4단계 + 희소, 또는 해시 |
| 스와핑 | 표준 (전체 프로세스) vs 페이지 단위; 모바일은 회피 |
| 아키텍처 | IA-32 (세그+페이지), x86-64 (4단계), ARMv8 (4단계 + ASID) |

## 점검 문제

1. CPU가 메인 메모리는 직접 접근 가능하지만 디스크는 그렇지 않은 이유는?

   > **답:** CPU의 명령어 집합은 메모리 버스에서 레지스터나 메인 메모리에 주소를 발행 — 이것이 아키텍처가 직접 접근 가능하다고 노출하는 저장소 계층입니다. 디스크는 별도 I/O 서브시스템(컨트롤러, DMA, 드라이버) 뒤에 위치; 도달하려면 명시적 OS 매개 전송 필요(~수백만 사이클). 디스크 데이터를 실행하거나 읽으려면 OS가 먼저 메모리에 **적재**해야 함.

2. 논리 주소와 물리 주소를 구분하시오. 어느 바인딩 방식에서 둘이 동일한가?

   > **답:** **논리 주소**는 CPU가 프로그램 실행 중 생성; 프로세스는 자기만의 공간 0…max를 봄. **물리 주소**는 DRAM에 도달. **컴파일 시점** 또는 **적재 시점** 바인딩에서 둘이 동일(런타임 변환 없음). **실행 시점** 바인딩에서는 MMU가 모든 참조를 변환하고 둘은 갈라짐.

3. MMU의 역할은? Base+Limit MMU가 프로세스를 어떻게 보호하는가?

   > **답:** MMU는 CPU와 메모리 버스 사이에 위치해 **모든 논리 주소를 물리 주소로 변환**. 가장 단순한 base+limit 형태에서 (a) 논리 주소를 **limit**과 비교해 초과 시 트랩, (b) **base** 레지스터를 더해 물리 주소 산출. base/limit은 **커널 모드**에서만 수정 가능 → 사용자 프로세스는 샌드박스 탈출 불가.

4. 정적 연결과 동적 연결을 비교. 공유 라이브러리가 *공유*라 불리는 이유는?

   > **답:** **정적 연결**은 빌드 시점에 라이브러리 코드를 각 실행 파일에 복사 — 프로세스마다 자기 사본. **동적 연결**은 라이브러리 호출을 stub으로 남기고; 실행 시점에 로더가 메모리상의 단일 사본을 매핑하고 stub을 패치. 라이브러리가 프로세스 간 *공유* — 한 세트의 물리 프레임이 코드(보통 읽기 전용)를 담고 모든 호출자가 사용. 메모리·디스크 절약, 한 번의 업데이트가 모든 호출자 수정.

5. 같은 홀 리스트에서 First-fit, Best-fit, Worst-fit을 따라가시오. Worst-fit이 보통 최악인 이유는?

   > **답:** First-fit은 충분한 첫 홀을 찾으면 멈춤. Best-fit은 전체를 스캔해 충분한 가장 작은 홀 선택; Worst-fit은 가장 큰 홀 선택. Worst-fit의 직관 — *"가장 큰 잔여 홀을 남기자"* — 은 역효과: 가장 큰 홀을 먼저 파괴해 미래의 큰 요청에 맞는 유일한 홀을 없앰. First-fit과 Best-fit은 큰 홀을 보존; First-fit은 전체 스캔이 필요 없어 더 빠름.

6. 외부 단편화와 내부 단편화를 구분. 50% 규칙을 진술.

   > **답:** **외부 단편화** — 총 free 메모리는 충분한데 단일 홀이 부족; 연속성 문제. **내부 단편화** — 할당 단위가 요청보다 크고 할당 *내부*의 잔여가 낭비됨. **50% 규칙** (Knuth의 first-fit 분석): N개의 블록이 할당되면 통계적으로 약 0.5N 추가 블록이 외부 단편화로 손실 — 총 메모리의 약 1/3이 사용 불가능해짐. 이것이 페이징으로의 전환 동기.

7. 페이징 주소 변환을 따라가시오. 오프셋이 변환되지 않고 그대로 통과하는 이유는?

   > **답:** 논리 주소를 `(page #, offset)`으로 분할. page #로 페이지 테이블을 인덱싱해 **frame #** 획득. 물리 주소 = `(frame # << offset_bits) | offset`. **오프셋은 그대로** — 페이지와 프레임 크기가 같으므로 페이지 내 위치가 프레임 내 위치와 동일. 상위 페이지 번호만 변환됨.

8. TLB가 해결하는 문제와 그 방법은? ASID가 추가로 무엇을 해결하는가?

   > **답:** TLB 없으면 모든 메모리 참조 비용이 메모리 접근 *2회*(PTE 읽기 + 데이터 읽기). TLB는 최근 (페이지, 프레임) 변환을 CPU 파이프라인 내부의 연관 메모리에 캐시; 히트는 0 사이클. **ASID**는 각 TLB 항목에 소유 프로세스를 태그하여 컨텍스트 전환에서 TLB 플러시 불필요 — 여러 프로세스 변환이 공존. ASID 없으면 매 컨텍스트 전환마다 다음 프로세스의 TLB 미스 폭주.

9. M = 100ns, T = 10ns, α = 0.99일 때 EAT 계산. 저하는 어디로 가는가?

   > **답:** `EAT = 0.99·(10+100) + 0.01·(10+200) = 108.9 + 2.1 = 111ns`. raw 메모리(100ns) 대비 약 11ns(11%) 저하; 그중 10ns는 모든 참조마다 지불되는 TLB 조회 자체, ~1ns는 분산된 미스 비용. 99% TLB 히트율이면 페이징이 실효 무비용에 가깝다는 시사점.

10. 단일 페이지 테이블이 64비트로 확장되지 않는 이유, 균일 2단계 페이징도 작동하지 않는 이유는?

    > **답:** 단일: 2^64 / 2^12 ≈ 4·10^15 항목 × 8 bytes ≈ 프로세스당 32 PB — 불가능. 균일 2단계: `p1` 42비트, `p2` 10비트이면 *외부* 테이블만 2^42 항목 ≈ 16 TB — 여전히 불가능. 3·4단계 균일도 중간 테이블이 너무 큼. 실제 64비트는 **계층 + 희소 할당**(미사용 분기 스킵) 또는 **해시/역** 페이지 테이블 같은 대체 구조 사용.

11. 역 페이지 테이블이 잘 맞는 상황은? 주요 한계는?

    > **답:** **물리 메모리가 가상 메모리보다 훨씬 작을 때** 유리 — 역 테이블 크기가 RAM에 비례, 프로세스별 가상 공간이 아님. 주요 한계: 각 물리 프레임이 *하나의* 항목 → **공유 페이지**(여러 프로세스 가상 주소 공간에 같은 프레임이 나타남)를 모델링하기 어려움 — 보조 자료 구조나 체인 필요. 검색도 해시 없이는 느림; 조회 해시 + TLB 조합으로 완화.

12. 모바일 OS가 대부분 스와핑을 구현하지 *않는* 이유는?

    > **답:** 세 가지 이유. (1) **플래시는 쓰기 수명 제한** — 스왑 페이지가 마모. (2) **플래시→RAM 처리량이 데스크탑 SSD보다 훨씬 낮음** — 스왑 인/아웃이 느림. (3) 폰의 **저장 용량 작음**. iOS와 Android는 대신 앱 종료로 메모리 회수 — iOS는 먼저 요청 후 종료; Android는 사용자가 빠른 재시작으로 인지하도록 앱 상태를 플래시에 직렬화.

13. 페이지 크기 트레이드오프는 무엇이며 시스템이 ~4–8 KB로 수렴한 이유는?

    > **답:** 작은 페이지 → 내부 단편화 적지만 페이지 테이블 크고 (작은) I/O가 많음; 큰 페이지 → 페이지 테이블 작고 대량 I/O 효율적이지만 내부 단편화 큼. 4–8 KB 스윗스팟이 일반 데스크탑/서버 워크로드에서 균형을 잡음. 현대 시스템은 TLB 압박이 내부 단편화 낭비를 압도하는 큰 RAM 워크로드(DB, VM)를 위해 **2 MB / 1 GB 거대 페이지**를 옵션으로 제공.

14. 공유 페이지가 작동하는 방식과 읽기 전용 요건이 보호하는 것은?

    > **답:** 두 프로세스의 페이지 테이블이 특정 가상 페이지를 *같은* 물리 프레임에 매핑(예: libc text). OS는 양쪽 항목에 **읽기 전용**을 표시. 읽기 전용 강제가 공유를 *안전*하게 만듦: 어느 프로세스도 공유 프레임을 수정할 수 없으므로 다른 프로세스가 "자기 메모리"에 외부 쓰기 변경을 관측할 수 없음. 재진입 코드는 자기 텍스트에 쓸 필요가 없어 자연스럽게 들어맞음.

15. OS가 압축을 쓰려면 실행 시점 바인딩이 필요한데, 페이징에는 왜 필요 없는가?

    > **답:** 압축은 프로세스의 바이트를 새 범위로 물리적 이동; 프로세스가 계속 실행되려면 *메모리에 있는 동안* base 레지스터가 갱신되어야 함 — 정확히 실행 시점 바인딩이 가능하게 하는 것. 페이징은 *물리 연속 할당을 요구하지 않음*으로써 질문 자체를 피함 — 페이지 테이블의 간접성이 이미 임의의 가상 페이지를 임의의 물리 프레임에 둘 수 있게 하므로 압축할 게 없음.
