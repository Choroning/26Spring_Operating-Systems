## 1. 우선순위 스케줄링

### 1.1 우선순위 스케줄링 — 개념

각 프로세스에 **우선순위(정수)** 를 부여하고, **가장 우선순위가 높은** 프로세스부터 CPU를 할당합니다.

- 일반적으로 **숫자가 작을수록 높은 우선순위** (Silberschatz 교재의 관례입니다)
  - 참고: Linux는 실시간 태스크에서 이 관례를 뒤집어 사용합니다(숫자가 클수록 높음). 항상 어떤 관례를 쓰는지 확인해야 합니다.
- **선점형(Preemptive)** 과 **비선점형(Nonpreemptive)** 두 가지 모두 가능합니다:
  - **선점형**: 더 높은 우선순위 프로세스가 도착하면 현재 실행 중인 프로세스를 **강제로 선점**합니다.
  - **비선점형**: 현재 프로세스가 자발적으로 CPU를 놓을 때까지 기다린 뒤, 준비 상태 중 최고 우선순위를 선택합니다.
- **SJF는 우선순위 스케줄링의 특수한 경우**입니다 — 우선순위 = 예측된 다음 CPU 버스트의 역수(짧은 버스트 = 높은 우선순위)입니다.

> **핵심 통찰**: 우선순위 스케줄링은 하나의 알고리즘이 아니라 *프레임워크* 입니다. 우선순위를 어떻게 부여할지(외부: 사용자/관리자가 지정, 내부: 버스트 시간·메모리 사용량 등으로 계산)는 설계자의 몫입니다. SJF, SRTF, 대부분의 실시간 알고리즘도 결국 우선순위 정의가 다를 뿐, 모두 우선순위 스케줄링의 일종입니다.

### 1.2 우선순위 스케줄링 — 예시

모든 프로세스가 시각 0에 도착하며, **작은 숫자가 높은 우선순위**입니다.

| 프로세스 | 버스트 시간 | 우선순위 |
|---------|-----------|----------|
| P1 | 10 | 3 |
| P2 | 1  | 1 |
| P3 | 2  | 4 |
| P4 | 1  | 5 |
| P5 | 5  | 2 |

실행 순서(우선순위에 따라): **P2 (1) → P5 (2) → P1 (3) → P3 (4) → P4 (5)**

```text
간트 차트:
+----+--------+------------------+------+----+
| P2 |   P5   |        P1        |  P3  | P4 |
+----+--------+------------------+------+----+
0    1        6                 16      18   19
```

| 프로세스 | 대기 시간 |
|---------|--------------|
| P1 | 6 |
| P2 | 0 |
| P3 | 16 |
| P4 | 18 |
| P5 | 1 |
| **평균** | **(6+0+16+18+1)/5 = 8.2 ms** |

### 1.3 우선순위 + 라운드 로빈 (동점 처리)

**같은 우선순위**의 프로세스들 사이에서는 **라운드 로빈(RR)** 을 적용하여, 한 프로세스가 동일 우선순위 안에서 CPU를 독점하지 못하게 합니다.

| 프로세스 | 버스트 | 우선순위 |
|---------|-------|----------|
| P1 | 4 | 3 |
| P2 | 5 | 2 |
| P3 | 8 | 2 |
| P4 | 7 | 1 |
| P5 | 3 | 3 |

타임 퀀텀 = 2 ms, 작은 숫자가 높은 우선순위입니다.

```text
간트 차트:
| P4          | P2 | P3 | P2 | P3 | P2 | P3     | P1 | P5 | P1 | P5 |
0             7    9   11   13   15   16   20     22   24   26  27
```

실행 흐름:
- **우선순위 1 (P4)**: 혼자이므로 완료까지 먼저 실행합니다.
- **우선순위 2 (P2, P3)**: RR로 번갈아 실행됩니다(퀀텀 = 2).
- **우선순위 3 (P1, P5)**: 마지막으로 RR로 번갈아 실행됩니다.

> **왜 우선순위 + RR 조합인가?** 순수 우선순위만 쓰면 같은 우선순위의 프로세스들이 도착 순서대로 실행되어, 우선순위 2의 긴 프로세스 하나가 같은 우선순위의 다른 모든 프로세스를 지연시킵니다. RR을 추가하면 **각 우선순위 클래스 내에서 공정성**이 보장됩니다.

### 1.4 기아와 노화

**기아(Starvation, Indefinite Blocking)** — 우선순위 스케줄링의 병폐
- 계속 높은 우선순위 프로세스가 도착하면 낮은 우선순위 프로세스는 **영원히 실행되지 못할 수** 있습니다.
- *역사적 일화*: 1973년 MIT가 IBM 7094 메인프레임을 종료했을 때, **1967년**에 제출된 저우선순위 작업이 한 번도 실행되지 못한 채 발견되었습니다. 6년을 기다린 셈입니다.

**노화(Aging) — 표준 해결책**
- 준비 큐에서 오래 기다린 프로세스의 우선순위를 **점진적으로 올려** 줍니다.
- 예: 우선순위 범위가 0(높음) ~ 127(낮음)일 때, 1초마다 우선순위를 1씩 증가시키면 127로 시작한 프로세스가 **약 127초 ≈ 2분**에 최고 우선순위에 도달합니다.
- 아무리 낮은 초기 우선순위라도 **결국에는** 선택되는 것을 보장합니다.

> **시험 팁**: "기아"와 "노화"는 한 쌍입니다. 우선순위 기반 알고리즘을 설명할 때는 기아 문제를 반드시 언급해야 하고, 표준 해결책으로 노화를 제시합니다. 같은 패턴이 MLFQ(하위 큐 프로모션), CFS(vruntime이 대기 태스크를 자연스럽게 우대)에도 반복됩니다.

## 2. Multilevel Queue와 MLFQ

### 2.1 Multilevel Queue 스케줄링

준비 큐를 프로세스 유형에 따라 여러 개의 **분리된 큐**로 나눕니다.

```text
     가장 높은 우선순위
     +-----------------------------+
 --> | 시스템 프로세스              |  -->
     +-----------------------------+
     +-----------------------------+
 --> | 인터랙티브 프로세스          |  -->
     +-----------------------------+
     +-----------------------------+
 --> | 인터랙티브 편집 프로세스     |  -->
     +-----------------------------+
     +-----------------------------+
 --> | 배치 프로세스                |  -->
     +-----------------------------+
     +-----------------------------+
 --> | 학생 프로세스                |  -->
     +-----------------------------+
     가장 낮은 우선순위
```
*Silberschatz, Figure 5.8 — Multilevel queue scheduling*

**큐 간 스케줄링**(두 가지 일반적인 정책):
- **고정 우선순위**: 상위 큐가 **모두 비어 있을 때만** 하위 큐가 실행됩니다. *(위험: 하위 큐의 기아)*
- **타임슬라이스 할당**: 각 큐에 정해진 CPU 시간 비율을 할당합니다(예: 전경 80%, 배경 20%).

**핵심 제약**: 프로세스는 큐 사이를 **이동할 수 없습니다** — 생성 시 큐 배정이 **고정**되어 수명 동안 유지됩니다.

> **언제 Multilevel Queue를 쓰는가**: 프로세스를 생성 시점에 명확히 분류할 수 있을 때입니다(예: "이건 인터랙티브 사용자 프로그램", "이건 배치 작업", "이건 데몬"). 분류가 정확하면 강점이지만, 그렇지 못하면 치명적 약점이 됩니다 — 그래서 MLFQ가 등장합니다.

### 2.2 Multilevel Feedback Queue (MLFQ)

단순 Multilevel Queue와의 **핵심 차이**: **프로세스가 관찰된 동작에 따라 큐 사이를 이동**할 수 있습니다.

```text
                      quantum = 8
     +--------+
 --> |   Q0   | -------> CPU
     +--------+   (소진)
         |  강등
         v
                      quantum = 16
     +--------+
 --> |   Q1   | -------> CPU
     +--------+   (소진)
         |  강등
         v
                      FCFS
     +--------+
 --> |   Q2   | -------> CPU
     +--------+
```
*Silberschatz, Figure 5.9 — Multilevel feedback queues*

**이동 규칙:**
- **강등(Demotion)**: 특정 큐의 **전체 타임 퀀텀을 소진**하면 더 낮은 큐로 강등됩니다. *(해석: CPU-bound 프로세스이므로, 강등하여 짧고 반응성 높은 프로세스의 길을 막지 않도록 합니다.)*
- **승격(Promotion, Aging)**: 하위 큐에서 너무 오래 기다리면 상위 큐로 승격됩니다. *(기아 방지)*

**자연스러운 동작:**
- **I/O-bound 프로세스**는 **CPU 버스트가 짧아** 퀀텀을 소진하기 전에 CPU를 놓기 때문에 **상위(고우선순위) 큐에 머무릅니다** → 응답성이 좋습니다.
- **CPU-bound 프로세스**는 퀀텀을 전부 소진해 **하위 큐로 내려갑니다** → CPU 시간을 받긴 하지만 인터랙티브 작업을 막지 않습니다.

> **MLFQ가 강력한 이유**: 프로그래머의 힌트 없이도 I/O-bound vs CPU-bound 프로세스를 **자동으로 분류**합니다. 스케줄러가 동작을 관찰하여 학습하고, 프로세스의 큐 레벨은 사실상 "향후 CPU 버스트 길이의 학습된 추정치"가 됩니다.

### 2.3 MLFQ 설계 매개변수

MLFQ를 완전히 정의하려면 다음을 결정해야 합니다.

| 매개변수 | 설명 |
|-----------|-------------|
| 큐의 개수 | 몇 단계 우선순위를 쓸 것인가(보통 3~5개) |
| 각 큐의 알고리즘 | RR(어떤 퀀텀)? 맨 아래는 FCFS? |
| 강등 기준 | 언제 아래 큐로 내리는가(보통 퀀텀 소진) |
| 승격 기준 | 언제 위 큐로 올리는가(노화 임계값) |
| 초기 큐 배정 | 새 프로세스는 어느 큐에서 시작하는가(보통 최상위 큐) |

MLFQ는 **가장 일반적이고 유연한** 스케줄링 알고리즘이지만, 이 모든 매개변수가 서로 영향을 주어 **조정이 가장 어려운** 알고리즘이기도 합니다. 현대의 스케줄러(CFS, Windows 가변 우선순위)는 본질적으로 MLFQ 아이디어의 정제판입니다.

### 2.4 MLFQ 동작 예시

CPU 버스트가 30 ms인 프로세스 **P**가 다음과 같이 구성된 MLFQ에 들어간다고 합시다.
- **Q0**: RR 퀀텀 8, **Q1**: RR 퀀텀 16, **Q2**: FCFS(퀀텀 없음).

```text
단계 1: Q0에서 8 ms 실행 (RR, q=8)
        남은 버스트 = 22 ms, 퀀텀 소진 -> Q1로 강등

단계 2: Q1에서 16 ms 실행 (RR, q=16)
        남은 버스트 = 6 ms, 퀀텀 소진 -> Q2로 강등

단계 3: Q2에서 6 ms 실행 (FCFS)
        남은 버스트 = 0 -> 완료
```

전체 버스트 길이에 따른 분류:
- 버스트 ≤ 8 ms → **Q0** 안에서 완료(최상위 우선순위, 최고 응답성).
- 8 < 버스트 ≤ 24 ms → **Q1**까지만 내려감.
- 버스트 > 24 ms → **Q2**(FCFS)까지 내려감.

**순 효과**: MLFQ는 **짧은 프로세스를 선호**(상위 큐에서 절대 벗어나지 않음)하면서도 **긴 프로세스의 실행을 결국 보장**(Q2에서 완료)합니다.

## 3. 스레드 스케줄링

### 3.1 경쟁 범위 — PCS vs SCS

스레드 스케줄링은 **경쟁 범위(Contention Scope)** — CPU를 두고 경쟁하는 스레드 집합 — 에 따라 분류됩니다.

**PCS (Process-Contention Scope, 프로세스 경쟁 범위)**
- **사용자 레벨 스레드** 스케줄링, **스레드 라이브러리**가 수행합니다(커널이 아님).
- 같은 프로세스 **내부**의 스레드들이 단일 커널 스케줄링 단위를 놓고 경쟁합니다.
- **Many-to-One**, **Many-to-Many** 스레드 모델에서 사용됩니다.
  - *(5주차 복습: Many-to-One은 다수의 사용자 스레드를 하나의 커널 스레드에 매핑, Many-to-Many는 N개의 사용자 스레드를 M ≤ N개의 커널 스레드에 매핑합니다.)*

**SCS (System-Contention Scope, 시스템 경쟁 범위)**
- **커널 레벨 스레드** 스케줄링, **OS 커널**이 수행합니다.
- 스레드가 **시스템 내 모든 스레드**와 경쟁합니다(같은 프로세스 내만이 아님).
- **One-to-One** 모델에서 사용됩니다.
- 현대 운영체제(**Linux, Windows, macOS**)는 모두 One-to-One을 사용 → 사실상 **SCS만 지원**합니다.

> **왜 이것이 중요한가**: PCS에서는 사용자 스레드 하나가 시스템 콜에서 블록되면 *해당 프로세스의 커널 스레드 전체가 블록*되어 라이브러리가 다른 사용자 스레드를 스케줄할 수 없습니다. SCS에서는 각 스레드가 자기만의 커널 스레드를 갖기에 하나가 블록되어도 다른 스레드는 영향을 받지 않습니다. 스레드당 비용이 더 크더라도 현대 OS가 One-to-One / SCS를 선호하는 이유입니다.

### 3.2 PCS vs SCS — 비교

| 속성 | PCS | SCS |
|----------|-----|-----|
| 스케줄링 주체 | 스레드 라이브러리(사용자 공간) | OS 커널 |
| 경쟁 범위 | 단일 프로세스 내부 | 시스템 전체(모든 스레드) |
| 스레드 모델 | Many-to-One, Many-to-Many | One-to-One |
| 우선순위 조정 | 프로그래머가 설정, 라이브러리는 동적 조정 안 함 | 커널이 **동적으로 조정** 가능(예: I/O 후 부스트) |
| 타임 슬라이싱 | 같은 우선순위 스레드 간에 보장 안 됨 | 커널 타이머가 보장 |

현대 OS가 One-to-One을 사용하므로 실무에서는 **SCS만 지원**됩니다. PCS는 역사적/임베디드 시스템의 개념입니다.

### 3.3 Pthread 스케줄링 API

POSIX 스레드는 **속성(attribute)** 을 통해 스레드별로 경쟁 범위를 조회/설정할 수 있게 합니다.

```c
#include <pthread.h>
#include <stdio.h>
#define NUM_THREADS 5

int main(int argc, char *argv[]) {
    int i, scope;
    pthread_t tid[NUM_THREADS];
    pthread_attr_t attr;

    pthread_attr_init(&attr);

    /* 현재 경쟁 범위 조회 */
    if (pthread_attr_getscope(&attr, &scope) != 0)
        fprintf(stderr, "Unable to get scheduling scope\n");
    else {
        if (scope == PTHREAD_SCOPE_PROCESS)
            printf("PTHREAD_SCOPE_PROCESS\n");
        else if (scope == PTHREAD_SCOPE_SYSTEM)
            printf("PTHREAD_SCOPE_SYSTEM\n");
    }

    /* SCS(시스템 전역 경쟁) 요청 */
    pthread_attr_setscope(&attr, PTHREAD_SCOPE_SYSTEM);

    for (i = 0; i < NUM_THREADS; i++)
        pthread_create(&tid[i], &attr, runner, NULL);
    for (i = 0; i < NUM_THREADS; i++)
        pthread_join(tid[i], NULL);
    pthread_exit(0);
}
```

- `PTHREAD_SCOPE_PROCESS`는 PCS, `PTHREAD_SCOPE_SYSTEM`은 SCS를 요청합니다.
- **Linux와 macOS에서는 `PTHREAD_SCOPE_SYSTEM`만 허용**되며, `PROCESS`를 요청하면 오류를 반환합니다. 이 커널들이 One-to-One 스레드 모델을 사용하기 때문입니다.

## 4. 다중 프로세서 스케줄링

### 4.1 다중 프로세서 아키텍처

"다중 프로세서"는 이제 다양한 하드웨어 구성을 포괄합니다.

- **멀티코어 CPU** — 단일 칩 위의 여러 코어(현대의 표준).
- **멀티스레드 코어** — 코어당 여러 하드웨어 스레드(예: Intel 하이퍼스레딩).
- **NUMA 시스템** — Non-Uniform Memory Access(코어마다 메모리 접근 지연이 다름).
- **이기종 다중 프로세싱(HMP)** — 성능/전력이 다른 코어(예: ARM big.LITTLE).

두 가지 상위 레벨 스케줄링 접근:

- **비대칭 다중 프로세싱(Asymmetric Multiprocessing, ASMP)**: 단일 **마스터 프로세서**가 모든 스케줄링 결정을 내리고, 나머지는 사용자 코드만 실행합니다. *단순하지만 마스터가 병목입니다.*
- **대칭 다중 프로세싱(Symmetric Multiprocessing, SMP)**: **모든 프로세서가 독립적으로 스케줄링**합니다 — 각 프로세서가 큐에서 프로세스를 뽑아 실행합니다. *현대의 지배적 방식(Linux, Windows, macOS).*

### 4.2 SMP — 준비 큐 구성

SMP 환경에서 준비 큐를 구성하는 두 가지 주요 방식:

*Silberschatz, Figure 5.11 — Organization of ready queues*

| 방식 | 장점 | 단점 |
|----------|------|------|
| **공통(공유) 준비 큐** | 자동 부하 분산 — 한 CPU가 노는 일 없음 | 접근마다 **락 경합**, 캐시 비친화적(프로세스가 CPU 사이를 넘나들 수 있음) |
| **프로세서별 큐** | 락 불필요, 캐시 친화적(한 CPU에 머무름) | 부하 불균형 가능 — 한 CPU는 한가한데 다른 CPU는 과부하 |

**현대 SMP 시스템(Linux CFS, Windows)은 프로세서별 큐를 사용**하며, 불균형은 **부하 분산**(4.9절)으로 해결합니다.

### 4.3 멀티코어와 메모리 스톨

**메모리 스톨(Memory Stall)**
- 캐시에 없는 메모리에 접근할 때 CPU가 데이터 도착을 기다려야 하는 시간입니다.
- 캐시 미스 하나로 **수십~수백 CPU 사이클**이 소모됩니다.
- 메모리 집약적 워크로드에서는 코어가 **최대 50%의 시간을 스톨** 상태로 낭비할 수 있습니다.

*Silberschatz, Figure 5.12 — Memory stall*
*Silberschatz, Figure 5.13 — Multithreaded multicore system*

**해결책 — 하드웨어 멀티스레딩**: 한 스레드가 메모리 스톨에 진입하면, 코어는 **일할 준비가 된 다른 하드웨어 스레드로 전환**합니다. 이를 통해 실행 유닛을 쉬게 두지 않고 코어 이용률을 극적으로 높입니다.

> **핵심 개념**: 메모리 스톨은 *하드웨어 레벨*의 스케줄링 문제입니다. 코어의 해결책(CMT / SMT, 다음 절)은 OS 스케줄러가 인지하지 못하는 아래 단계에서 돌아가는 작은 스케줄러입니다. OS 입장에서는 각 하드웨어 스레드가 완전한 "논리 프로세서"로 보이며, 스톨을 알 필요가 없습니다.

### 4.4 칩 멀티스레딩 (CMT)

단일 물리 코어에 **여러 하드웨어 스레드**를 배치합니다.

*Silberschatz, Figure 5.14 — Chip multithreading*

예시:
- **Intel 하이퍼스레딩(SMT)** — 코어당 하드웨어 스레드 2개.
- **Oracle SPARC M7** — 코어당 8개 × 코어 8개 = **논리 CPU 64개**.

각 하드웨어 스레드는 OS에 **논리 프로세서**로 보입니다. 즉 하이퍼스레딩이 있는 쿼드코어 CPU에서 Linux의 `/proc/cpuinfo`에는 물리 코어가 4개뿐이어도 **CPU 8개**가 표시됩니다.

### 4.5 Coarse-grained vs Fine-grained 멀티스레딩

하드웨어가 하드웨어 스레드 간에 전환하는 시점에 관한 두 가지 전략:

| 속성 | Coarse-grained | Fine-grained |
|----------|---------------|-------------|
| 스레드 전환 조건 | **장시간 대기 이벤트**에서만(메모리 스톨 등) | **매 명령어 사이클 경계**마다 |
| 전환 비용 | **높음** — 파이프라인 플러시 필요 | **낮음** — 전환 로직이 파이프라인에 내장 |
| 장점 | 하드웨어 설계가 단순 | 매우 빠른 전환, 높은 코어 이용률 |
| 단점 | 파이프라인 플러시 오버헤드 | 하드웨어 복잡도 증가 |

> **"파이프라인(pipeline)"이란?** 현대 CPU는 명령어를 조립 라인처럼 처리합니다 — 명령어 1이 실행 단계에 있는 동안, 명령어 2는 디코딩, 명령어 3은 페치(fetch) 등 보통 5~20 단계로 깊이 구성됩니다. 이 "파이프라인" 덕분에 각 명령어가 실제로는 여러 사이클이 걸리더라도 CPU가 대략 사이클당 하나씩 명령어를 완료할 수 있습니다. **파이프라인 플러시(pipeline flush)** 란 파이프라인에서 부분적으로 처리 중이던 모든 명령어를 버리는 것입니다(예: 스레드를 전환하면 이전 스레드의 in-flight 명령어를 유지할 수 없습니다). 플러시할 때마다 그 단계들에서 이미 수행한 작업이 버려지므로 — coarse-grained 전환의 비용이 높은 이유입니다.

현대 Intel / SPARC 설계는 **fine-grained** (혹은 하이브리드) 방식을 써서 명령어 사이클 단위로 코어를 바쁘게 유지합니다.

### 4.6 두 단계의 스케줄링

**멀티스레드 멀티코어** 프로세서에서는 스케줄링이 두 단계로 일어납니다.

*Silberschatz, Figure 5.15 — Two levels of scheduling*

**Level 1 (OS 스케줄러)**: **소프트웨어 스레드**를 **논리 프로세서**(하드웨어 스레드)에 배정합니다. CFS, Windows 스케줄러 등이 담당합니다.

**Level 2 (코어 하드웨어)**: 매 사이클마다 코어 내 어느 하드웨어 스레드를 실행할지 결정합니다. 접근 방식:
- **라운드 로빈**(UltraSPARC T3): 모든 하드웨어 스레드를 공정하게 순환.
- **동적 긴급도(Dynamic urgency)**(Intel Itanium): 각 하드웨어 스레드가 긴급도(0~7)를 가지며, 스톨 중에는 상승, 명령어 완료 시 하강 — 가장 긴급한 스레드를 선택.

> **왜 두 단계인가**: OS는 사이클 단위로 결정을 내릴 수 없습니다 — 타이밍 정보도 없고, 사이클마다 시스템 콜을 하면 오버헤드가 실제 작업량을 압도합니다. 따라서 OS는 거친 단위로 배치하고, 하드웨어가 사이클 단위의 간격을 채웁니다.

### 4.7 프로세서 친화도

**친화도(Affinity)** = 프로세스가 한번 실행되기 시작한 프로세서에 **그대로 머무르려는 성향**.

**왜 필요한가:**
- 모든 CPU는 자기만의 **캐시**를 가집니다. CPU 0에서 실행되던 프로세스는 CPU 0의 캐시를 자신의 작업 집합으로 채웁니다.
- OS가 이 프로세스를 CPU 1로 옮기면 캐시 데이터가 **무효화**되어, CPU 1에서 메모리로부터 다시 가져와야 합니다(**워밍업 비용**).

| 유형 | 설명 |
|------|-------------|
| **Soft Affinity** | OS가 같은 CPU에 두려고 *시도*하지만 보장은 안 함 |
| **Hard Affinity** | 프로세스(또는 관리자)가 실행 가능한 CPU 집합을 **명시적으로 지정** — OS가 그 밖으로 이동시키지 않음 |

Linux에서 hard affinity는 `sched_setaffinity()` 시스템 콜(혹은 `taskset` 명령)로 설정합니다.

### 4.8 NUMA와 프로세서 친화도

**NUMA (Non-Uniform Memory Access)** — 각 CPU(소켓)가 자기 전용 **로컬 메모리 뱅크**를 가지면서도, 인터커넥트를 통해 **다른 소켓의 메모리**에도 접근할 수 있는 다중 소켓 구조입니다.

*Silberschatz, Figure 5.16 — NUMA and CPU scheduling*

- **로컬 메모리 접근**: 빠름(낮은 지연).
- **원격 메모리 접근**: 느림(인터커넥트 경유 필요).

**NUMA-aware 스케줄링**: 각 프로세스를 자신이 할당한 메모리에 **가까운 CPU**에 배치합니다. Linux에서는 CFS가 **스케줄링 도메인 계층**을 통해 NUMA를 고려합니다(6.7절 참조).

> **실무 영향**: 다중 소켓 서버에서 NUMA를 무시하면 메모리 집약적 워크로드에서 **2배의 성능 저하**가 날 수 있습니다. 고성능 DB나 HPC 코드가 `numactl`이나 명시적 NUMA API를 쓰는 이유입니다.

### 4.9 부하 분산

프로세서별 큐를 쓰면 불균형이 생길 수 있습니다 — 한 CPU는 실행 큐가 길고 다른 CPU는 한가한 상황이죠. **부하 분산(Load Balancing)** 이 작업을 재분배합니다.

| 방식 | 설명 |
|----------|-------------|
| **Push migration** | 주기적 태스크가 모든 CPU 부하를 확인, 과부하 CPU에서 한가한 CPU로 태스크를 **밀어내기** |
| **Pull migration** | 한가한 CPU가 바쁜 CPU의 큐에서 태스크를 **끌어당기기** |

대부분의 시스템은 **둘을 함께 사용**합니다(예: Linux CFS).

**부하 분산 vs 프로세서 친화도 — 충돌**:
- **부하 분산**: 부하 균등화를 위해 프로세스를 이동시키려 함.
- **친화도**: 캐시 상태를 보존하려고 같은 CPU에 머무르게 하려 함.

OS는 둘 사이에서 균형을 잡습니다: 불균형이 **충분히 클 때만** 이주시켜, 재분산 이득이 캐시 워밍업 비용을 상회하도록 합니다.

### 4.10 이기종 다중 프로세싱 (HMP)

**동일한 명령어 집합(ISA)** 을 갖지만 **클록 속도와 전력 특성이 다른** 코어들로 구성된 시스템입니다.

**ARM big.LITTLE 아키텍처**(대부분의 현대 스마트폰에서 사용):
- **big 코어**: 고성능, 고전력 — 짧은 인터랙티브 작업에 적합(터치 반응, 앱 시작).
- **LITTLE 코어**: 저성능, 저전력 — 백그라운드 작업에 적합(폴링, 음악 재생).

**스케줄링 전략:**
- 인터랙티브/전경 앱 → **big 코어**.
- 백그라운드 작업 → **LITTLE 코어**.
- 배터리 절약 모드 → big 코어를 완전히 끄고 LITTLE만 사용.

**ASMP와의 핵심 차이**: 비대칭 다중 프로세싱과 달리, HMP에서는 **모든 코어가 사용자 태스크와 시스템 태스크 모두 실행 가능**합니다 — 비대칭성은 역할이 아니라 성능/전력에 있습니다.

## 5. 실시간 CPU 스케줄링

### 5.1 Soft vs Hard 실시간

| 분류 | Soft 실시간 | Hard 실시간 |
|----------|----------------|----------------|
| 마감 | 미달 **허용**(성능이 점진적으로 저하) | 마감 **반드시 준수** — 미달은 시스템 실패 |
| 보장 수준 | 우선순위 기반 우대 | 마감 내 완료 보장 |
| 예 | 영상 재생, VoIP, 온라인 게임 | 항공기 비행 제어, ABS 브레이크, 심박조율기 |

**실시간 스케줄링 요구사항:**
- 스케줄러는 **반드시 선점형**이어야 함(더 높은 우선순위의 실시간 태스크가 어떤 것이든 선점할 수 있어야 함).
- **인터럽트 지연**과 **디스패치 지연**이 **유계(bounded)이고 짧아야** 함(밀리초가 아닌 마이크로초 단위).

### 5.2 이벤트 지연

**이벤트 지연(Event Latency)** = 이벤트 발생부터 서비스 완료까지의 총 시간.

*Silberschatz, Figure 5.17 — Event latency*

실시간 시스템에서 중요한 두 지연 요소:

1. **인터럽트 지연(Interrupt Latency)** — 인터럽트 도착부터 **ISR**(Interrupt Service Routine) 시작까지의 시간.
   - 현재 명령어 완료 → 인터럽트 종류 판별 → 컨텍스트 저장.

2. **디스패치 지연(Dispatch Latency)** — 스케줄러가 현재 실행 중인 프로세스에서 목표 실시간 프로세스로 전환하는 데 걸리는 시간.
   - **충돌 단계(conflict phase)**: 커널 선점 + 선점당한 프로세스가 보유한 자원 해제.
   - **디스패치 단계(dispatch phase)**: 고우선순위 프로세스를 CPU에 적재(컨텍스트 복원, PC로 점프).

Hard 실시간에서는 두 지연 모두 **수 마이크로초 이내로 유계**여야 합니다.

### 5.3 주기적 태스크 모델

실시간 워크로드는 흔히 **주기적 태스크** — 동일 작업이 매 *p* 시간마다 반복 — 로 모델링합니다.

*Silberschatz, Figure 5.20 — Periodic task*

- **t** — 처리 시간(주기당 CPU 버스트).
- **d** — 마감(이 주기의 작업이 완료되어야 하는 시점).
- **p** — 주기(얼마나 자주 반복되는가).
- **관계식**: `0 ≤ t ≤ d ≤ p` — 주기 내 마감 내에 충분한 시간이 필요합니다.
- **태스크 발생률** = 1/p (초당 발생 횟수).
- 태스크의 **CPU 이용률** = t/p (CPU를 요구하는 비율).

> **사고 모형**: 30 FPS 영상 디코더가 매 33 ms마다 한 프레임을 디코딩해야 한다고 합시다. 주기 = 33 ms, 마감 = 33 ms, 버스트 = 디코딩 소요 시간(예: 10 ms). 스케줄러가 한 번이라도 마감을 놓치면 사용자가 프레임 드롭을 보게 됩니다.

### 5.4 Rate-Monotonic 스케줄링

**Rate-Monotonic (RM)** — 고정 우선순위, 선점형 알고리즘:

- **주기가 짧을수록 더 높은 우선순위**. (주기가 빠를수록 더 급합니다.)
- 우선순위 = 1 / 주기.
- 우선순위는 **정적** — 실행 중 변하지 않습니다.
- 주기마다 마감 안에 완료해야 합니다.

**최적성**:
- Rate-Monotonic은 **고정 우선순위 알고리즘 중 최적**입니다.
- 즉, RM으로 스케줄 불가능한 주기적 태스크 집합은 **어떤 고정 우선순위 알고리즘으로도 스케줄 불가능**합니다.

### 5.5 Rate-Monotonic — 성공 예시

- **P1**: 주기 = 50, 버스트 = 20.
- **P2**: 주기 = 100, 버스트 = 35.

**CPU 이용률**: 20/50 + 35/100 = 0.40 + 0.35 = **75%**.

Rate-Monotonic 우선순위: **P1 > P2**(주기가 더 짧음).

한 하이퍼 주기(0~200) 내 마감: P1 @ 50, P2 @ 100, P1 @ 100, P1 @ 150, P2 @ 200.

```text
간트 차트 (주기 0 - 100):

|<-- P1 (20) -->|<------ P2 (30) ------>|<-- P1 (20) -->|<P2(5)>|<---- idle (25) ---->|
0              20                      50              70      75                    100
                                                       ^
                                                   시각 50에 P1의 새 주기 시작
                                                   -> P1이 P2를 선점
```

단계별 해설:
- **0–20**: P1 실행(20 ms 완료). P1의 마감 50은 이미 충족. ✓
- **20–50**: P2가 자신의 35 ms 예산 중 30 ms 실행.
- **50**: P1의 **다음 주기** 시작(새 마감 100). RM에서 P1이 고정적으로 더 높은 우선순위이므로 P1이 P2를 **선점**.
- **50–70**: P1 실행(20 ms 완료). P1의 마감 100 충족. ✓
- **70–75**: P2가 남은 5 ms를 마저 실행. 75에 완료 — 마감 100 훨씬 이전. ✓
- **75–100**: CPU 유휴(시각 100에 P1의 다음 주기가 시작될 때까지 실행 가능한 태스크 없음).

**모든 마감 충족.**

### 5.6 Rate-Monotonic — 실패 예시

- **P1**: 주기 = 50, 버스트 = 25.
- **P2**: 주기 = 80, 버스트 = 35.

**CPU 이용률**: 25/50 + 35/80 = 0.50 + 0.4375 = **93.75%**.

N=2 태스크에 대한 이론적 RM 상한: `N(2^(1/N) - 1) = 2(√2 - 1) ≈ 82.8%`.

**93.75% > 82.8%** 이므로 RM은 실패할 수 있습니다 — 실제로 보면:

```text
 |<-- P1(25ms) -->|<-- P2(25ms) -->| P1선점 |<P2(10ms)>|
 0               25               50       75        85
                                                    ^
                                          P2 마감 = 80
                                          미달!
```

- **P1**: 0–25(완료), 50–75(완료). P1은 마감 충족.
- **P2**: 25–50(25 ms) 실행 후 P1에 선점당함, 75에 재개 후 **85**에 완료.
- **하지만 P2의 마감은 80** → **마감 미달!**

이용률이 RM 상한을 초과 → RM은 스케줄 가능성을 보장하지 못합니다.

### 5.7 Rate-Monotonic CPU 이용률 한계

RM 하에서 N개 태스크에 대한 최악의 CPU 이용률 한계:

> **U = N(2^(1/N) − 1)**

| N (태스크 수) | 이용률 한계 |
|:-------------------:|:-----------------:|
| 1 | 100% |
| 2 | 82.8% |
| 3 | 78.0% |
| 4 | 75.7% |
| 5 | 74.3% |
| N → ∞ | **69.3%** (= ln 2) |

**총 이용률 ≤ 이 한계이면, RM이 모든 마감 충족을 보장합니다.**

한계 초과 시에도 성공할 수 있지만(충분조건이지 필요조건은 아님) 보장되지 않습니다.

> **직관**: 태스크 수가 많아질수록 주기가 다른 태스크들의 선점이 운 나쁘게 겹쳐 한계가 타이트해집니다. 극한에서는 이런 최악의 겹침을 흡수하려면 "CPU의 약 30%를 비워둬야" 합니다.

### 5.8 Earliest-Deadline-First (EDF)

**EDF** — 동적 우선순위, 선점형 알고리즘:

- 매 순간, **마감이 가장 가까운** 태스크를 실행.
- 마감이 변하면(주기적 태스크는 완료할 때마다 새 마감 시작) 우선순위도 **동적으로 변경**.
- **주기적 및 비주기적** 태스크 모두에 작동.
- **이론상 CPU 이용률 100%까지 스케줄 가능** — 엄밀한 의미에서 EDF는 최적.

**실무**: 컨텍스트 스위치 오버헤드와 인터럽트 처리로 실제로는 100%에 도달하지 못합니다. 하지만 EDF는 이용률 면에서 RM을 상당히 능가합니다.

### 5.9 EDF — Rate-Monotonic 실패 사례 해결

RM이 실패했던 동일한 태스크 집합:
- **P1**: 주기 = 50, 버스트 = 25.
- **P2**: 주기 = 80, 버스트 = 35.

EDF 실행 과정:

```text
 시각 0:  P1 마감=50, P2 마감=80  -> P1 먼저(마감 더 가까움)
 시각 25: P1 완료; P2 시작
 시각 50: P1 새 주기, P1 마감=100, P2 마감=80(여전히)
          P2의 마감이 더 가까움 -> P2 계속 실행! (RM과 달리 선점 안 함)
 시각 60: P2 완료(총 35 ms 실행); P1 시작
 시각 85: P1 완료(25 ms 실행)
```

```text
 |<-- P1(25) -->|<----- P2(35) ----->|<-- P1(25) -->|
 0             25                    60             85
```

- **P1**: 마감 50을 시각 25에 충족, 마감 100을 시각 85에 충족. ✓
- **P2**: 마감 80을 시각 60에 충족. ✓

**모든 마감 충족** — P2의 마감이 더 가까울 때 P2가 CPU를 계속 쥐게 함으로써 RM이 처리하지 못한 케이스를 EDF는 해결했습니다.

### 5.10 Rate-Monotonic vs EDF

| 속성 | Rate-Monotonic | EDF |
|----------|---------------|-----|
| 우선순위 | **고정**(주기 기반) | **동적**(마감 기반) |
| CPU 이용률 한계 | 약 69.3%(N → ∞) | **100%**(이론) |
| 구현 복잡도 | **낮음** | 높음 |
| 오버헤드 | **낮음** | 높음 — 마감 이벤트마다 우선순위 재계산 |
| 지원 태스크 종류 | 주기적 태스크만 | 주기적 + 비주기적 |
| 최적성 | **고정 우선순위** 중 최적 | **이론적으로** 최적 전반 |
| 실무 사용 | 항공, 자동차 — **예측 가능성** 중시 | 드묾 — 형식적 분석이 어려움 |

> **안전 필수 시스템에서 RM이 더 선호되는 이유**: RM은 **예측 가능하고 인증이 쉽습니다** — 이용률이 한계 이하임을 손으로 증명할 수 있습니다. EDF의 동적 우선순위는 형식적 추론이 어려워, "실패 = 사람 목숨" 상황에서는 부담이 큽니다.

### 5.11 Proportional Share 스케줄링

총 **T개의 점유 지분(shares)** 을 애플리케이션에 분배하고, 각 애플리케이션은 지분에 비례해 CPU 시간을 받습니다.

**예**: T = 100 지분.
- 프로세스 A: 50 지분 → CPU 시간 **50%**.
- 프로세스 B: 15 지분 → CPU 시간 **15%**.
- 프로세스 C: 20 지분 → CPU 시간 **20%**.
- 남은 15 지분 → 여분.

**승인 제어(Admission Control)**:
- 새 프로세스가 30 지분을 요청.
- 사용 중: 50 + 15 + 20 = 85; 남은 지분: 15.
- 15 < 30 → **거부**.

충분한 지분이 있을 때만 새 작업을 승인 → 승인된 모든 프로세스에 **보장된 비례 몫**의 CPU 시간을 제공합니다. 한 프로세스가 다른 프로세스를 굶길 수 없습니다.

> **현대의 친척들**: Linux cgroups의 CPU quota / CFS 가중치, 컨테이너 CPU 한도(Docker, Kubernetes), 하이퍼바이저 CPU 지분(VMware)이 모두 Proportional Share의 파생입니다. 클라우드 제공자가 "이 vCPU의 X%를 드립니다"라고 보장하는 방식입니다.

## 6. 운영체제 사례

### 6.1 Linux CFS — 개요와 역사

**CFS(Completely Fair Scheduler)** — Linux 커널 **2.6.23**(2007)부터의 기본 스케줄러.

**역사:**
- Linux 2.5 이전: 전통적인 UNIX 스케줄러(단순하지만 SMP 확장성 부족).
- **Linux 2.5**: **O(1) 스케줄러** — 상수 시간 연산, 좋은 SMP 지원.
- **Linux 2.6.23+**: **CFS** — 인터랙티브 반응성 개선, 깔끔한 이론.

**핵심 목표**: 실행 가능한 태스크에 CPU 시간을 우선순위 가중치로 "완전히 공정하게" 분배.

**스케줄링 클래스**(Linux는 실시간과 일반 워크로드를 분리):
- **실시간 클래스**: `SCHED_FIFO`, `SCHED_RR` — 우선순위 0~99(일반 태스크보다 항상 높음).
- **일반 클래스**: **CFS** — 우선순위 100~139, **nice** 값(−20 ~ +19)으로 제어.

### 6.2 Linux 우선순위 범위

*Silberschatz, Figure 5.26 — Scheduling priorities on a Linux system*

- **숫자가 작을수록 우선순위가 높음**(교재 관례를 따릅니다).
- 실시간 태스크(0~99)가 일반 태스크(100~139)를 **항상 선점**합니다.
- 일반 태스크 우선순위 = `100 + (nice + 20) = 120 + nice`.
  - `nice = −20` → 우선순위 100(일반 중 최고).
  - `nice =   0` → 우선순위 120(기본값).
  - `nice = +19` → 우선순위 139(최저).

> **왜 "nice"인가?** 역사적으로 "nice한" 프로세스란 *다른 이에게 양보하는* 프로세스입니다 — 즉 "+nice = 우선순위 낮춤". 셸의 `nice`/`renice` 명령으로 조절할 수 있습니다.

### 6.3 CFS — vruntime 개념

CFS는 **vruntime(가상 실행 시간)** 을 추적합니다 — "공정하게 소모한 CPU 시간"의 추상적 지표입니다.

- CFS는 항상 **vruntime이 가장 작은** 태스크를 실행합니다.
- vruntime은 태스크가 실행되며 증가하지만, **nice 값에 가중**됩니다.

| nice 값 | vruntime 증가율 |
|:----------:|:----------------------:|
| 낮은 nice(−20, 높은 우선순위) | 실제 시간보다 **느리게** 증가 |
| nice = 0 (기본) | 실제 시간과 **같은 속도**로 증가 |
| 높은 nice(+19, 낮은 우선순위) | 실제 시간보다 **빠르게** 증가 |

**예**:
- `nice = 0`인 태스크가 실제 200 ms 실행 → vruntime이 약 200 증가.
- `nice = −5`인 태스크가 실제 200 ms 실행 → vruntime이 200보다 작게 증가.

CFS가 항상 최소 vruntime을 선택하므로 **높은 우선순위 태스크(vruntime 증가 느림)가 실제 CPU 시간을 더 많이 받습니다**.

### 6.4 CFS — vruntime 단계별 예시

세 태스크: A(nice = 0), B(nice = 0), C(nice = 5).
초기 vruntime: A = 0, B = 0, C = 0.

```text
 단계 1: A 실행(10 ms)  -> A.vruntime = 10
         vruntime 상태: A=10, B=0, C=0  -> B가 최소 (C와 동점, B 선택) -> B 선택

 단계 2: B 실행(10 ms)  -> B.vruntime = 10
         vruntime 상태: A=10, B=10, C=0  -> C가 최소 -> C 선택

 단계 3: C 실행(10 ms)  -> C.vruntime = 15 (nice=5로 빠르게 증가)
         vruntime 상태: A=10, B=10, C=15  -> A (B와 동점, A 선택) -> A 선택

 단계 4: A 실행(10 ms)  -> A.vruntime = 20
         vruntime 상태: A=20, B=10, C=15  -> B가 최소 -> B 선택
```

관찰:
- 우선순위가 높은 태스크는 vruntime이 천천히 쌓임 → 더 자주 선택됨.
- **I/O-bound 태스크는 자연스럽게 유리**: 잠깐 실행하고 I/O로 블록됩니다. 블록된 동안 자신의 vruntime은 **증가하지 않는** 반면, 실행 가능한 다른 태스크들의 vruntime은 올라갑니다. 깨어나면 자신의 vruntime이 다른 모두보다 낮아 **즉시 실행**됩니다. 명시적 "I/O 부스트"가 필요 없습니다.

### 6.5 CFS — Target Latency와 타임 슬라이스

CFS는 **고정된 타임 퀀텀을 사용하지 않습니다**. 대신 동적으로 타임 슬라이스를 계산합니다.

**Target Latency** — 모든 실행 가능한 태스크가 **한 번씩** 실행되어야 할 목표 시간 구간.
- 예: target latency = 20 ms, 태스크 4개 → 각각 약 5 ms.

**Minimum Granularity** — 타임 슬라이스가 너무 짧아지지 않도록 하는 하한.
- 많은 태스크가 실행 가능할 때 불합리하게 짧은 슬라이스를 방지.
- 예: 최소 1 ms.

**타임 슬라이스 계산** — target latency를 nice 값에 따라 **가중 비례로 분배**:
- 높은 우선순위(낮은 nice) → **긴** 타임 슬라이스.
- 낮은 우선순위(높은 nice) → **짧은** 타임 슬라이스.

> **왜 고정 퀀텀보다 동적 슬라이스가 나은가**: 고정 퀀텀이면 태스크 100개가 추가되면 시간당 컨텍스트 스위치도 100배 늘어납니다. target latency를 쓰면 슬라이스가 minimum granularity까지 줄어들며, 그 이후에는 컨텍스트 스위치 빈도가 capped됩니다. 이것이 CFS가 태스크 2개에서 2000개까지 매끄럽게 확장되는 이유입니다.

### 6.6 CFS — 레드-블랙 트리

CFS는 실행 가능한 태스크를 **vruntime**을 키로 하는 **레드-블랙 트리**에 저장합니다.

```text
                    T3 (vruntime=25)
                   /                \
           T1 (vruntime=15)    T5 (vruntime=35)
           /        \              /        \
    T0 (vr=10)  T2 (vr=20)  T4 (vr=30)  T6 (vr=40)
     ^
     |
   leftmost 노드
   = 최소 vruntime
   = 다음에 실행될 태스크
```

- **leftmost 노드** = 최소 vruntime = 다음에 실행할 태스크.
- 커널은 이를 `rb_leftmost` 변수에 캐시 → 다음 태스크 선택은 **O(1)**.
- 삽입/삭제: **O(log N)**.
- 태스크가 **블록**되면(I/O 대기 등) 트리에서 **제거**, 다시 실행 가능해지면 **삽입**.

> **왜 힙이 아닌 레드-블랙 트리인가?** 힙도 O(1) 최소값 조회와 O(log N) 삽입이 가능합니다(leftmost 캐시된 RB-트리와 동일). 하지만 CFS는 임의 원소 **제거**(태스크 블록 시)와 **우선순위 갱신**도 필요합니다 — RB-트리가 힙보다 깔끔한 불변식으로 이런 연산을 O(log N)에 지원합니다.

### 6.7 CFS — 부하 분산과 NUMA

CFS 부하 분산은 머신의 메모리 위상에 맞춘 **스케줄링 도메인 계층**을 사용합니다.

*Silberschatz, Figure 5.27 — NUMA-aware load balancing with Linux CFS scheduler*

**전략 — 저비용에서 고비용 마이그레이션 순:**
1. **같은 도메인 내**(L2 캐시를 공유하는 코어) — 저렴, 자유롭게 이주.
2. **도메인 간**(L3 캐시를 공유) — 중간 비용.
3. **NUMA 노드 간**(메모리 뱅크가 다름) — 고비용, **극단적 불균형일 때만**.

CFS에서 **부하 = 우선순위 × CPU 이용률 비율**로 정의되므로, 같은 CPU에서도 고우선순위 태스크가 저우선순위 태스크보다 "부하"에 더 크게 기여합니다.

> **요약**: CFS는 단순히 "최소 vruntime 선택"이 아닙니다 — 실제 다중 소켓 서버에서는 단일 CPU 알고리즘(vruntime + RB-트리)을 감싸는 다중 CPU 균형자가 캐시와 메모리 지역성을 존중하는 계층 시스템입니다.

### 6.8 Windows 스케줄링

**우선순위 기반 선점형 스케줄링**.

우선순위 구조: **32 단계**
- **0** — 유휴 스레드(메모리 관리 용도).
- **1–15** — 가변(Variable) 클래스(일반 사용자/시스템 작업).
- **16–31** — 실시간(Real-time) 클래스.

**디스패처**: 항상 최고 우선순위 준비 스레드를 실행.

세부 동작:
- **같은 우선순위** → 그들 사이 **라운드 로빈**.
- **가변 클래스**: 타임 퀀텀 소진 시 우선순위 **감소**(단, 기본 우선순위 이하로는 내려가지 않음).
- **우선순위 부스트**: I/O 완료, 키보드 입력 등 이벤트 발생 → 인터랙티브 스레드가 빠르게 반응하도록 우선순위를 높입니다.

### 6.9 Windows — 우선순위 클래스

Windows API는 우선순위를 **6개의 우선순위 클래스**로 묶고, 각 클래스 안에 **7개의 상대 우선순위 레벨**을 둡니다.

| 우선순위 클래스 | 기본 우선순위 |
|----------------|:-------------:|
| REALTIME_PRIORITY_CLASS | 24 |
| HIGH_PRIORITY_CLASS | 13 |
| ABOVE_NORMAL_PRIORITY_CLASS | 10 |
| NORMAL_PRIORITY_CLASS | 8 |
| BELOW_NORMAL_PRIORITY_CLASS | 6 |
| IDLE_PRIORITY_CLASS | 4 |

각 클래스 내: IDLE, LOWEST, BELOW_NORMAL, **NORMAL**, ABOVE_NORMAL, HIGHEST, TIME_CRITICAL.

**예**: `ABOVE_NORMAL_PRIORITY_CLASS`(기본 10) + NORMAL 상대 = **우선순위 10**.

API:
- `SetPriorityClass()` — 프로세스의 우선순위 클래스 변경.
- `SetThreadPriority()` — 클래스 내에서 스레드의 상대 우선순위 변경.

### 6.10 Windows — 가변 우선순위 동작

Windows는 동작에 따라 스레드 우선순위를 동적으로 조정합니다:

```text
 기본 우선순위 = 8 (NORMAL 클래스, NORMAL 상대)

 타임 퀀텀 소진
 --> 우선순위 감소(예: 8 -> 7)
 --> 기본 우선순위 아래로는 내려가지 않음

 I/O 완료(키보드 입력)
 --> 큰 부스트(예: 8 -> 14)

 I/O 완료(디스크)
 --> 중간 부스트(예: 8 -> 10)

 전경 윈도우
 --> 타임 퀀텀 3배 증가
```

**목표**: 인터랙티브 스레드에 **좋은 반응 시간**을 주고(사용자 이벤트가 도착할 때 큰 부스트) **I/O 장치를 바쁘게**(I/O 완료 스레드가 빠르게 스케줄) 유지. 반면 퀀텀을 소진한 CPU-bound 스레드는 기본 우선순위로 수렴합니다 — 여전히 실행되지만 인터랙티브를 방해하지 않는 선에서입니다.

### 6.11 Windows — 다중 프로세서 지원

**SMT 집합(SMT Set)** — **같은 코어**에 있는 하드웨어 스레드들의 그룹.
- 예: 코어 4개 × 스레드 2개/코어 → SMT 집합 `{0,1}`, `{2,3}`, `{4,5}`, `{6,7}`.
- 스레드를 같은 SMT 집합 안에 두면 → **캐시 효율**(공유 L1/L2)이 좋습니다.

**Ideal Processor** — 각 스레드에 **선호 CPU 번호**를 할당:
- 프로세스별 **seed 값**에서 시작하여 스레드 생성마다 증가.
- **서로 다른 SMT 집합**에 분산: 0, 2, 4, 6, 0, 2, ...
- 프로세스마다 seed가 달라 → 시스템 전체적으로 스레드가 코어에 고르게 분포.

> **왜 그냥 "아무 유휴 CPU"가 아닌가**: SMT가 있는 멀티코어에서 *어느* 유휴 CPU인지가 중요합니다 — 새 스레드를 빈 SMT 형제에 두면 L1 캐시가 공유되고, 다른 소켓에 두면 원격 메모리 비용이 듭니다. Ideal Processor는 "이 CPU 우선, 그다음 SMT 형제, 그다음 근처, 마지막으로 먼 곳" 정책을 체계화합니다.

## 7. 알고리즘 평가

### 7.1 네 가지 평가 방법 — 개요

| 방법 | 설명 | 정확도 |
|--------|-------------|----------|
| **Deterministic Modeling** | 특정 워크로드에 대한 해석적 계산 | 낮음(특정 케이스만) |
| **Queueing Models** | 수학적 모델(Little's formula 등) | 중간(근사) |
| **Simulation** | 스케줄러 컴퓨터 시뮬레이션 | 높음 |
| **Implementation** | 실제 OS에 구현하여 측정 | 최고(실환경) |

**트레이드오프**: 정확도가 높을수록 비용과 복잡도도 증가합니다.

### 7.2 Deterministic Modeling

**특정 워크로드**를 가정하고 각 알고리즘의 정확한 성능을 계산합니다.

| 프로세스 | 버스트 시간 |
|---------|-----------|
| P1 | 10 |
| P2 | 29 |
| P3 | 3  |
| P4 | 7  |
| P5 | 12 |

모든 프로세스가 시각 0에 도착.

| 알고리즘 | 평균 대기 시간 |
|-----------|---------------------|
| FCFS | (0 + 10 + 39 + 42 + 49) / 5 = **28 ms** |
| SJF | (10 + 32 + 0 + 3 + 20) / 5 = **13 ms** |
| RR (q = 10) | (0 + 32 + 20 + 23 + 40) / 5 = **23 ms** |

**장점**: 단순, 빠름, 주어진 워크로드에 대해 정밀.
**단점**: 결론이 **해당 워크로드에만** 적용 — 실제 워크로드는 다양합니다.

### 7.3 Queueing Models — Little's Formula

시스템을 **서버 + 큐의 네트워크**로 모델링합니다.

**Little's Formula:**

> **n = λ × W**

- **n** — 큐 내 평균 프로세스 수(큐 길이).
- **λ** — 평균 도착률(초당 프로세스).
- **W** — 평균 대기 시간(프로세스당 초).

**예**: λ = 초당 7 프로세스, n = 14 프로세스.
- W = n / λ = 14 / 7 = **2초**.

**한계**: Little's formula는 일반적 항등식이지만, 현실적인 스케줄링 정책에서 `n`과 `λ`를 계산하려면 **수학적으로 다루기 어려운** 분포가 필요합니다. 독립성 가정(도착과 서비스가 무상관)도 항상 성립하지 않습니다.

> **Little's formula가 아름다운 이유**: 도착/서비스 시간의 분포에 대한 어떤 가정도 필요로 하지 않습니다 — 단지 흐름 보존(conservation of flow)일 뿐입니다. 세 값 중 두 개를 알면 나머지 하나를 유도할 수 있습니다.

### 7.4 Simulation과 Implementation

**Simulation(시뮬레이션)**
- 컴퓨터 시스템의 소프트웨어 모델을 만들어 합성 워크로드로 스케줄러를 돌립니다.
- 워크로드 원천:
  - **난수 생성** — 편리하지만 실제 사용과 일치하지 않을 수 있음.
  - **트레이스 파일** — 실제 시스템의 이벤트 시퀀스를 기록 → 실제 행동을 정확히 재현, 훨씬 높은 정확도.
- **단점**: 개발 비용, 긴 실행 시간, 트레이스 저장 공간.

**Implementation(구현)**
- 알고리즘을 실제 OS에 구현해 실제 하드웨어, 실제 사용자에게 배포.
- **유일하게 완전히 정확한** 평가 방법.
- **단점**:
  - 구현/테스트 비용이 큼.
  - **환경이 바뀌면 사용자 행동도 바뀜** — 사용자가 스케줄러가 특정 패턴을 선호한다고 알게 되면 이를 악용(예: 짧은 작업 할인을 받으려 작업을 쪼개기).

> **유명한 경고성 일화**: 초기 멀티 유저 OS에서 스케줄러가 I/O를 많이 하는 작업을 부스트했는데, 사용자들이 이를 알아채고 CPU-bound 작업에 쓸데없는 I/O 호출을 추가해 스케줄러를 속였습니다. 시뮬레이션에서 훌륭했던 평가가 배포 후 무너졌습니다. 실제 환경 평가가 대체 불가능한 이유입니다.

## 8. 실습 — MLFQ 시뮬레이터

### 8.1 실습: 큐 구성과 규칙

**3단계 Multilevel Feedback Queue** 시뮬레이터를 구현합니다.

**큐 구성**:

| 큐 | 알고리즘 | 타임 퀀텀 |
|:-----:|-----------|:------------:|
| Q0 | 라운드 로빈 | 8 |
| Q1 | 라운드 로빈 | 16 |
| Q2 | FCFS | 없음(완료까지 실행) |

**규칙**:
- 새 프로세스는 **Q0**에 진입.
- Q0 퀀텀 내에 완료하지 못하면 → **Q1**로 강등.
- Q1 퀀텀 내에 완료하지 못하면 → **Q2**로 강등.
- **Aging**: Q2에서 임계값(예: 30 시간 단위) 이상 대기하면 → **Q0**로 승격.

### 8.3 실습: 핵심 실행 로직

```python
def run_process(proc, queue_level, quanta, time):
    """선택된 프로세스를 해당 큐의 퀀텀만큼 실행."""
    if queue_level < 2:            # Q0 또는 Q1 -> RR
        quantum = quanta[queue_level]
        run_time = min(proc.remaining, quantum)
    else:                          # Q2 -> FCFS, 퀀텀 없음
        run_time = proc.remaining

    proc.remaining -= run_time
    new_time = time + run_time

    if proc.remaining == 0:
        # 프로세스 완료
        proc.finish_time = new_time
        return new_time, 'completed'
    else:
        # 퀀텀 소진 -> 강등
        proc.queue_level = min(queue_level + 1, 2)
        return new_time, 'demoted'
```

테스트 데이터: `P1(도착=0, 버스트=30), P2(2, 12), P3(4, 6), P4(6, 20), P5(8, 4)`.

### 8.4 실습: 샘플 출력

간트 차트:

```text
| P1(Q0) | P2(Q0) | P3(Q0) | P4(Q0) | P5(Q0) | P2(Q1) | P4(Q1) | P1(Q1) | P1(Q2) |
0        8       16       22       30       34       38       50       66       72
```

통계:

| 프로세스 | 도착 | 버스트 | 완료 | 반환 시간 | 대기 시간 |
|---------|:-------:|:-----:|:------:|:----------:|:-------:|
| P1 | 0 | 30 | 72 | 72 | 42 |
| P2 | 2 | 12 | 38 | 36 | 24 |
| P3 | 4 | 6  | 22 | 18 | 12 |
| P4 | 6 | 20 | 50 | 44 | 24 |
| P5 | 8 | 4  | 34 | 26 | 22 |

- **평균 반환 시간** = (72 + 36 + 18 + 44 + 26) / 5 = **39.2 ms**.
- **평균 대기 시간** = (42 + 24 + 12 + 24 + 22) / 5 = **24.8 ms**.

짧은 작업(P3, P5)은 Q0 안에서 모두 끝내 최고 우선순위로 빠르게 반응하는 반면, 가장 긴 작업 P1은 Q2까지 내려가 가장 늦게 끝나는 것을 보세요 — 설계대로 MLFQ가 짧은 프로세스를 우대하는 모습입니다.

| 우선순위 스케줄링 | 우선순위 기반(작을수록 높음); 기아는 **Aging**으로 해결 |

| Rate-Monotonic | **고정 우선순위** = 1/주기; 한계 약 69.3%(N→∞); 고정 우선순위 중 최적 |
| EDF | **동적 우선순위** = 가장 가까운 마감; 이론상 100% 스케줄 가능; 비주기 태스크도 처리 |
| Proportional Share | **T 지분** 분배; 지분 부족 시 승인 제어가 거부 |
| Linux CFS | **vruntime** + **레드-블랙 트리**; 고정 퀀텀 대신 target latency; NUMA-aware |
| Windows | 우선순위 32단계; 가변(1~15) + 실시간(16~31); I/O 시 부스트; Ideal Processor + SMT 집합 |
