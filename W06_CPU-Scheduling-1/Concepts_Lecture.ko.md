## 1. 기본 개념

### 1.1 다중 프로그래밍과 CPU 스케줄링

**핵심 아이디어**: CPU를 항상 바쁘게 유지하여 **이용률을 최대화** 한다

- 단일 CPU 코어 시스템에서는 한 번에 하나의 프로세스만 실행할 수 있다
- 프로세스가 I/O를 기다리는 동안 CPU는 **유휴 상태** — 낭비
- **다중 프로그래밍(Multiprogramming)**: 여러 프로세스를 메모리에 유지; 하나가 대기하면 다른 프로세스에 CPU를 할당
- CPU 스케줄링은 운영체제의 **가장 근본적인 기능** 중 하나이다

> CPU 스케줄링을 화구(CPU)가 하나인 레스토랑 주방으로 생각해보자. 셰프가 요리를 오븐에 넣고(I/O 버스트) 그냥 서서 기다리면 시간 낭비이다. 좋은 주방 매니저(스케줄러)는 첫 번째 요리가 구워지는 동안 셰프에게 다른 요리를 준비하도록 배정한다(다른 프로세스의 CPU 버스트). 다중 프로그래밍은 화구가 항상 사용되도록 보장한다.

### 1.2 CPU-I/O 버스트 사이클

프로세스 실행은 **CPU 버스트** 와 **I/O 버스트** 의 반복 사이클로 구성된다:

- **CPU 버스트**: CPU가 명령어를 실행하는 구간
- **I/O 버스트**: 프로세스가 I/O 완료를 기다리는 구간
- 마지막 CPU 버스트에서 프로세스는 **시스템 호출을 통해 종료를 요청** 한다

```c
// 프로세스 실행 패턴 (의사코드)
while (process_not_terminated) {
    // === CPU 버스트 ===
    execute_instructions();   // load, store, add, ...

    // === I/O 버스트 ===
    request_io();             // 파일 읽기, 파일 쓰기, ...
    wait_for_io_completion(); // 블로킹
}
terminate();                  // 마지막 CPU 버스트 후 종료
```

- 모든 프로세스가 이 패턴을 따르지만 **버스트 길이와 빈도** 가 다르다
- 이 차이가 스케줄링 알고리즘을 선택하는 핵심 근거이다

> CPU-I/O 버스트 사이클은 프로세스 실행의 근본적인 리듬이다. 모든 프로세스는 "생각하기"(CPU 버스트)와 "대기하기"(I/O 버스트)를 번갈아 한다. 스케줄링 알고리즘의 역할은 빈틈을 효율적으로 채우는 것이다 — 한 프로세스가 I/O 버스트에 진입하면 다른 프로세스가 CPU 버스트를 할 준비가 되어 있어야 한다.

### 1.3 CPU 버스트 분포

CPU 버스트 길이의 분포는 **지수적 또는 초지수적(hyperexponential)** 감소 패턴을 따른다:

- **짧은 CPU 버스트** 가 매우 빈번하다 — 대부분의 버스트는 짧다
- **긴 CPU 버스트** 는 드물게 발생한다

> 이 분포가 중요한 이유는 SJF와 같은 알고리즘의 설계를 정당화하기 때문이다: 대부분의 버스트가 짧으므로 짧은 버스트를 우선하면 대부분의 프로세스가 빠르게 서비스되고, 드문 긴 버스트 프로세스만 더 오래 기다린다.

### 1.4 I/O 바운드 vs CPU 바운드 프로세스

CPU 버스트 분포에 기반한 프로세스 분류:

| 범주 | I/O 바운드 프로세스 | CPU 바운드 프로세스 |
|------|---------------------|---------------------|
| CPU 버스트 | 많은 **짧은** 버스트 | 적은 **긴** 버스트 |
| I/O 버스트 | 빈번한 I/O 요청 | 드문 I/O 요청 |
| 예시 | 웹 서버, 데이터베이스 | 과학 계산, 비디오 인코딩 |
| 스케줄링 요구 | 빠른 응답 시간 | 긴 CPU 할당 시간 |

- **스케줄링 알고리즘** 은 두 유형을 모두 효율적으로 처리해야 한다
- 이 분포 특성은 SJF와 RR 같은 알고리즘의 이론적 근거가 된다

> **시험 팁:** 프로세스를 I/O 바운드 또는 CPU 바운드로 분류할 수 있는 것은 서로 다른 스케줄링 알고리즘이 서로 다른 워크로드를 선호하는 이유를 이해하는 데 필수적이다. FCFS는 CPU 바운드 프로세스에 유리하고(긴 무중단 실행을 얻음), RR은 I/O 바운드 프로세스에 유리하다(빠르고 빈번한 CPU 접근을 얻음).

### 1.5 CPU 스케줄러 (단기 스케줄러)

CPU가 유휴 상태가 되면 **준비 큐(ready queue)** 에서 다음에 실행할 프로세스를 선택한다:

```text
                  +------------+
  새 프로세스 --->   | 준비 큐     |  ---> CPU 스케줄러 ---> CPU
                  +------------+       선택 알고리즘
                         ^
                         |
                     I/O 완료 등
```

- **CPU 스케줄러** = 단기 스케줄러(Short-term Scheduler)
- 준비 큐는 반드시 FIFO일 필요가 없다
  - FIFO 큐, **우선순위 큐**, 트리, 연결 리스트 등으로 구현 가능
- 큐의 레코드는 일반적으로 **PCB (프로세스 제어 블록)**
- 큐의 내부 순서는 스케줄링 알고리즘에 의해 결정된다

> **핵심:** "준비 큐"는 반드시 자료구조 의미의 큐가 아니다. 실행 준비가 된 프로세스의 모음이다. 스케줄링 알고리즘이 그 모음의 정렬 방식을 결정한다 — FCFS는 FIFO 큐를 사용하고, SJF는 버스트 길이로 정렬된 우선순위 큐를 사용한다. "준비 큐"라는 이름은 역사적 관례이다.

### 1.6 스케줄링이 발생하는 4가지 상황

CPU 스케줄링 결정이 필요한 **4가지 상황**:

| 번호 | 상태 전이 | 예시 | 선택 |
|------|----------|------|------|
| **1** | 실행 -> 대기 | I/O 요청, wait() 호출 | 선택 없음 |
| **2** | 실행 -> 준비 | 인터럽트 발생 (타이머 등) | **선택 있음** |
| **3** | 대기 -> 준비 | I/O 완료 | **선택 있음** |
| **4** | 종료 | 프로세스 종료 | 선택 없음 |

- 상황 **1, 4**: 새 프로세스를 선택해야 한다 (선택 없음)
- 상황 **2, 3**: 스케줄링 방식에 따라 다르다 (선택 있음)

> 이 네 가지 상황을 이해하는 것은 선점형과 비선점형 스케줄링의 차이를 이해하는 데 근본적이다. 상황 1과 4는 필수 스케줄링 지점이다 — 실행 중인 프로세스가 더 이상 CPU를 사용할 수 없다. 상황 2와 3은 선택적이다 — 선점형 시스템에서만 스케줄링 결정을 발생시킨다.

### 1.7 비선점형 vs 선점형 스케줄링

| 범주 | 비선점형 | 선점형 |
|------|---------|--------|
| 스케줄링 시점 | **1, 4** 에서만 | **1, 2, 3, 4** (전부) |
| CPU 반환 | 프로세스가 **자발적** 으로 반환 | OS가 **강제로** 회수 가능 |
| 다른 이름 | 협력적(Cooperative) 스케줄링 | - |
| 사용 | 초기 OS | **대부분의 현대 OS** |

- **비선점형**: 프로세스가 자발적으로 CPU를 반환한다 (종료 또는 I/O 대기)
- **선점형**: OS가 타이머 인터럽트 등을 통해 강제로 CPU를 회수할 수 있다
- Windows, macOS, Linux, UNIX — 모두 **선점형** 스케줄링을 사용한다

> **시험 팁:** "비선점형 = 상황 1, 4만" 그리고 "선점형 = 네 가지 상황 모두"는 자주 출제되는 구분이다. 비선점형이 "협력적"이라고도 불리는 이유는 프로세스가 자발적으로 CPU를 양보하며 협력하기 때문이다.

### 1.8 선점형 스케줄링의 문제점

선점형 스케줄링은 강력하지만 **주의사항** 이 있다:

**1. 경쟁 조건(Race Condition)**
- 공유 데이터 수정 중 선점 -> 다른 프로세스가 일관성 없는 데이터를 읽음
- -> **동기화 메커니즘** (뮤텍스, 세마포어 등)이 필요 (6장에서 상세 다룸)

**2. 커널 모드에서의 선점**
- 시스템 호출 내에서 커널 자료 구조 수정 중 선점 -> 커널 불일치
- **비선점형 커널**: 시스템 호출 완료까지 컨텍스트 스위치 연기 (단순하지만 실시간에 부적합)
- **선점형 커널**: 뮤텍스 락 등으로 커널 데이터 보호 (대부분의 현대 OS)

**3. 인터럽트 처리**
- 인터럽트는 언제든 발생할 수 있음 -> 임계 영역에서 인터럽트 비활성화

> 이러한 문제들은 9주차의 동기화 주제를 예고한다. 핵심 통찰은 선점이 동시성 문제를 도입한다는 것이다: OS가 어느 시점에서든 프로세스를 중단할 수 있다면, 다른 프로세스가 읽을 때 공유 데이터가 일관성 없는 상태일 수 있다. 이것이 동기화 메커니즘이 존재하는 근본적인 이유이다.

### 1.9 디스패처

**디스패처(Dispatcher)**: 스케줄러가 선택한 프로세스에 실제로 CPU 제어권을 넘겨주는 모듈

디스패처의 세 가지 역할:
1. **컨텍스트 스위치** 수행 (현재 프로세스 상태 저장 -> 새 프로세스 상태 복원)
2. **사용자 모드** 로 전환
3. 새 프로세스의 적절한 위치(**PC**)로 점프

*Silberschatz, Figure 5.3 — 디스패처의 역할*

> **스케줄러 vs 디스패처**: 스케줄러는 다음에 어떤 프로세스를 실행할지 **결정** 하고, 디스패처는 그 결정을 **실행** 한다. 스케줄러를 다음에 어떤 직원이 일할지 고르는 관리자로, 디스패처를 실제 서류 작업과 사무실 재배정을 처리하는 인사 담당자로 생각하면 된다. 서로 다른 책임을 가진 별개의 구성 요소이므로 이 구분이 중요하다.

### 1.10 디스패치 지연

**디스패치 지연(Dispatch Latency)** = 디스패처가 한 프로세스를 중단하고 다른 프로세스를 시작하는 데 걸리는 시간

- 모든 컨텍스트 스위치마다 발생하므로 **가능한 한 짧아야** 한다

Linux에서 컨텍스트 스위치 횟수 확인:

```bash
# 시스템 전체 컨텍스트 스위치 횟수 (1초 간격, 3회)
vmstat 1 3

# 특정 프로세스의 컨텍스트 스위치 횟수
cat /proc/<PID>/status | grep ctxt
# voluntary_ctxt_switches:    150   (자발적: I/O 대기 등)
# nonvoluntary_ctxt_switches: 8     (비자발적: 타임 슬라이스 만료 등)
```

- **자발적 컨텍스트 스위치**: 자원 부족으로 프로세스가 자발적으로 CPU를 반환
- **비자발적 컨텍스트 스위치**: 타임 슬라이스 만료 등으로 OS가 강제 전환

> **[시스템 프로그래밍]** 현대 시스템에서 컨텍스트 스위치는 보통 1-10 마이크로초가 걸린다. 이는 미미해 보이지만 누적된다: 초당 수천 번의 컨텍스트 스위치를 수행하는 바쁜 서버는 디스패치 지연으로 상당한 CPU 시간을 잃을 수 있다. 이것이 라운드 로빈 스케줄링에서 타임 퀀텀을 신중하게 선택해야 하는 이유이다 — 퀀텀이 너무 작으면 컨텍스트 스위치가 너무 많아진다.

## 2. 스케줄링 기준

### 2.1 스케줄링 기준 — 개요

스케줄링 알고리즘을 **비교** 하고 **평가** 하기 위한 다섯 가지 기준:

| 기준 | 설명 | 최적화 방향 |
|------|------|------------|
| **CPU 이용률(CPU Utilization)** | CPU 사용률 (%) | **최대화** |
| **처리량(Throughput)** | 단위 시간당 완료된 프로세스 수 | **최대화** |
| **반환 시간(Turnaround Time)** | 제출에서 완료까지의 총 시간 | **최소화** |
| **대기 시간(Waiting Time)** | 준비 큐에서 대기한 총 시간 | **최소화** |
| **응답 시간(Response Time)** | 제출에서 첫 응답까지의 시간 | **최소화** |

### 2.2 CPU 이용률과 처리량

**CPU 이용률**
- CPU가 유용한 작업을 수행하는 시간의 비율
- 이론적 범위: 0%에서 100%
- 실제 시스템: 가벼운 부하에서 약 **40%**, 높은 부하에서 **90%**
- Linux에서 `top` 명령으로 확인 가능

**처리량**
- 단위 시간당 완료된 프로세스 수
- 긴 프로세스: 몇 초에 하나
- 짧은 트랜잭션: 초당 수십 개

- 목표는 두 기준 모두 **최대화** 하는 것이다
- 서로 트레이드오프 관계일 수 있다 (컨텍스트 스위치 오버헤드)

### 2.3 반환 시간과 대기 시간

**반환 시간(Turnaround Time)**
- 프로세스 **제출** 에서 **완료** 까지의 총 시간
- = 준비 큐 대기 + CPU 실행 + I/O 대기 시간의 합

```text
반환 시간 = 완료 시각 - 도착 시각
```

**대기 시간(Waiting Time)**
- 준비 큐에서 **대기** 한 총 시간
- 스케줄링 알고리즘이 **직접 영향을 미치는** 유일한 시간
- CPU 실행 시간과 I/O 시간은 알고리즘과 무관하다

```text
대기 시간 = 반환 시간 - 버스트 시간
         (단일 버스트, I/O 없는 경우)
```

> **시험 팁:** 대기 시간은 교재 문제에서 스케줄링 알고리즘을 비교할 때 가장 많이 사용되는 메트릭이다. 스케줄링 알고리즘의 효과를 분리하기 때문이다. 반환 시간은 어떤 알고리즘을 사용하든 동일한 CPU 실행 시간과 I/O 시간을 포함한다. 대기 시간은 스케줄링 결정에 의해 도입된 지연 *만* 을 포착한다.

### 2.4 응답 시간

**응답 시간(Response Time)**
- 프로세스 **제출** 에서 **첫 응답이 생성되는** 순간까지의 시간
- 대화형 시스템에서 가장 중요한 기준

```text
응답 시간 = 첫 실행 시각 - 도착 시각
```

**반환 시간 vs 응답 시간**:

| 비교 | 반환 시간 | 응답 시간 |
|------|----------|----------|
| 측정 구간 | 제출에서 **완료** 까지 | 제출에서 **첫 응답** 까지 |
| 중요한 시스템 | 배치 시스템 | **대화형** 시스템 |
| 예시 | 작업 완료까지 30초 | 첫 출력까지 0.5초 |

- 대화형 시스템에서는 응답 시간의 **분산을 최소화** 하는 것이 중요하다
- **예측 가능한** 응답 시간이 평균보다 사용자 만족도에 더 유리하다

### 2.5 스케줄링 기준 — 요약

```text
+----------------------------------+
|               최대화               |
|  +------------+  +------------+  |
|  |  CPU 이용률  |  |   처리량    |   |
|  +------------+  +------------+  |
+----------------------------------+
|               최소화               |
|  +------------+  +------------+  |
|  |  반환 시간   |  |   대기 시간   |  |
|  +------------+  +------------+  |
|  +------------+                  |
|  |  응답 시간   |                  |
|  +------------+                  |
+----------------------------------+
```

- 대부분의 경우 **평균** 을 최적화한다
- 일부 경우 **최솟값** 또는 **최댓값** 을 최적화한다 (예: 최대 응답 시간 최소화)

## 3. FCFS 스케줄링

### 3.1 FCFS (선입선처리)

**가장 간단한** CPU 스케줄링 알고리즘: **도착 순서** 대로 실행

- **FIFO 큐** 로 구현
- 프로세스가 준비 큐에 도착하면 **꼬리(tail)** 에 추가
- CPU가 비면 **머리(head)** 의 프로세스에 할당
- **비선점형** 스케줄링

```text
준비 큐 (FIFO):

  head ---> [ P1 ] ---> [ P2 ] ---> [ P3 ] <--- tail
              |
              v
           CPU 할당
```

### 3.2 FCFS — 예제 1 (도착 순서: P1, P2, P3)

프로세스 도착 순서: P1 -> P2 -> P3 (모두 시각 0에 도착)

| 프로세스 | 버스트 시간 |
|---------|-----------|
| P1 | 24 ms |
| P2 | 3 ms |
| P3 | 3 ms |

```text
간트 차트
+-------------------------+----+----+
|            P1           | P2 | P3 |
+-------------------------+----+----+
0                         24   27   30
```

| 프로세스 | 대기 시간 | 반환 시간 |
|---------|----------|----------|
| P1 | 0 | 24 |
| P2 | 24 | 27 |
| P3 | 27 | 30 |
| **평균** | **(0+24+27)/3 = 17** | **(24+27+30)/3 = 27** |

### 3.3 FCFS — 예제 2 (도착 순서: P2, P3, P1)

같은 프로세스지만 도착 순서가 다른 경우: P2 -> P3 -> P1

```text
간트 차트:
+----+----+-------------------------+
| P2 | P3 |            P1           |
+----+----+-------------------------+
0    3    6                         30
```

| 프로세스 | 대기 시간 | 반환 시간 |
|---------|----------|----------|
| P1 | 6 | 30 |
| P2 | 0 | 3 |
| P3 | 3 | 6 |
| **평균** | **(6+0+3)/3 = 3** | **(30+3+6)/3 = 13** |

- 평균 대기 시간: **17 -> 3** (순서만 바꿨는데 극적으로 감소!)
- FCFS 성능은 도착 순서에 따라 **크게 변한다**

### 3.4 FCFS — 호위 효과

**호위 효과(Convoy Effect)**: 긴 CPU 버스트를 가진 단일 프로세스 뒤에 짧은 프로세스들이 줄을 서는 현상

```text
시나리오: 1개의 CPU 바운드 프로세스 + 여러 I/O 바운드 프로세스

시간 ->  ==========================================================>

CPU:  [ CPU 바운드 (긴 버스트) ............][ I/O-1 ][ I/O-2 ][ I/O-3 ]
I/O:  [    유휴 (I/O 장치가 놀고 있음)      ][  사용  ][  사용  ][  사용  ]
                                       ^
                                       |
                               I/O 바운드 프로세스들이
                               CPU 바운드 뒤에서 대기
```

- CPU 바운드 프로세스가 CPU를 독점 -> I/O 바운드 프로세스들이 준비 큐에서 대기
- 한편 **I/O 장치는 유휴 상태** -> CPU와 I/O 모두 **이용률이 떨어진다**
- **대화형 시스템에 부적합**: 단일 프로세스가 장시간 CPU를 독점할 수 있다

> 호위 효과는 단일 차선 도로의 느린 트럭에 비유된다: 모든 빠른 자동차(짧은 프로세스)가 그 뒤에서 기다려야 한다. 스케줄링 용어로는 CPU와 I/O 장치 모두 과소 활용된다 — I/O 바운드 프로세스가 CPU를 기다리는 동안 I/O 장치는 놀고 있고, 나중에 모든 I/O 바운드 프로세스가 동시에 I/O를 할 때 CPU가 놀게 된다.

### 3.5 FCFS — 특성 요약

| 특성 | 설명 |
|------|------|
| 유형 | **비선점형** |
| 구현 | FIFO 큐 — 매우 간단 |
| 평균 대기 시간 | 일반적으로 **길다** (최적이 아님) |
| 호위 효과 | **발생** (CPU 바운드가 I/O 바운드를 차단) |
| 대화형 적합성 | **부적합** (일정 간격으로 CPU를 분배할 수 없음) |
| 장점 | 구현이 간단하고 이해하기 쉬움 |
| 단점 | 성능이 도착 순서에 크게 의존 |

## 4. SJF 스케줄링

### 4.1 SJF (최단 작업 우선)

**다음 CPU 버스트가 가장 짧은** 프로세스를 먼저 선택:

- 더 정확한 이름: **최단 다음 CPU 버스트(Shortest-Next-CPU-Burst)** 알고리즘
- 버스트 길이가 같으면 -> **FCFS** 로 순서 결정
- 비선점형과 선점형 버전이 모두 존재

**핵심 속성**: 주어진 프로세스 집합에 대해 **최소 평균 대기 시간** 을 보장 — **최적(Optimal)**

**근본적 문제**: 다음 CPU 버스트 길이를 **정확히 알 수 없다**

### 4.2 SJF — 최적성 증명

**왜 SJF가 최적인가?**

```text
경우 1: 긴 것 먼저 (버스트: 10, 3)
+-----------+----+
|   P1(10)  | P2 |   P1 대기=0, P2 대기=10  -> 평균 = 5
+-----------+----+
0           10   13

경우 2: 짧은 것 먼저 (버스트: 3, 10)  <- SJF
+----+-----------+
| P2 |   P1(10)  |   P2 대기=0, P1 대기=3   -> 평균 = 1.5
+----+-----------+
0    3           13
```

- **짧은 프로세스** 를 앞으로 이동: 그것의 대기 시간이 **크게 감소**
- **긴 프로세스의** 대기 시간은 **약간 증가**
- 결과: 전체 평균 대기 시간이 **감소**
- 이를 일반화하면 SJF가 **증명 가능하게 최적** 임을 보인다

> **[알고리즘]** 증명은 교환 논법(exchange argument)으로 일반화된다: SJF 순서가 아닌 어떤 스케줄에서도 긴 작업이 짧은 작업 앞에 오는 인접 쌍을 찾을 수 있다. 그것들을 교환하면 다른 작업에 영향을 주지 않으면서 총 대기 시간이 줄어든다. 이를 반복하면 SJF 순서가 된다 — 버스트 시간으로 정렬하면 총 가중 완료 시간이 최소화된다는 정렬 논법과 동등하다.

### 4.3 SJF — 예제 (비선점형)

모든 프로세스가 시각 0에 도착 (비선점형 SJF):

| 프로세스 | 버스트 시간 |
|---------|-----------|
| P1 | 6 ms |
| P2 | 8 ms |
| P3 | 7 ms |
| P4 | 3 ms |

```text
간트 차트 (SJF 순서: P4 -> P1 -> P3 -> P2):
+----+--------+----------+------------+
| P4 |   P1   |    P3    |     P2     |
+----+--------+----------+------------+
0    3        9         16            24
```

| 프로세스 | 대기 시간 |
|---------|----------|
| P1 | 3 |
| P2 | 16 |
| P3 | 9 |
| P4 | 0 |
| **평균** | **(3+16+9+0)/4 = 7** |

FCFS를 사용했다면? (P1->P2->P3->P4 순서): 평균 = (0+6+14+21)/4 = **10.25**

### 4.4 다음 버스트 길이 예측

SJF는 **최적** 이지만 실제로 구현이 불가능하다 — **다음 CPU 버스트 길이를 알 수 없기** 때문

해결책: 과거 버스트 이력을 기반으로 **예측**

- 과거 CPU 버스트는 비슷한 길이로 반복되는 경향이 있다
- 예측에 **지수 평균(Exponential Averaging)** 을 사용

### 4.5 지수 평균 — 공식

다음 CPU 버스트를 예측하는 공식:

```text
tau(n+1) = alpha * t(n) + (1 - alpha) * tau(n)
```

| 기호 | 의미 |
|------|------|
| `t(n)` | n번째 **실제** CPU 버스트 길이 (가장 최근 관측값) |
| `tau(n)` | n번째 **예측값** (이전 예측) |
| `tau(n+1)` | 다음 CPU 버스트 **예측값** |
| `alpha` | 가중치 (0 <= alpha <= 1) |

**alpha에 따른 특성:**

| alpha 값 | 의미 |
|----------|------|
| alpha = 0 | `tau(n+1) = tau(n)` — 최근 관측을 무시, 과거 예측만 사용 |
| alpha = 1 | `tau(n+1) = t(n)` — 마지막 관측만 사용, 과거 예측 무시 |
| alpha = 1/2 | 최근과 과거 값의 **균형** (일반적인 선택) |

### 4.6 지수 평균 — 전개

공식을 반복 대입으로 전개:

```text
tau(n+1) = alpha * t(n)
         + (1-alpha) * alpha * t(n-1)
         + (1-alpha)^2 * alpha * t(n-2)
         + ...
         + (1-alpha)^j * alpha * t(n-j)
         + ...
         + (1-alpha)^(n+1) * tau(0)
```

- alpha < 1이면 **(1 - alpha)** 도 < 1 -> 과거로 갈수록 가중치가 **지수적으로 감소**
- 가장 최근 관측값이 예측에 **가장 큰 영향** 을 미친다
- **"지수 평균"** 이라는 이름의 유래: 가중치가 지수적으로 감쇠

> **[확률/통계]** 지수 평균은 신호 처리와 시계열 분석에서 광범위하게 사용되는 지수 가중 이동 평균(EWMA)의 특수한 경우이다. 매개변수 alpha는 추정기의 "메모리"를 제어한다: alpha가 높으면 메모리가 짧고(변화에 빠르게 반응), alpha가 낮으면 메모리가 길다(더 안정적인 예측). 네트워킹에서 TCP는 왕복 시간을 추정할 때 유사한 접근법을 사용한다.

### 4.7 지수 평균 — 계산 예제

alpha = 0.5, tau(0) = 10으로 시작:

| n | t(n) | tau(n) | 계산 |
|---|------|--------|------|
| 0 | 6 | **10** | 초기값 |
| 1 | 4 | **8** | 0.5*6 + 0.5*10 |
| 2 | 6 | **6** | 0.5*4 + 0.5*8 |
| 3 | 4 | **6** | 0.5*6 + 0.5*6 |
| 4 | 13 | **5** | 0.5*4 + 0.5*6 |
| 5 | 13 | **9** | 0.5*13 + 0.5*5 |
| 6 | 13 | **11** | 0.5*13 + 0.5*9 |
| 7 | - | **12** | 0.5*13 + 0.5*11 |

*Silberschatz, Figure 5.4 — 다음 CPU 버스트 길이의 예측*

- 실제 버스트가 갑자기 변해도 예측값은 점진적으로 따라간다
- 같은 값이 연속으로 반복되면 예측이 수렴한다

### 4.8 지수 평균 — 알파 값 비교

같은 버스트 시퀀스에 대해 alpha를 달리한 예측 차이 (tau_0 = 10, 버스트 = [6, 4, 6]):

```text
               alpha = 0.2    alpha = 0.5    alpha = 0.8
tau(0) = 10        10             10             10
t(0) = 6
tau(1)             9.2            8.0            6.8
t(1) = 4
tau(2)             8.16           6.0            4.56
t(2) = 6
tau(3)             7.73           6.0            5.71
```

- **alpha가 작을 때**: 예측값이 **느리게** 변한다 (이력에 더 의존)
- **alpha가 클 때**: 예측값이 최근 값에 **빠르게 반응**
- 실무에서는 **alpha = 0.5** 근처 값이 일반적으로 사용된다

## 5. SRTF — 선점형 SJF

### 5.1 SRTF (최단 잔여 시간 우선)

**SJF의 선점형 버전** = SRTF

작동 방식:
- 새 프로세스가 도착할 **때마다 비교** 를 수행
- 새 프로세스의 버스트 < 현재 프로세스의 **잔여 버스트** -> **선점 (교체)**
- 그렇지 않으면 현재 프로세스가 계속 실행

| 비교 | SJF (비선점형) | SRTF (선점형) |
|------|---------------|--------------|
| 비교 시점 | CPU가 비었을 때만 | 새 프로세스가 도착할 **때마다** |
| 비교 대상 | 전체 버스트 길이 | **잔여** 버스트 길이 |
| 선점 | 없음 | 있음 |

### 5.2 SRTF — 단계별 예제

| 프로세스 | 도착 시각 | 버스트 시간 |
|---------|----------|-----------|
| P1 | 0 | 8 |
| P2 | 1 | 4 |
| P3 | 2 | 9 |
| P4 | 3 | 5 |

**시각 0**: P1만 도착 -> P1 실행 시작 (잔여: 8)

**시각 1**: P2 도착 (버스트=4) vs P1 (잔여=7) -> 4 < 7 -> **P1 선점, P2 실행**

**시각 2**: P3 도착 (버스트=9) vs P2 (잔여=3) -> 9 > 3 -> **P2 계속**

**시각 3**: P4 도착 (버스트=5) vs P2 (잔여=2) -> 5 > 2 -> **P2 계속**

**시각 5**: P2 완료 -> 준비: P1(잔여=7), P3(9), P4(5) -> **P4 선택** (5가 최소)

**시각 10**: P4 완료 -> 준비: P1(잔여=7), P3(9) -> **P1 선택**

**시각 17**: P1 완료 -> 준비: P3(9) -> **P3 선택**

**시각 26**: P3 완료 -> 모든 프로세스 종료

> **시험 팁:** SRTF 문제를 풀 때는 새 프로세스가 도착할 때마다 선점이 발생하는지 항상 확인하라. 비교는 항상 새 프로세스의 *전체* 버스트와 현재 프로세스의 *잔여* 버스트 사이에서 이루어진다. 새 프로세스가 도착하지 않는 시점에서는 비교가 필요 없다 — 현재 프로세스가 계속된다.

### 5.3 SRTF — 간트 차트와 계산

```text
간트 차트:
+----+----------+------------+-----------------+--------------------+
| P1 |    P2    |     P4     |       P1        |         P3         |
+----+----------+------------+-----------------+--------------------+
0    1          5            10                17                   26
```

| 프로세스 | 도착 | 버스트 | 완료 | 반환 시간 | 대기 시간 |
|---------|------|-------|------|----------|----------|
| P1 | 0 | 8 | 17 | 17-0 = 17 | 17-8 = **9** |
| P2 | 1 | 4 | 5 | 5-1 = 4 | 4-4 = **0** |
| P3 | 2 | 9 | 26 | 26-2 = 24 | 24-9 = **15** |
| P4 | 3 | 5 | 10 | 10-3 = 7 | 7-5 = **2** |

```text
평균 대기 시간    = (9 + 0 + 15 + 2) / 4 = 26/4 = 6.5 ms
평균 반환 시간    = (17 + 4 + 24 + 7) / 4 = 52/4 = 13.0 ms
```

### 5.4 SJF vs SRTF — 동일 데이터 비교

동일 데이터로 비선점형 SJF와 비교 (P1:0,8 / P2:1,4 / P3:2,9 / P4:3,5):

```text
비선점형 SJF:
+------------------+----------+------------+--------------------+
|        P1        |    P2    |     P4     |         P3         |
+------------------+----------+------------+--------------------+
0                  8         12           17                   26
```

| 알고리즘 | P1 대기 | P2 대기 | P3 대기 | P4 대기 | **평균** |
|---------|--------|--------|--------|--------|---------|
| **SJF** | 0 | 7 | 15 | 9 | **7.75** |
| **SRTF** | 9 | 0 | 15 | 2 | **6.5** |

- SRTF가 평균 대기 시간에서 **우수** (6.5 < 7.75)
- 선점을 통해 짧은 프로세스를 더 빠르게 처리할 수 있다

## 6. 라운드 로빈 스케줄링

### 6.1 라운드 로빈 (RR)

각 프로세스에 **고정된 타임 퀀텀(q)** 을 할당하고 순환하며 실행:

1. 준비 큐를 **원형 큐** 로 관리
2. 각 프로세스는 최대 **q 시간 단위** 동안 CPU를 사용
3. q 시간 후에도 실행 중이면 -> **선점**, 준비 큐의 **끝** 으로 이동
4. CPU 버스트가 q보다 짧으면 -> 자발적으로 CPU를 반환

- **선점형** 스케줄링
- 본질적으로 **선점** 이 추가된 FCFS

> 라운드 로빈은 스케줄링의 "돌아가며 하기"와 같다. 학생들이 5분씩 돌아가며 발표하듯이, 프로세스들은 q 시간 단위씩 돌아가며 CPU를 사용한다. 어떤 프로세스도 CPU를 독점할 수 없고, 모든 프로세스가 공정한 몫을 받는다. 이것이 데스크톱과 서버 같은 대화형 시스템의 자연스러운 선택인 이유이다.

### 6.2 RR — 공정성 보장

n개의 프로세스와 타임 퀀텀 = q일 때:

```text
각 프로세스는 CPU 시간의 1/n을 받는다 (최대 q 단위의 청크로)

최대 대기 시간: (n - 1) * q

예시: n = 5, q = 20ms
-> 각 프로세스가 100ms마다 최대 20ms의 CPU를 받음
-> 다시 CPU를 받기까지 최대 80ms 대기
```

- 어떤 프로세스도 **(n-1) * q** 이상 대기하지 않는다
- **응답 시간** 보장에 유리
- 모든 프로세스에 **공정한** CPU 할당

### 6.3 RR — 예제 (q = 4ms)

모든 프로세스가 시각 0에 도착, 타임 퀀텀 = 4ms:

| 프로세스 | 버스트 시간 |
|---------|-----------|
| P1 | 24 ms |
| P2 | 3 ms |
| P3 | 3 ms |

```text
간트 차트:
+------+----+----+------+------+------+------+------+------+
|  P1  | P2 | P3 |  P1  |  P1  |  P1  |  P1  |  P1  |  P1  |
+------+----+----+------+------+------+------+------+------+
0      4    7    10     14     18     22     26     30
```

- P1: 4ms 실행 후 선점 -> P2: 3ms (< q) 자발적 반환 -> P3: 3ms 반환 -> P1 반복

| 프로세스 | 실행 구간 | 대기 시간 | 반환 시간 |
|---------|----------|----------|----------|
| P1 | 0-4, 10-14, 14-18, 18-22, 22-26, 26-30 | 10-4 = **6** | **30** |
| P2 | 4-7 | **4** | **7** |
| P3 | 7-10 | **7** | **10** |

```text
평균 대기 시간 = (6 + 4 + 7) / 3 = 17/3 = 5.67 ms
```

- FCFS (P1,P2,P3 순서): 평균 대기 = **17 ms**
- RR (q=4): 평균 대기 = **5.67 ms** -> 상당한 개선!

### 6.4 RR — 타임 퀀텀이 너무 크거나 작을 때

**q가 매우 클 때** (모든 버스트보다 클 때):

```text
q = 100일 때 (모든 버스트 < 100):
+----------------------------+-----+-----+
|            P1(24)          |P2(3)|P3(3)|
+----------------------------+-----+-----+
0                            24    27    30

-> FCFS와 완전히 동일하게 동작!
-> 선점이 절대 발생하지 않음
```

- 타임 퀀텀이 **모든 CPU 버스트보다 크면** RR = FCFS
- 응답 시간 개선 효과가 사라진다

**q가 매우 작을 때** (예: q = 1ms):

*Silberschatz, Figure 5.5 — 타임 퀀텀이 작을수록 컨텍스트 스위치가 증가하는 방식*

- 컨텍스트 스위치 횟수가 **급격히 증가** -> 오버헤드 증가
- 컨텍스트 스위치 시간이 퀀텀의 10%를 초과하면 -> **비효율적**
- 실제 시스템: 컨텍스트 스위치 ~ **10 us**, 퀀텀 ~ **10-100 ms**

### 6.5 RR — 타임 퀀텀과 반환 시간

q가 반환 시간에 미치는 영향 (3개 프로세스, 각 버스트 = 10):

*Silberschatz, Figure 5.6 — 타임 퀀텀에 따른 반환 시간의 변화*

**경험 법칙(Rule of Thumb)**:
- 이상적으로 CPU 버스트의 **80%** 가 타임 퀀텀 내에 완료되어야 한다
- q가 너무 작으면: 컨텍스트 스위치 오버헤드로 반환 시간 악화
- q가 너무 크면: FCFS처럼 동작하여 응답 시간 악화
- 적절한 q를 선택하는 것이 **매우 중요**

> **시험 팁:** 80% 규칙은 기억해야 할 핵심 경험법칙이다. 타임 퀀텀은 대부분의 프로세스가 단일 퀀텀 내에 CPU 버스트를 완료할 수 있을 만큼 충분히 커야 하고(불필요한 선점 방지), 긴 버스트 프로세스가 CPU를 독점하지 않을 만큼 충분히 작아야 한다. 실제 시스템에서의 일반적인 값은 10-100 ms이다.

### 6.6 RR — 상세 예제 (4개 프로세스)

q = 3ms, 모든 프로세스가 시각 0에 도착:

| 프로세스 | 버스트 시간 |
|---------|-----------|
| P1 | 8 ms |
| P2 | 5 ms |
| P3 | 3 ms |
| P4 | 6 ms |

```text
간트 차트 (q = 3):
+-----+-----+-----+-----+-----+-----+-----+----+
| P1  | P2  | P3  | P4  | P1  | P2  | P4  | P1 |
+-----+-----+-----+-----+-----+-----+-----+----+
0     3     6     9     12    15    17    20   22
```

- P3은 버스트=3=q -> 한 턴에 완료
- P2는 버스트=5 -> 두 턴에 완료 (3 + 2)
- P4는 버스트=6 -> 두 턴에 완료 (3 + 3)
- P1은 버스트=8 -> 세 턴에 완료 (3 + 3 + 2)

| 프로세스 | 버스트 | 완료 | 반환 시간 | 대기 시간 |
|---------|-------|------|----------|----------|
| P1 | 8 | 22 | 22 | **14** |
| P2 | 5 | 17 | 17 | **12** |
| P3 | 3 | 9 | 9 | **6** |
| P4 | 6 | 20 | 20 | **14** |

```text
평균 대기 시간    = (14 + 12 + 6 + 14) / 4 = 46/4 = 11.5 ms
평균 반환 시간    = (22 + 17 + 9 + 20) / 4 = 68/4 = 17.0 ms
```

- RR은 SJF보다 평균 반환 시간이 길지만 **응답 시간은 우수**
- 모든 프로세스가 비교적 **공정하게** CPU를 사용

## 7. 알고리즘 비교

### 7.1 동일 데이터에 대한 4가지 알고리즘

데이터: P1(0,24), P2(0,3), P3(0,3) — 모두 시각 0에 도착

| 알고리즘 | P1 대기 | P2 대기 | P3 대기 | **평균 WT** | 비고 |
|---------|--------|--------|--------|-----------|------|
| FCFS (P1->P2->P3) | 0 | 24 | 27 | **17.0** | 최악의 경우 |
| FCFS (P2->P3->P1) | 6 | 0 | 3 | **3.0** | 순서 의존적 |
| SJF | 6 | 0 | 3 | **3.0** | 최적 |
| RR (q=4) | 6 | 4 | 7 | **5.67** | 공정성 |

- **SJF** 는 평균 대기 시간에서 최적
- **RR** 은 응답 시간 면에서 유리
- **FCFS** 는 순서에 따른 높은 성능 변동성

### 7.2 종합 특성 비교

| 특성 | FCFS | SJF | SRTF | RR |
|------|------|-----|------|-----|
| 선점 | 아니오 | 아니오 | **예** | **예** |
| 최적 WT | 아니오 | **예** | **예** (선점형) | 아니오 |
| 기아 | 아니오 | 가능 | 가능 | **아니오** |
| 응답 시간 | 나쁨 | 보통 | 보통 | **우수** |
| 구현 복잡도 | 매우 간단 | 버스트 예측 필요 | 버스트 예측 필요 | 타이머 필요 |
| 호위 효과 | **발생** | 없음 | 없음 | 없음 |

> 이 표는 시험에서 매우 유용한 참고 자료이다. 트레이드오프를 주목하라: SJF/SRTF는 대기 시간에서 최적이지만 기아(starvation)를 유발할 수 있다(짧은 프로세스가 계속 도착하면 긴 프로세스가 무한정 대기할 수 있다). RR은 기아를 방지하고 우수한 응답 시간을 가지지만 평균 대기 시간이 더 높다. FCFS는 간단하지만 전반적인 특성이 가장 나쁘다. 이러한 트레이드오프가 7주차의 더 고급 알고리즘(우선순위 스케줄링, 다단계 큐)의 동기가 된다.

## 8. 실습 — 스케줄링 알고리즘 시뮬레이터

### 8.1 실습: 자료 구조

```python
class Process:
    def __init__(self, pid, arrival, burst):
        self.pid = pid
        self.arrival = arrival
        self.burst = burst
        self.remaining = burst    # SRTF, RR에서 사용
        self.completion = 0
        self.waiting = 0
        self.turnaround = 0
        self.first_run = -1       # 응답 시간 계산용

def print_gantt(timeline):
    """timeline: [(pid, start, end), ...]"""
    header = "|"
    footer = ""
    for pid, start, end in timeline:
        width = max(len(f" P{pid} "), end - start)
        header += f" P{pid} ".center(width) + "|"
    print(header)
    # 시간 표시
    print("0", end="")
    for pid, start, end in timeline:
        print(f"{'':>{end - start + 2}}{end}", end="")
    print()
```

### 8.2 실습: FCFS 구현

```python
def fcfs(processes):
    """선입선처리 스케줄링"""
    # 도착 순서로 정렬
    procs = sorted(processes, key=lambda p: p.arrival)
    time = 0
    timeline = []

    for p in procs:
        if time < p.arrival:
            time = p.arrival  # CPU 유휴

        start = time
        time += p.burst
        p.completion = time
        p.turnaround = p.completion - p.arrival
        p.waiting = p.turnaround - p.burst
        timeline.append((p.pid, start, time))

    return timeline

# 테스트
procs = [Process(1,0,24), Process(2,0,3), Process(3,0,3)]
fcfs(procs)
# -> 평균 대기 시간 = 17.0
```

### 8.3 실습: SJF 구현 (비선점형)

```python
def sjf(processes):
    """최단 작업 우선 (비선점형)"""
    procs = [p for p in processes]  # 복사
    n = len(procs)
    completed = 0
    time = 0
    done = [False] * n
    timeline = []

    while completed < n:
        # 도착한 프로세스 중 가장 짧은 버스트 선택
        candidates = [(i, procs[i]) for i in range(n)
                      if not done[i] and procs[i].arrival <= time]

        if not candidates:
            time += 1
            continue

        idx, p = min(candidates, key=lambda x: x[1].burst)
        start = time
        time += p.burst
        p.completion = time
        p.turnaround = p.completion - p.arrival
        p.waiting = p.turnaround - p.burst
        done[idx] = True
        completed += 1
        timeline.append((p.pid, start, time))

    return timeline
```

### 8.4 실습: RR 구현

```python
from collections import deque

def round_robin(processes, quantum):
    """라운드 로빈 스케줄링"""
    procs = sorted(processes, key=lambda p: p.arrival)
    for p in procs:
        p.remaining = p.burst

    queue = deque()
    time = 0
    idx = 0
    timeline = []

    # 시각 0에 도착하는 프로세스 추가
    while idx < len(procs) and procs[idx].arrival <= time:
        queue.append(procs[idx])
        idx += 1

    while queue:
        p = queue.popleft()
        run_time = min(quantum, p.remaining)
        start = time
        time += run_time
        p.remaining -= run_time

        # 이 구간 동안 도착한 프로세스 추가
        while idx < len(procs) and procs[idx].arrival <= time:
            queue.append(procs[idx])
            idx += 1

        if p.remaining > 0:
            queue.append(p)  # 미완료 -> 뒤로 이동
        else:
            p.completion = time
            p.turnaround = p.completion - p.arrival
            p.waiting = p.turnaround - p.burst

        timeline.append((p.pid, start, time))

    return timeline
```

### 8.5 실습: 테스트 및 결과 비교

```python
# 테스트 데이터
data = [(1, 0, 10), (2, 1, 5), (3, 2, 8), (4, 3, 6)]

print("=== FCFS ===")
procs = [Process(*d) for d in data]
fcfs(procs)
for p in procs:
    print(f"P{p.pid}: WT={p.waiting}, TT={p.turnaround}")

print("\n=== SJF ===")
procs = [Process(*d) for d in data]
sjf(procs)
for p in procs:
    print(f"P{p.pid}: WT={p.waiting}, TT={p.turnaround}")

print("\n=== RR (q=4) ===")
procs = [Process(*d) for d in data]
round_robin(procs, 4)
for p in procs:
    print(f"P{p.pid}: WT={p.waiting}, TT={p.turnaround}")
```

테스트 데이터: P1(0,10), P2(1,5), P3(2,8), P4(3,6) — 3가지 알고리즘 결과를 비교.

> **이전 실습과의 비교**: 4-5주차에서는 스레드를 사용하여 계산을 병렬화했다. 여기서는 OS 자체의 스케줄링 로직을 구현하고 있다. 알고리즘의 선택이 프로세스 동작을 극적으로 바꾸는 것에 주목하라 — 동일한 프로세스 집합이 어떤 알고리즘을 사용하느냐에 따라 매우 다른 대기 시간과 반환 시간을 갖는다. 이것이 CPU 스케줄링이 가장 중요한 OS 설계 결정 중 하나로 여겨지는 이유이다.

## 요약

| 개념 | 핵심 요약 |
|:-----|:---------|
| CPU-I/O 버스트 | 프로세스는 CPU 버스트와 I/O 버스트를 **교대** 로 수행 |
| CPU 스케줄러 | 준비 큐에서 다음 프로세스를 **선택** (단기 스케줄러) |
| 선점형 vs 비선점형 | 대부분의 현대 OS는 **선점형** 사용 (4가지 상황 모두) |
| 디스패처 | 컨텍스트 스위치 수행, 사용자 모드 전환, PC로 점프 |
| 스케줄링 기준 | CPU 이용률, 처리량 **최대화** / 반환 시간, 대기 시간, 응답 시간 **최소화** |
| FCFS | 간단하지만 **호위 효과** 발생 |
| SJF | 최적이지만 버스트 **예측 불가능** (지수 평균) |
| SRTF | SJF의 **선점형 버전** |
| RR | 타임 퀀텀을 통한 **공정한** CPU 분배, 좋은 응답 시간 |
| 교재 범위 | Silberschatz Ch 5, Sections 5.1–5.3 |
