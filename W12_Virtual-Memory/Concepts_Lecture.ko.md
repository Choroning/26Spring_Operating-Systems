## 1. 배경과 Demand Paging

### 1.1 가상 메모리가 주는 것

**가상 메모리(virtual memory)** 는 **논리(가상) 메모리와 물리 메모리를 분리** 하는 기법이다. 11주차의 페이징 위에 세워져 페이지 테이블의 간접층을 강력한 거짓말로 바꾼다: *프로세스는 거대하고 연속된 주소 공간을 가지고 있다고 믿지만, OS는 실제로 사용되는 부분만 조용히 제공한다*.

왜 필요한가? 구조적 이점 셋:

- **프로그램이 물리 RAM보다 클 수 있다.** 실제 건드리는 페이지만 메모리에 상주하면 됨. 에러 처리 코드, 사용하지 않는 기능, 과도하게 선언된 배열은 참조되기 전까지 물리 메모리를 0 만큼 차지.
- **더 많은 프로세스가 동시에 RAM에 들어간다.** 다중 프로그래밍 정도 증가 → CPU 사용률 증가(6주차).
- **빠른 프로세스 시작.** 새 프로세스는 일부 페이지만 메모리에 있으면 즉시 실행 가능; 나머지는 요청 시 스트리밍.

같은 기제를 타고 오는 보너스:

- **공유 프레임** 으로 공유 라이브러리(11주차 §6.2가 토대).
- **공유 메모리** 를 통한 프로세스 간 통신.
- copy-on-write를 통한 **빠른 `fork()`** (§2).

> **왜 이것이 옳은 추상화인가:** 11주차의 주소 지정 계층(페이지 테이블, MMU, TLB)은 *공간적* 가상화를 준다. Demand paging은 *시간적* 가상화를 추가한다 — 페이지는 CPU가 참조하는 순간에만 RAM에 존재하면 되고 사전에는 그럴 필요가 없다. 이번 강의의 모든 것은 "부재 페이지를 싸게 만들라"의 어떤 귀결이다.

### 1.2 프로세스의 가상 주소 공간

각 프로세스는 자기만의 독립된 가상 주소 공간을 본다. 보통 0부터 시작해 *연속되어 보이지만*, 그 아래의 물리 프레임은 RAM 어디든 흩어져 있거나 일부는 디스크에 있을 수 있다.

```
가상 주소 공간 (한 프로세스):

  HIGH ┌─────────────┐
       │   Stack     │  ← 아래로 자람
       ├─────────────┤
       │             │
       │   HOLE      │  ← 물리 메모리 소비 없음
       │             │
       ├─────────────┤
       │   Heap      │  ← 위로 자람
       ├─────────────┤
       │   Data      │
       ├─────────────┤
       │   Text      │  ← 읽기 전용 코드
   LOW └─────────────┘
```

- **희소 주소 공간(sparse address space)**: text/data, heap, stack 사이에 거대한 *홀*.
- 홀은 **물리 메모리를 차지하지 않음** — 프레임을 가리키는 PTE가 없다.
- 공유 라이브러리는 보통 가운데 영역(heap과 stack 사이)에 매핑된다.

> **핵심 통찰:** 홀이 공짜이기 때문에 프로세스는 거대한 최대 크기를 선언해도 비용을 치르지 않는다. 64비트 프로세스가 2 TB 희소 파일을 mmap 해도, 실제 건드린 페이지만 프레임을 할당.

### 1.3 가상 메모리를 통한 공유 라이브러리

여러 프로세스가 표준 C 라이브러리 같은 라이브러리를 **공유** 가능:

- 라이브러리가 각 프로세스의 가상 주소 공간에 어떤 가상 주소로 매핑됨(편의를 위해 모든 프로세스에서 같은 주소이기도 하지만 필수는 아님).
- 라이브러리 코드를 담는 **물리 프레임은 공유** — 한 세트의 프레임, 그것을 가리키는 다수의 PTE.
- 라이브러리는 **읽기 전용** 으로 매핑되며, 이것이 공유의 안전성을 보장(11주차 §6.2).
- 공유 메모리 IPC도 동일 메커니즘.
- `fork()` 는 초기에 페이지 테이블 복사로 부모와 자식이 모든 페이지를 공유 — 아래의 copy-on-write 참조.

### 1.4 Demand Paging과 Lazy Pager

**Demand paging**: 페이지가 실제로 **참조될 때만** 메모리에 적재.

적재를 수행하는 OS 구성요소를 **lazy swapper** 또는 **pager** 라 부른다 — 명확히 *게으름*: 어떤 페이지도 투기적으로 가져오지 않음.

장점:
- **I/O 감소** — 건드리지 않는 페이지는 디스크에서 절대 읽지 않음.
- **메모리 절약** — working set만 상주.
- **빠른 시작** — 첫 명령어 페이지가 있는 즉시 실행 시작.
- **더 많은 동시 프로세스** — 각자 RAM 사용 적음.

### 1.5 유효/무효 비트

각 페이지 테이블 엔트리(PTE)는 **유효/무효 비트** 를 가짐(11주차 §6.1, 이제 demand paging 관점에서 재해석):

- **유효(Valid)** — 페이지가 합법적 *이면서* 현재 물리 프레임에 상주.
- **무효(Invalid)** — 페이지가 프로세스 주소 공간 밖이거나, **또는** 합법이지만 현재 디스크에 있음.

CPU는 두 종류의 "무효"를 구분하지 않음 — 둘 다 같은 하드웨어 트랩 발생. 어떤 종류인지는 OS의 트랩 핸들러가 판단.

> **왜 이 설계가 깔끔한가:** "존재 vs 상주" 검사가 MMU가 이미 수행 중인 보호 검사와 합쳐짐. MMU가 유효 비트를 지원하기만 하면 demand paging은 본질적으로 "공짜" — 데이터 경로 변경 없이 트랩 위의 OS 로직만 추가.

### 1.6 페이지 폴트 처리 — 단계별

CPU가 PTE가 무효인 페이지를 건드리면 **페이지 폴트 트랩** 으로 커널 제어 이전:

```
1. CPU가 가상 주소 참조  →  MMU가 PTE 확인.
2. 유효 비트 = invalid  →  OS로 페이지 폴트 트랩.
3. OS가 분류:
     (a) 불법 주소  →  SIGSEGV 전달, 프로세스 종료.
     (b) 합법이지만 디스크에 있음 → 계속.
4. OS가 free-frame list에서 빈 프레임 선택
     (리스트가 비었으면 페이지 교체 실행 — §3).
5. OS가 디스크 I/O로 페이지를 그 프레임으로 읽음.
6. OS가 PTE 갱신: 프레임 번호, 유효 비트 세팅, 권한 세팅.
7. OS가 페이지 폴트를 일으킨 명령어를 RESTART.
8. 명령어가 재실행됨; 이번엔 PTE 유효 → 성공.
```

전체 시퀀스 — 회전 디스크 기준 보통 5–10 ms, SSD 수백 μs, NVMe 수십 μs — 는 일반 메모리 접근(~100 ns) 대비 **수 자릿수 더 느림**. 이것이 페이지 폴트 율을 극소로 유지해야 하는 이유.

### 1.7 Pure Demand Paging과 하드웨어 요구사항

**Pure demand paging** 은 극단 전략: **어떤** 페이지도 메모리에 두지 않고 프로세스 시작. 첫 명령어도 페이지 폴트 유발. 실행이 진행되면서 **참조의 지역성** 이 working set을 빠르게 채우고 폴트 율이 급락.

Demand paging을 지원하기 위해 **필요한 하드웨어**:

- **유효/무효 비트 (와 보호 비트)를 가진 페이지 테이블**.
- 상주하지 않는 페이지를 담을 **보조 메모리(swap 공간)**.
- **명령어 재시작(instruction restart) 기능** — CPU가 폴트 처리 후 트랩된 명령어를 깔끔히 재실행할 수 있어야 함.

### 1.8 명령어 재시작의 어려움

명령어 재시작은 개념적으로 단순하지만 아키텍처적으로 까다롭다:

- **쉬운 경우** — 명령어 *fetch* 중 폴트: 그냥 다시 fetch하고 시작.
- **어려운 경우** — 다단계 명령어 `ADD A, B, C`:
  1. 명령어 fetch & decode.
  2. A의 피연산자 fetch.
  3. B의 피연산자 fetch.
  4. A + B 계산.
  5. 결과를 C에 store — **여기서 페이지 폴트!**
  - 단순히 재실행하면 1–4가 반복되는데, 부작용이 없으니 괜찮음. 하지만 *부작용이 있는* 명령어는 어떻게?
- **IBM MVC 명령어** — 최대 256 바이트 복사; 목적지가 페이지 경계를 가로지르고 두 번째 페이지가 무효이면 복사 도중 폴트.
  - **해법 1:** 어떤 바이트를 쓰기 전에 양쪽 끝 주소를 미리 검사; 어느 쪽이든 폴트 날 거면 *부작용 전에* 미리 폴트.
  - **해법 2:** 덮어쓸 목적지 바이트들을 임시 레지스터에 저장; 폴트 시 트랩 핸들러로 가기 전에 복원.

> **이게 역사적으로 중요한 이유:** 명령어 재시작이 가능한가가 그 아키텍처가 demand paging을 지원할 수 있는가를 결정한다. 초기 일부 CPU는 이를 깔끔히 지원할 수 없었고, 그래서 가상 메모리가 보편화되는 데 한 세대의 하드웨어 설계가 필요.

### 1.9 Free-Frame List와 Zero-Fill-on-Demand

OS는 **free-frame list** 를 유지 — 할당 가능한 물리 프레임의 풀.

- 시스템 부팅 시 *모든* 사용자 가용 메모리를 리스트에 등록.
- 페이지 폴트 처리가 리스트에서 한 프레임을 꺼내 사용.
- 리스트가 임계값 이하로 줄면 **페이지 교체**(§3)가 희생자를 추방해 리스트 보충.

**Zero-fill-on-demand**: 새 프로세스에게 프레임을 넘기기 전에 커널이 **0으로 초기화**. 왜? *보안* — 그 프레임은 다른 프로세스의 것이었고, 남은 내용이 새 프로세스에 누출되면 안 됨.

### 1.10 Demand-Paging 성능 — EAT

기호:
- $p$ = 페이지 폴트 율, $0 \leq p \leq 1$
- $\text{mem}$ = 메모리 접근 시간
- $\text{pf}$ = 페이지 폴트 서비스 시간(인터럽트 + 디스크 I/O + 프로세스 재시작)

$$
\text{EAT} = (1 - p) \cdot \text{mem} + p \cdot \text{pf}
$$

**예시** — $\text{mem} = 200\ \text{ns}$, $\text{pf} = 8\ \text{ms} = 8\,000\,000\ \text{ns}$:

$$
\text{EAT} = 200 + 7\,999\,800 \cdot p
$$

- $p = 1/1000$ (1000 접근당 1번 폴트): $\text{EAT} = 200 + 7999.8 \approx 8200\ \text{ns}$ — 원시 메모리 대비 **40배 느림**!
- 저하를 10% 이내로 유지($\text{EAT} \leq 220$): $p < 0.0000025 \approx 1/400\,000$.

> **교훈:** 페이지 폴트는 **극히 드물어야** 한다 — 약 1/400,000 참조당 1회 이하 — demand paging이 공짜처럼 느껴지려면. 이것이 이번 강의의 나머지(좋은 교체, working set, 지역성)가 중요한 이유: 폴트 율이 조금만 올라가도 유효 접근 시간이 크게 증가.

### 1.11 Swap 공간 활용

Swap 공간은 디스크 상의 영역이고, 거기에서 읽는 것은 보통 일반 파일 시스템 읽기보다 빠르다 — 이유:
- OS가 **더 큰 블록 전송** 사용(파일별 lookup 오버헤드 없음).
- 파일 시스템 메타데이터 간접 참조 없음.

페이지가 어디에 "사는가"에 대한 두 전략:

- **전략 1:** 프로세스 시작 시 실행 파일 전체를 swap에 복사한 뒤 swap에서 demand paging.
- **전략 2:** 처음에는 파일 시스템에서 demand paging; 페이지가 추방될 때 *dirty* 버전을 swap에 쓰고 이후엔 swap에서 읽음. **Linux와 Windows** 가 이 방식.

미묘한 점:
- **바이너리 실행 파일 페이지**(text, 읽기 전용 데이터)는 파일 시스템에서 옴; 절대 수정되지 않으므로 추방 시 단순히 폐기하고 다음에 다시 파일 시스템에서 읽음.
- **익명(anonymous) 메모리** (stack, heap — 파일 백킹 없음)는 추방 시 swap에 써야 함; 여기에서 swap 공간이 실제로 쓰임.

## 2. Copy-on-Write

### 2.1 COW 아이디어

`fork()` 는 단순 구현이면 부모의 전체 주소 공간을 자식의 새 프레임 셋으로 복제할 것 — 자식이 거의 항상 즉시 `exec()` 를 호출해 모두 버리므로 낭비.

**Copy-on-write (COW)** 는 복사를 정말 필요해질 때까지 미룬다:

```
1. fork():
      부모의 PTE를 복사해 자식의 페이지 테이블 구성.
      부모와 자식의 PTE 모두 같은 물리 프레임을 가리킴.
      모든 공유 쓰기 가능 페이지를 양쪽 PTE에서 READ-ONLY로 표시.

2. 읽기 접근:
      폴트 없음. 양쪽 프로세스가 공유 프레임에서 행복하게 읽음.

3. 쓰기 접근:
      읽기 전용 표시 때문에 protection fault 발생.
      OS가 그 페이지를 새 프레임으로 복사.
      쓰는 쪽의 PTE 갱신: 새 프레임, 다시 쓰기 가능.
      다른 쪽 PTE는 옛 프레임을 가리킨 채 (역시 다시 쓰기 가능).

4. 어느 쪽도 수정하지 않은 페이지는 NEVER 복사됨.
```

**Linux, Windows, macOS** 가 사용. `fork()`-후-`exec()` 워크로드(셸, 서버)에서 성능 이득이 큼.

### 2.2 수정 불가 페이지와 vfork()

- **수정 불가 페이지** (실행 코드)는 COW가 필요 없음 — 본질적으로 항상 공유, 읽기 전용. COW는 *쓰일 수 있는* 페이지에만 관련.
- **`vfork()` (virtual memory fork)** 는 더 공격적 변형:
  - 부모 프로세스가 자식이 종료하거나 `exec()` 호출까지 **suspended(중단)**.
  - 페이지가 **복사도 COW 표시도 없이 공유** — 자식이 글자 그대로 부모의 주소 공간에서 실행.
  - 극히 빠름(페이지 테이블 복사도 없음), 그러나 **위험** — 자식이 최소 셋업-후-exec 이외의 짓을 하면 부모 손상.
  - 자식이 즉시 자신을 교체하는 고전적 UNIX 셸 구현에서 사용.

## 3. 페이지 교체

### 3.1 교체가 필요한 이유

Demand paging은 **over-allocation** 으로 이어진다: OS가 폴트가 날 때마다 페이지를 적재하다 보면 결국 모든 프레임이 사용 중. 그 후의 새 페이지 폴트는 들어올 페이지를 둘 곳이 없음.

**페이지 교체** = 상주 페이지 중 추방할 ("**희생자(victim)**") 페이지를 선택해 들어올 페이지의 자리를 만듦.

이것이 가상 메모리의 거짓말을 완성한다 — 20-페이지 프로세스가 10 프레임으로 실행 가능, 왜냐하면 OS가 가장 유용한 10개를 끊임없이 회전시키기 때문.

### 3.2 기본 교체 절차와 Dirty Bit

```
1. 원하는 페이지를 디스크에서 찾음.
2. 빈 프레임 찾기:
   - free-frame list에 있으면 사용.
   - 없으면 페이지 교체 알고리즘으로 VICTIM 프레임 선택.
       Victim의 dirty bit = 1이면 디스크로 write-back.
       Victim의 dirty bit = 0이면 그냥 폐기 (디스크에 이미 존재).
3. 원하는 페이지를 이제 빈 프레임으로 읽음.
4. 페이지 테이블 갱신 (victim을 invalid로, 새 페이지를 valid로).
5. 폴트 낸 프로세스 재시작.
```

**Dirty bit (modify bit)** 는 페이지가 쓰일 때마다 하드웨어가 세팅. 추방 시점에도 0이면 메모리 사본이 디스크 사본과 동일하므로 write-back을 스킵 가능.

> **왜 중요한가:** swap-out I/O는 페이지 폴트에서 OS가 하는 가장 비싼 단일 작업. Clean 페이지에 대해 이를 건너뛰는 것은 일반 워크로드에서 디스크 트래픽을 대략 절반으로 줄임.

### 3.3 페이지 교체가 주는 것

페이지 교체가 **논리 메모리를 물리 메모리로부터 진정으로 분리** 한다. 교체 없이는 프로세스가 전체 주소 공간이 RAM에 들어가야만 실행 가능. 교체와 함께:

- 20-페이지 프로세스가 10 프레임으로 실행.
- 가상 크기 합이 RAM을 초과하는 다수 프로세스가 편안히 공존.

두 알고리즘적 문제를 답해야 함:

1. **프레임 할당** — 각 프로세스에 몇 프레임을 줄 것인가? (§5)
2. **페이지 교체** — 고정 프레임 예산에서 어떤 페이지를 추방할 것인가? (§3.5 – §4)

목표는 항상 **페이지 폴트 율 최소화** (§1.10의 EAT 계산을 떠올릴 것).

### 3.4 참조 문자열(Reference String)

알고리즘을 공정히 비교하기 위해, 프로세스가 건드린 페이지 번호의 시퀀스인 **참조 문자열** 에 대해 평가.

**예**: 페이지 크기 100 바이트, 바이트 주소
```
0100, 0432, 0101, 0612, 0102, 0103, ...
```
이 페이지 번호 참조 문자열
```
1, 4, 1, 6, 1, 6, 1, 6, 1, 6, 1
```
로 축약.

같은 문자열로 알고리즘을 시뮬레이션해 폴트 수를 셈. 일반적으로 **프레임이 많을수록 폴트 감소** (§3.6의 유명한 예외 제외).

아래에서 사용할 표준 예시 참조 문자열:
```
7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 1, 2, 0, 1, 7, 0, 1
```

### 3.5 FIFO

**FIFO** (First-In, First-Out)는 메모리에 가장 오래 있던 페이지를 교체.

- FIFO 큐로 관리: 새 페이지는 꼬리로, 희생자는 머리에서.
- 구현은 간단하지만 최근 활동과 무관하게 모든 페이지를 동등하게 취급.

**FIFO 추적** — 3 프레임, 참조 문자열 `7 0 1 2 0 3 0 4 2 3 0 3 2 1 2 0 1 7 0 1`:

| 7   | 7    | -    | -    | F      |
| 0   | 7    | 0    | -    | F      |
| 1   | 7    | 0    | 1    | F      |
| 2   | **2**| 0    | 1    | F      |
| 0   | 2    | 0    | 1    |        |
| 3   | 2    | **3**| 1    | F      |
| 0   | 2    | 3    | **0**| F      |
| 4   | **4**| 3    | 0    | F      |
| 2   | 4    | **2**| 0    | F      |
| 3   | 4    | 2    | **3**| F      |
| 0   | **0**| 2    | 3    | F      |
| 3   | 0    | 2    | 3    |        |
| 2   | 0    | 2    | 3    |        |
| 1   | 0    | **1**| 3    | F      |

### 3.6 Belady의 변칙

**프레임 추가는 항상 폴트를 줄인다** 고 기대할 텐데, FIFO에서는 *항상 그렇지는 않다*.

**참조 문자열**: `1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5`

### 3.7 Optimal (OPT / MIN)

**OPT** 알고리즘은 **미래에 가장 오랫동안 사용되지 않을** 페이지를 교체.

- 증명적으로 **가능한 최저 페이지 폴트 율** — 어떤 알고리즘도 더 잘할 수 없음.
- **Belady 변칙 없음** — 프레임 추가가 절대 해를 끼치지 않음.
- 실용에서는 **구현 불가능** (미래에 대한 완벽한 지식 필요).
- 실제 알고리즘 평가를 위한 **기준선**(품질의 상한)으로 사용.

**OPT 추적** — 3 프레임, FIFO와 같은 문자열:

| Ref | F1    | F2    | F3    | Fault? | 이유 |

| 7   | 7     | -     | -     | F      |      |
| 0   | 7     | 0     | -     | F      |      |

### 3.8 LRU (Least Recently Used)

미래를 볼 수 없으니 과거로 근사한다: **가장 오랫동안 사용되지 않은 페이지를 교체**.

- **핵심 가정:** 참조의 지역성 — 오랫동안 idle 이었던 페이지는 곧 필요할 가능성도 낮다.
- 시간을 역방향으로 한 OPT 적용과 동치.
- **Belady 변칙 없음** — LRU는 **스택 알고리즘**(§3.10).

**LRU 추적** — 3 프레임, 같은 참조 문자열:

| Ref | F1    | F2    | F3    | Fault? | 이유 (LRU victim) |
|-----|-------|-------|-------|--------|--------------------|
| 7   | 7     | -     | -     | F      |                    |
| 0   | 7     | 0     | -     | F      |                    |
| 1   | 7     | 0     | 1     | F      |                    |
| 2   | **2** | 0     | 1     | F      | 7이 가장 오래된 사용 |
| 0   | 2     | 0     | 1     |        |                    |
| 3   | 2     | 0     | **3** | F      | 1이 가장 오래된 사용 |
| 0   | 2     | 0     | 3     |        |                    |
| 4   | **4** | 0     | 3     | F      | 2가 가장 오래된 사용 |
| 2   | 4     | 0     | **2** | F      | 3이 가장 오래된 사용 |
| 3   | 4     | **3** | 2     | F      | 0이 가장 오래된 사용 |
| 0   | **0** | 3     | 2     | F      |                    |

| 7   | 1     | 0     | **7** | F      |                    |

| 1   | 1     | 0     | 7     |        |                    |

**총: 12 페이지 폴트.** 비교: **FIFO 15 > LRU 12 > OPT 9.**

### 3.9 LRU 구현

진짜 LRU는 모든 상주 페이지의 **상대적 최근성** 을 알아야 함. 하드웨어 지원 두 가지:

**방법 1 — 카운터 (타임 스탬프).**
- 각 PTE가 "time-of-use" 필드 저장.
- CPU에 논리 시계 추가; 메모리 접근마다 tick.
- 모든 참조에서 시계 값을 참조된 PTE에 복사.
- 희생자 = 가장 작은 타임 스탬프를 가진 페이지.
- **비용:** 접근마다 메모리 쓰기(PTE) + 최소 찾기 위한 상주 페이지 선형 스캔.

**방법 2 — 스택 (이중 연결 리스트).**
- 상주 페이지의 최근성 순 이중 연결 리스트 유지.
- 참조 시 페이지를 스택의 **맨 위** 로 이동 — 최악 6번의 포인터 갱신.
- 희생자 = 스택 **맨 아래** 페이지 — O(1) 조회, 스캔 불필요.

둘 다 하드웨어 도움이 필요; 없으면 소프트웨어 인터럽트로 구현 시 메모리 접근이 한 자릿수 느려짐.

### 3.10 스택 알고리즘과 Belady 회피 이유

교체 알고리즘이 **스택 알고리즘** 이라 함은, 모든 참조 문자열에 대해 $n$ 프레임에서의 상주 페이지 집합이 $n + 1$ 프레임에서의 상주 페이지 집합의 **부분집합** 임을 의미.

- **LRU는 스택 알고리즘:** $n$ 프레임에서 상주 집합은 정확히 "가장 최근 참조된 $n$ 개 페이지". $n + 1$ 프레임에서도 그 $n$ 개 페이지는 여전히 상주 집합 안에 있고 추가로 하나 더.
- **OPT도 스택 알고리즘** (미래에 대한 대칭 논증).
- **FIFO는 스택 알고리즘이 *아님*** — $n$ 프레임에서의 상주 집합이 $n + 1$ 프레임에서의 부분집합이라는 보장이 없음. 정확히 이것이 Belady 변칙을 허용.

**정리(직관):** 임의의 스택 알고리즘에 대해 프레임 추가는 폴트 수를 **절대 증가시키지 않음** — 작은 캐시에서 폴트였을 페이지는 큰 캐시에서도 여전히 폴트(또는 히트)이며 역은 절대 안 됨.

### 3.11 알고리즘 비교

같은 참조 문자열, 3 프레임:

| 알고리즘 | 페이지 폴트 | 비고 |
|----------|-------------|------|
| FIFO     | **15**      | 간단하지만 Belady 변칙 발생 |
| LRU      | **12**      | 성능 좋지만 구현 비용 큼 |
| OPT      | **9**       | 최적이지만 구현 불가 |

## 4. LRU 근사 알고리즘

### 4.1 왜 근사하는가?

진짜 LRU는 **모든** 메모리 참조에서 시계나 스택을 갱신하는 하드웨어를 요구. 대부분의 아키텍처는 그것을 제공하지 않음. 대부분의 아키텍처가 *제공하는* 것은 **reference bit**:

- 페이지를 읽거나 쓸 때 하드웨어가 1로 세팅.
- OS가 자기 일정대로 클리어.
- 어떤 최근 윈도에서 **어떤 페이지가 사용되었는지** 알려주지만 **순서는 아님**.

LRU 근사 알고리즘들은 이 한 비트(또는 몇 비트)를 써서 LRU에 가까운 결정을 훨씬 적은 비용으로 내림.

### 4.2 Additional-Reference-Bits

각 페이지마다 OS가 주기적 타이머 인터럽트(예: 100 ms마다)에 갱신하는 **8비트 바이트**(시프트 레지스터) 유지:

```
매 타이머 tick:
  1. 현재 reference bit를 페이지 바이트의 MSB로 시프트 인.
  2. 바이트의 나머지를 오른쪽으로 1비트 시프트.
  3. 하드웨어 reference bit 클리어 (다음 구간 준비).
```

```
예시 히스토리:
  11000100  →  최근 3 구간 중 2번 사용, 그 후 한 번 더
  01110111  →  최근에는 사용 없지만 과거 사용 많음

unsigned 정수로 비교하면 11000100 > 01110111
→ 두 번째 페이지가 더 좋은 교체 후보(작은 히스토리 값)
```

- 희생자: **가장 작은** 바이트 값을 가진 페이지(= 가장 최근 사용 적음).
- 동점은 FIFO 순서로.
- 히스토리 비트 수 늘리면 정확도↑이지만 인터럽트 비용↑.

### 4.3 Second-Chance (Clock) 알고리즘

**FIFO** 와 **reference bit** 결합:

```
페이지를 FIFO 순서로 유지(원형 큐).
교체가 필요할 때:
   FIFO 머리 페이지 본다.
   reference bit = 0 이면 → 이 페이지를 REPLACE.
   reference bit = 1 이면 → 비트를 클리어, 페이지를 "방금 도착한" 것처럼 취급,
                            FIFO 다음 페이지로 진행, 반복.
```

손 포인터를 가진 **원형 큐** 로 구현: 알고리즘이 정확히 손이 도는 시계처럼 보임 — 그래서 **Clock 알고리즘**.

- 최악(모든 비트 = 1): 한 바퀴 돌면서 모든 비트 클리어; 다음 바퀴는 순수 FIFO로 회귀.
- 자주 쓰는 페이지는 손이 도달할 때 ref bit = 1 → 사실상 *절대* 교체되지 않음.

> **왜 작동하는가:** reference bit가 이진 "최근 사용했는가" 신호를 줌. Clock은 그 한 비트를 LRU에 가까운 것으로 바꿈 — 최근 사용된 페이지에 면제권을 주고 진짜 오래된 페이지만 추방되도록.

### 4.4 Clock 알고리즘 — 시각화

```
[초기 상태]                    [검색 후]
   v (포인터)                    v (포인터, 새 victim 지정)
+------+                    +------+
| P1: 1| → bit=0 으로 클리어 | P1: 0|
| P2: 1| → bit=0 으로 클리어 | P2: 0|
| P3: 0| → VICTIM!           | NEW  | ← 들어오는 페이지 삽입
| P4: 1|                     | P4: 1|
| P5: 0|                     | P5: 0|
+------+                    +------+

P1, P2 생존 — 비트 클리어로 "두 번째 기회" 받음.
P3 교체 — 비트가 이미 0 이었음.
```

### 4.5 Enhanced Second-Chance — (R, M) 클래스

**reference bit (R)** 와 **modify (dirty) bit (M)** 모두 사용해 clock 정교화. 각 페이지가 **4 클래스** 중 하나:

| 클래스 | (R, M) | 의미                              | 교체 우선순위 |
|--------|--------|-----------------------------------|----------------|
| 1      | (0, 0) | 최근 미사용, 미수정                 | **최우선**(폐기 싸다) |
| 2      | (0, 1) | 최근 미사용, 수정됨                 | 2순위 — write-back 필요 |
| 3      | (1, 0) | 최근 사용됨, 미수정                 | 3순위           |
| 4      | (1, 1) | 최근 사용됨 **및** 수정됨            | **최후순위**    |

낮은 클래스 페이지를 먼저 추방. 수정된 페이지 교체는 추가 디스크 쓰기 비용 — clean 페이지를 선호해 추방하고, 최근이면서 dirty인 페이지는 절대로 먼저 추방하지 않음.

### 4.6 LFU와 MFU

각 페이지의 참조 횟수를 셈:

- **LFU (Least Frequently Used)** — **가장 적은** 참조의 페이지 추방.
  - 문제: 초기화 동안 무겁게 쓰이고 그 뒤 안 쓰는 페이지가 큰 카운트로 영원히 살아남음.
  - 해법: 주기적으로 모든 카운터를 오른쪽 시프트(지수적 감쇠)해서 오래된 활동을 약화.
- **MFU (Most Frequently Used)** — **가장 많은** 참조의 페이지 추방.
  - 논리: 작은 카운트의 페이지는 방금 도착해 *앞으로* 쓰일 가능성 큼; 많이 참조된 페이지는 이미 할 일을 다 함.

어느 쪽도 일반적으로 OPT를 잘 근사하지 못함. **실전에서 거의 쓰이지 않음** — 구현 비용 대비 (소박한) 이득.

### 4.7 Page Buffering

교체 정책 위에 얹는 보완 기법들:

- **Free-frame 풀 유지.** 페이지 폴트 시 *즉시* 풀에서 프레임을 꺼냄 — 교체 알고리즘을 기다리지 않음. Victim은 **나중에 비동기로** write-back. 폴트 낸 프로세스가 빨리 재시작.
- **수정된 페이지 리스트(일괄 write-back).** Dirty 페이지를 모아 페이징 디바이스가 idle 일 때 한 번에 write-back.
- **Free-frame 풀에 victim 캐시.** 프레임이 해제될 때 어떤 페이지를 담고 있었는지 기억. 그 페이지가 *프레임이 재사용되기 전에* 다시 필요하면 디스크 I/O 없이 복구. (Second-chance와 함께 UNIX 스타일 시스템에서 사용.)

## 5. 프레임 할당과 Thrashing

### 5.1 최소 프레임 수와 그 한계의 출처

각 프로세스는 첫 시도부터 thrashing 없이 *어떤* 명령어든 실행 가능하려면 적어도 **최소 프레임 수** 가 필요.

- 최소는 **아키텍처가 결정**: 단일 명령어가 참조 가능한 서로 다른 페이지의 최대 수.
- 간접 주소 지정은 한계를 늘림. 예: 1단계 간접 → 최소 **3 프레임** (명령어 페이지, 피연산자 포인터 페이지, 피연산자 페이지).

**최대** 는 가용 물리 메모리에 의해 결정.

### 5.2 균등 vs 비례 할당

$m$ 개 가용 프레임과 $n$ 개 프로세스가 있을 때 어떻게 분배할까?

- **균등 할당:** 각자 $m / n$ 프레임.
  - 예: 93 프레임, 5 프로세스 → 각 18 (여분 3).
- **비례 할당:** 크기 $s_i$ 에 비례:
  $$ a_i = \frac{s_i}{S} \cdot m, \qquad S = \sum_i s_i $$
  - 예: 62 프레임, 프로세스 A (10 KB), B (127 KB), $S = 137\ \text{KB}$:
    - A: $10/137 \times 62 \approx 4$ 프레임.
    - B: $127/137 \times 62 \approx 57$ 프레임.
  - **우선순위 기반 할당:** 크기 대신 우선순위 가중치.

### 5.3 Global vs Local 교체

프로세스가 페이지 폴트를 내고 빈 프레임이 없을 때 어디서 victim을 찾는가?

- **Global 교체:** 시스템 **전체** 프레임에서 victim 선택. 폴트 낸 프로세스가 다른 프로세스의 프레임을 가져갈 수 있음.
  - 전체적으로 **처리율(throughput) 좋음** — 현대 OS에서 일반적으로 사용.
  - **단점:** 프로세스가 자기 페이지 폴트 율을 예측 불가 — working set을 다른 프로세스가 조용히 가져갈 수 있음.
- **Local 교체:** 폴트 낸 **프로세스 자신의** 프레임에서만 victim 선택.
  - 프로세스별 폴트 율 예측 가능.
  - **단점:** 한 프로세스가 프레임 부족이면 다른 프로세스에 여분이 있어도 혼자 thrash; 가용 메모리가 활용되지 않을 수 있음.

### 5.4 페이지 회수와 Reaper

실용적 global 교체는 백그라운드에서 동작하는 **reaper** 루틴 사용:

- Free-frame list가 **최소 임계값** 이하로 떨어지면 트리거.
- LRU 근사 알고리즘으로 페이지를 훑으며 회수.
- Free 메모리가 **최대 임계값** 이상이면 중지.

Free 메모리가 결정적으로 낮아져 reaper가 따라가지 못하면 **Linux는 OOM Killer(Out-Of-Memory Killer)** 실행 — 크기·나이·"종료 가능성"을 결합한 OOM 점수가 가장 나쁜 프로세스를 종료.

### 5.5 NUMA와 프레임 할당

**NUMA (Non-Uniform Memory Access)** 머신에서는 각 CPU가 자기 *로컬* 메모리 뱅크를 갖고, 로컬 메모리 접근이 다른 CPU의 메모리 접근보다 현저히 빠름.

- 페이지 폴트 시 OS는 폴트 낸 프로세스를 실행 중인 **CPU에 가장 가까운** NUMA 노드에 프레임을 할당해야 함.
- Linux는 NUMA 노드별 free-frame list 유지.
- 스케줄러는 프로세스를 가장 최근 실행했던 CPU에 유지하려고 시도 — 캐시 재사용과 물리 메모리 지역성 모두 향상.

### 5.6 Thrashing

**Thrashing** = 프로세스가 실제 명령어 실행보다 페이지 폴트 처리에 더 많은 시간을 쓰는 상태.

**악순환:**

```
1. 프로세스가 현재 working set에 비해 프레임 부족
2. 거의 모든 명령어에서 페이지 폴트
3. I/O 폭주; 프로세스는 대부분 블록
4. CPU 사용률 하락 (모두 I/O 대기)
5. OS가 낮은 CPU 사용률 보고 더 많은 프로세스 스케줄
6. 이제 모든 프로세스가 더 적은 프레임
7. 단계 2로 — 폴트 폭발
```

**증상 그래프:** 다중 프로그래밍 정도가 오르면 CPU 사용률 처음엔 상승(좋음), 임계점 넘으면 **갑자기 붕괴**. 그 붕괴가 thrashing.

> **왜 이것이 OS의 정석적 병리인가:** thrashing은 *추상화가 깨지는* 순간이다. 가상 메모리의 핵심은 프로그램이 RAM보다 클 수 있다는 것 — 단, *active* 풋프린트가 들어맞을 때만. 그 풋프린트를 가용 프레임을 넘어 밀면 시스템은 계산 대신 페이지 셔플링에 시간을 쓴다.

**방어:** **local 교체** 선호(한 프로세스의 나쁜 행동이 다른 프로세스를 오염시키지 않음), 또는 더 좋게는 working set / PFF(아래)로 애초에 과다 할당을 막음.

### 5.7 Locality Model

프로세스는 실행 중 **locality에서 locality로 이동**:

- **Locality** 는 어떤 구간 동안 집중적으로 사용되는 페이지들의 집합.
- 함수 호출이 새 locality 정의 — 함수의 코드, 지역 변수, 건드리는 일부 글로벌.
- 함수에서 리턴하면 그 locality에서 떠남.
- Locality들은 중첩 가능(callee가 리턴된 뒤에도 caller의 데이터는 따끈).

**시사점:** 프로세스에 **현재** locality를 위한 충분한 프레임을 주면 locality 사이 짧은 전환에서만 폴트가 발생 — 정상 상태 실행에서는 거의 발생하지 않음. Locality가 안 들어맞으면 프로세스는 thrash.

### 5.8 Working-Set Model

"현재 locality"의 **실용적** 근사. Working-set window $\Delta$ (최근 메모리 참조 수)를 정의.

$$
WS_i(t) = \{\text{프로세스 } i \text{ 가 최근 } \Delta \text{ 회의 참조에서 건드린 페이지}\}
$$

- $\Delta$ 너무 작음: locality 포착 못 함 → 거짓 미스.
- $\Delta$ 너무 큼: 여러 locality 포괄 → 과다 할당.
- $\Delta = \infty$: 프로세스가 지금까지 건드린 모든 페이지.

**예, $\Delta = 10$:**

```
...2 6 1 5 7 7 7 7 5 1 | 6 2 3 4 1 2 3 4 4 4 | 3 4 3 4...
                    t1                     t2

WS(t1) = {1, 2, 5, 6, 7}   WS(t2) = {3, 4}
```

**WS로 thrashing 방지:**
- $WSS_i = |WS_i|$ — 프로세스 $i$ 가 필요로 하는 프레임 수.
- $D = \sum_i WSS_i$ — 시스템 전체 프레임 수요.
- $D > m$ (가용 프레임)이면 → **thrashing 임박**.
  - **해법:** 한 프로세스를 suspend 해 그것의 프레임 해제.
- $D \leq m$ 이면 → 남는 프레임으로 새 프로세스 적재 가능.

### 5.9 Working-Set 구현

WS 정확 추적은 비쌈; 보통 **타이머 인터럽트 + reference bit** 로 근사:

```
Δ = 10,000 references; 타이머 인터럽트 5,000 references 마다.

매 타이머 tick:
  각 페이지에 대해:
    현재 reference bit를 in-memory history bit 슬롯으로 복사
    reference bit 클리어

페이지 폴트 결정 시:
  현재 reference bit + 가장 최근 2개 history bit 확인
  셋 중 하나라도 1 → 최근 ~10,000-15,000 ref 내에 사용됨
                     → working set의 일부로 간주
  모두 0 → working set 밖
```

History bit 수가 많을수록 정확도 향상이지만 타이머 인터럽트당 부기 증가. 늘 그렇듯 근사의 적절한 다이얼 설정은 워크로드 의존 절충.

### 5.10 Page-Fault Frequency (PFF)

**PFF** 전략은 각 프로세스의 페이지 폴트 율을 모니터링해 thrashing을 **직접** 제어:

| 상황                                     | 동작                          |
|------------------------------------------|--------------------------------|
| 프로세스 PFF > 상한                        | **프레임 추가 할당**            |
| 프로세스 PFF < 하한                        | **프레임 회수**                 |
| PFF > 상한, 시스템에 free frame 없음        | **프로세스 suspend**            |

Working-set 모델과 비교:
- **구현 더 간단** — 폴트 카운트만.
- 궁극적으로 신경 쓰는 것(폴트 율)을 **직접 제어**.
- 폴트 율이 *왜* 그런지에 대한 정보는 적음.

## 6. 메모리 압축

Dirty 페이지를 디스크로 쓰는 대신 **RAM에서 압축** 하는 대안.

- 여러 페이지를 모아 컴프레서(Apple WKdm, Microsoft Xpress)로 압축 후 **압축된 형태를 한 프레임에 패킹**.
- 디스크 I/O 대비 CPU를 약간 더 씀. 현대 하드웨어에서 압축은 SSD 페이징보다 빠른 경우 多 — swap이 제한적이거나 없는 모바일에서는 사실상 유일한 선택지.

```
Free-frame list                  압축 후:
[F12][F45][F88][F03]             [Compressed page-pack F99]
                                 [F12][F45][F88] free-frame list로 반환
```

압축 페이지가 다시 참조되면 OS가 새 프레임으로 압축 해제. 페이지 폴트 + 압축 해제 비용은 들지만 디스크 I/O 없음.

**macOS, iOS, Android, Windows 10+** 사용. 압축 비율 보통 30–50% — 4 KB 페이지가 1.2–2 KB가 됨.

## 7. 커널 메모리 할당

### 7.1 커널 메모리가 특별한 이유

커널은 사용자 프로세스에 주는 동일한 페이징 기반 할당자를 못 씀. 이유 둘:

1. **많은 작은 객체.** 커널 구조체(PCB, 파일 객체, 세마포어, 네트워크 패킷)는 *페이지보다 작은* 경우가 많음. 객체당 프레임 하나 할당은 90%+ 메모리 낭비.
2. **일부 하드웨어는 물리적으로 연속한 메모리 요구** — 특히 DMA 컨트롤러. 페이지 단위 할당은 페이지는 주지만 *연속* 페이지는 보장 못 함.

실제 커널에 공존하는 두 고전 전략:

- **Buddy system** — 거시 입자, 연속, 빠른 합치기.
- **Slab allocation** — 미세 입자, 객체 종류별 캐시.

### 7.2 Buddy System

메모리를 연속 물리 영역 내에서 **2의 거듭제곱 크기 블록** 으로 관리.

- 크기 $r$ 요청은 다음 2의 거듭제곱으로 올림.
  - 예: 21 KB 요청 → 32 KB 블록.
- 해당 크기 블록이 없으면 더 큰 것을 재귀적으로 반(각 반이 다른 쪽의 "buddy")으로 쪼개 적절 크기까지 도달.

```
256 KB 세그먼트에서 21 KB 할당:

   [           256 KB             ]
   [   128 KB AL  ][   128 KB AR  ]   ← 쪼개기
   [ 64 BL ][ 64 BR ][   128 KB   ]   ← AL 쪼개기
   [32 CL][32 CR][ 64 BR ]...         ← BL 쪼개기 — 21 KB 요청에 32 KB 할당
```

**해제는 합치기 유발:** 블록이 해제되고 그 buddy도 free이면 다음 레벨 크기로 병합 — 트리 위쪽으로 재귀.

- **장점:** 합치기가 자명하고 빠름(레벨당 상수 시간, 총 $O(\log n)$).
- **단점:** **내부 단편화** — 33 KB 요청도 64 KB 블록(~50% 낭비).

### 7.3 Slab Allocation

**객체 종류별 캐시** 가 특정 커널 데이터 구조의 사전 할당, 사용 준비된 인스턴스를 유지.

- **Slab** = 하나 이상의 연속 물리 페이지.
- **Cache** = 특정 객체 타입의 인스턴스를 담는 slab들의 모음 (예: "PCB 캐시", "파일 객체 캐시", "세마포어 캐시").
- 각 캐시가 자기 slab과 자기 free 객체 리스트를 가짐.

```
Cache: "PCB"   ─── slab 1: [PCB][PCB][PCB][PCB]
                ── slab 2: [PCB][PCB][PCB][PCB]
                ── ...
```

**동작:**
- 할당 → 캐시의 첫 free 객체 가져옴.
- 해제 → 캐시의 free 리스트로 반환(파괴도 재초기화도 없음).

**Slab 상태:**
- **Full** — slab 내 모든 객체 사용 중.
- **Partial** — 혼합.
- **Empty** — 모든 객체 free.

할당자 선호: **partial → empty → 새 slab 할당**.

**장점:**
- **내부 단편화 0** — 슬롯 크기가 객체 크기와 정확히 일치.
- **빠름** — malloc/free 비용 없음; 사전 초기화된 객체 재사용.
- **캐시 친화적** — 동종 객체가 클러스터링.

Linux 변형:
- **SLAB** — 원래 Linux 구현.
- **SLOB** — 최소/리스트 기반, 메모리 제한 임베디드용.
- **SLUB** — Linux 2.6.24 이후 기본; 메타데이터 오버헤드 적고 SMP 확장성 좋음.

## 8. 기타 고려 사항

### 8.1 Prepaging

순수 demand paging은 프로세스 시작 시 폴트 폭주. **Prepaging** 은 곧 필요할 페이지를 미리 적재:

- 프로세스 suspend 시 working set 기억.
- Resume 시 첫 명령어 실행 **전** 그 페이지들 모두 적재.

**비용-편익:** $s$ 페이지 prepage 중 분율 $\alpha$ 가 실제 사용.
- 이득: $s \cdot \alpha$ 폴트 회피.
- 낭비: $s \cdot (1 - \alpha)$ 페이지가 무용도 적재.
- $\alpha$ 가 1에 가까울 때 이득.

Linux의 **`readahead()`** 는 관련 기제 — "곧 파일의 이 범위를 읽을 것이니 미리 폴트해줘"를 커널에 명시 통보.

### 8.2 프로그램 구조와 지역성

프로그래머가 제어하는 지역성은 OS 최적화를 넘어설 수 있음. 한 행이 정확히 한 페이지인 128 × 128 `int` 배열을 고려:

```c
// BAD — 행-우선 언어에서 열-우선 접근: 접근마다 다른 페이지로 점프
int data[128][128];
for (int j = 0; j < 128; j++)
    for (int i = 0; i < 128; i++)
        data[i][j] = 0;          // 128 × 128 = 16,384 페이지 폴트!
```

```c
// GOOD — 행-우선 접근이 저장 레이아웃과 일치
int data[128][128];
for (int i = 0; i < 128; i++)
    for (int j = 0; j < 128; j++)
        data[i][j] = 0;          // 128 페이지 폴트만
```

자료 구조 지역성도 중요:
- **스택** — 훌륭한 지역성(항상 top 건드림).
- **해시 테이블** — 빈약한 지역성(설계상 참조가 흩어짐).

### 8.3 I/O Interlock과 페이지 잠금

Global 교체와 동시 I/O의 미묘한 문제:

```
프로세스 A: 자기 버퍼 페이지 X로 디스크 READ 시작
프로세스 B: 페이지 폴트; OS가 페이지 X를 victim으로 선택 (global 교체!)
         OS가 페이지 X를 프로세스 B의 새 내용으로 다시 씀
DMA 완료: 원래 디스크 read 결과를 (이제 가로채진) 페이지 X에 씀
        → 손상!
```

**해법:**

- **커널 버퍼를 통한 간접 I/O** — 항상 커널 메모리로 먼저 읽고 사용자로 복사. 안전하지만 복사 추가.
- **Lock bit (pin)** — "이 프레임은 잠김, 교체 금지" 비트 세팅. 교체 알고리즘이 잠긴 프레임 건너뜀. I/O 완료 시 잠금 해제. 커널 페이지도 같은 이유로 영구 pin.

## 9. 실습 — 페이지 교체 알고리즘 시뮬레이터

**목표:** 같은 참조 문자열과 같은 프레임 수에 대해 **FIFO, LRU, OPT** 를 시뮬레이션; 페이지 폴트 수 비교.

**요구사항:**
- **입력:** 참조 문자열(정수 배열)과 프레임 수.
- **알고리즘별 출력:** 각 참조 후 프레임 상태와 폴트면 `* Fault` 표시.
- **최종 출력:** 비교 표.

**입력 예시:**
```
Reference String: 7 0 1 2 0 3 0 4 2 3 0 3 2 1 2 0 1 7 0 1
Number of Frames: 3
```

**Python 핵심 FIFO 스케치:**

```python
def fifo(ref_string, num_frames):
    frames = []
    faults = 0
    for page in ref_string:
        if page not in frames:
            faults += 1
            if len(frames) < num_frames:
                frames.append(page)
            else:
                frames.pop(0)        # 가장 오래된 추방
                frames.append(page)
        print(f"Ref: {page}  Frames: {frames}  {'* Fault' if page not in frames else ''}")
    return faults
```

**기대 출력:**

```
=== FIFO Algorithm ===
Ref: 7  Frames: [7, -, -]  * Page Fault
Ref: 0  Frames: [7, 0, -]  * Page Fault
Ref: 1  Frames: [7, 0, 1]  * Page Fault
Ref: 2  Frames: [2, 0, 1]  * Page Fault
Ref: 0  Frames: [2, 0, 1]
...
Total Page Faults: 15

=== Comparison ===
| Algorithm | Page Faults |
|-----------|-------------|
| FIFO      | 15          |
| LRU       | 12          |
| Optimal   |  9          |
```

## 요약

| 주제 | 핵심 내용 |
|------|---------|
| **가상 메모리** | 논리/물리 주소 공간 분리; 프로그램이 RAM보다 커도 됨 |
| **Demand paging** | Pager(lazy)가 참조 시에만 적재; 유효/무효 비트 + 페이지 폴트 트랩 |
| **EAT** | $(1-p)\cdot \text{mem} + p\cdot \text{pf}$; $p$ 가 극소($\lesssim 10^{-5}$) 여야 함 |
| **Copy-on-Write** | `fork()` 가 페이지 공유, 쓸 때만 복사; `vfork()` 는 복사 전혀 안 함 |
| **페이지 교체** | FIFO, OPT, LRU; dirty bit가 write-back을 절반으로 |
| **Belady 변칙** | FIFO에서 *더 많은* 프레임이 더 많은 폴트 가능; 스택 알고리즘(LRU/OPT)은 안 됨 |
| **LRU 근사** | Reference bit + clock; enhanced clock은 (R, M) 클래스 사용 |
| **프레임 할당** | 균등 vs 비례; 최소 한계는 명령어 아키텍처에서 |
| **교체 범위** | Global(처리율 좋음, 예측 어려움) vs Local(예측 가능, 메모리 활용 부족 가능) |
| **Thrashing** | 프레임 부족 → 폴트 폭발 → CPU 사용률 붕괴 |
| **Working-set / PFF** | 최근 참조 / 폴트 율 추적해 thrashing 탐지·방지 |
| **메모리 압축** | 디스크 swap 대신 RAM에서 압축 |
| **커널 메모리** | Buddy system(2의 거듭제곱, 연속, 쉬운 합치기) + slab(객체별 캐시) |
| **기타** | Prepaging, 프로그램 구조 지역성, I/O용 페이지 pin |

**핵심 정리:**
- **EAT는 폴트 율에 잔혹할 정도로 민감.** 이후 모든 기법은 $p$ 를 0 근처로 유지하기 위해 존재.
- **지역성이 엔진** — 가상 메모리가 작동하게 만드는 것. 지역성이 없으면 여기 모든 알고리즘이 "무작위 교체"로 격하.
- **OPT는 천장, LRU는 실용적 이상, clock이 실제 시스템이 쓰는 것.** 근사의 계층은 하드웨어 제약에 대한 직접적 대응.
- **Belady 변칙은 FIFO의 잘못, 가상 메모리의 잘못이 아님.** 스택 알고리즘이 구성적으로 제거.
- **Thrashing은 가상 메모리의 실패 모드** — working-set / PFF 모델은 정확히 그것을 방지하기 위해 존재.
- **커널 메모리는 별세계** — 커널이 수많은 작은 구조체를 할당하고 때로 물리적 연속을 요구하기 때문 — buddy + slab이 교과서적 답.

> "가상 메모리는 OS의 모든 다른 거짓말을 가능케 하는 거짓말이다. 페이지 교체는 그 거짓말을 믿을 만하게 유지하는 것이다."

**다음 주:** Mass-Storage Structure — 디스크 스케줄링(FCFS, SCAN, C-SCAN, LOOK), RAID, 저장 장치 관리.

## 점검 문제

1. **정의:** "논리 메모리를 물리 메모리로부터 분리한다"가 무슨 의미이며, 그것이 무엇을 사주는가?

   > **정답:** 논리 주소는 각 프로세스의 사적 가상 주소 공간에, 물리 주소는 공유 DRAM에 존재. 페이지 테이블이 전자를 후자로 매핑하지만 **모든 논리 페이지가 매 순간 매핑될 필요는 없음** — 부재 페이지는 PTE에서 invalid로 표시. 이 분리가 OS에게 (1) 페이지를 demand로 스트리밍해 RAM보다 큰 프로그램 실행, (2) 다수 프로세스 동시 상주, (3) 완전 적재 전 프로세스 시작, (4) 프로세스 간 물리 프레임 공유, (5) copy-on-write로 빠른 `fork()` — 모두 간접층 없이는 불가능.

2. **페이지 폴트 따라가기:** 프로세스가 `MOV [0x4000], EAX` 실행. 0x4000을 포함하는 페이지는 invalid. CPU에서 명령어 재시작까지 모든 단계를 따라가라.

   > **정답:** (1) CPU가 유효 주소 0x4000을 만들어 MMU에 제시. (2) MMU가 페이지 테이블을 워크해 PTE의 유효 비트 = 0 발견, **페이지 폴트 트랩** 발생. (3) 커널 트랩 핸들러가 폴트 주소와 PTE 검사. (4) "불법 주소"(SIGSEGV 전달) vs "합법이지만 디스크에" 구분. 여기서는 후자. (5) 커널이 free-frame list에서 빈 프레임 가져옴(list 비면 reaper 실행). (6) 디스크 I/O로 페이지를 그 프레임으로 읽음; 프로세스는 I/O 대기로 블록. (7) I/O 완료 시 PTE에 (프레임 #, valid=1, 권한) 기록. (8) 트랩에서 복귀, 폴트 낸 `MOV` 명령어를 **재시작**. (9) MMU가 0x4000을 다시 변환 — 이번엔 valid — 저장 성공.

3. **EAT 영향:** mem = 100 ns, pf = 5 ms 일 때 저하를 5% 이내로 유지할 페이지 폴트 율은?

   > **정답:** EAT = $(1-p)\cdot 100 + p \cdot 5{,}000{,}000$ = $100 + 4{,}999{,}900 p$. 5% 이내는 EAT ≤ 105, 즉 $5 \geq 4{,}999{,}900 p$, $p \leq 10^{-6}$ — **백만 메모리 접근당 약 한 폴트**. 회전 디스크 폴트는 너무 비싸 폴트 율이 ppm 범위여야 함; SSD는 약간 완화(pf ~50 μs 시 같은 5% 목표에 ~10⁻⁵).

4. **Copy-on-write 추적:** 부모가 페이지 P1(코드, 읽기 전용), P2(데이터, 쓰기 가능) 보유. 부모가 `fork()` 호출 후 자식이 P2에 씀. 각 단계의 프레임/PTE 상태를 기술하라.

   > **정답:** **Fork 전:** 부모 PTE가 F1(P1, RO), F2(P2, RW) 가리킴. **fork():** 커널이 부모 PTE를 자식 페이지 테이블에 복사. 두 프로세스 모두 F1(여전히 RO — 코드, COW 필요 없음)과 F2(논리적으론 쓰기 가능이지만 **양쪽 PTE에서 RO로 표시**) 가리킴. **자식이 P2에 씀:** MMU가 RO PTE를 보고 **protection fault** 발생. 커널이 새 프레임 F2' 할당; F2 → F2' 복사; 자식의 P2 PTE를 F2' 가리키고 RW로 갱신. 부모의 P2 PTE도 RW로 되돌림(다음 쓰기에서 폴트 안 나도록). **결과:** F1은 여전히 공유(코드 프레임 하나); F2와 F2'가 쓰기 가능 데이터의 독립 사본 — 단, *자식이 썼기 때문* 만. 자식이 `exec()` 했다면 F2가 절대 복제되지 않았음.

5. **FIFO vs LRU vs OPT:** 참조 문자열 `1 2 3 4 1 2 5 1 2 3 4 5`, 4 프레임에서 셋을 실행. 폴트 수 표로 만들어라.

   > **정답:** **FIFO (4 프레임)** — §3.6 추적대로 **10 폴트**. **LRU (4 프레임)**: 1,2,3,4 (4F, fill). 1,2 hit; 5 fault (LRU=3, 3 추방) — {1,2,4,5}. 1,2 hit; 3 fault (LRU=4, 4 추방) — {1,2,3,5}. 4 fault (LRU=5, 5 추방) — {1,2,3,4}. 5 fault (LRU=1, 1 추방) — {2,3,4,5}. **8 폴트**. **OPT (4 프레임)**: 1,2,3,4 fault (4). 1,2 hit. 5 fault — 미래 거리 따라 victim 결정 — {1,2,3,5} (4 추방). 1,2 hit. 3 hit. 4 fault — 미래 사용 없는 페이지 우선 추방, 적절한 선택 후 {2,3,4,5}. 5 hit. **총 OPT ≈ 6 폴트**. 비교: FIFO 10 > LRU 8 > OPT 6 — 그리고 여기서 FIFO는 Belady 변칙(3프레임에서 9, 4프레임에서 10) — LRU/OPT는 단조.

6. **Belady 변칙:** FIFO는 겪는데 LRU는 왜 안 겪는가? 스택 알고리즘 성질로 정당화하라.

   > **정답:** 교체 알고리즘이 **스택 알고리즘** 이라 함은 모든 참조 문자열의 모든 접두에 대해 $n$ 프레임의 상주 집합이 $n+1$ 프레임의 상주 집합의 부분집합임을 의미. **LRU** 는 자명히 만족 — $n$ 프레임에서 상주 집합은 "가장 최근 참조된 $n$ 개 페이지", 이는 "가장 최근 참조된 $n+1$ 개"의 부분집합. 프레임 추가는 상주 집합에 추가만 할 뿐 어떤 다른 페이지가 유지되는지 절대 바꾸지 않음; 따라서 폴트 수는 유지 또는 감소만. **FIFO** 는 스택 알고리즘 *아님*: $n$ 프레임 상주 집합이 삽입 순서에 의존하고, $n+1$ 프레임에서는 순서가 바뀌어 $n$ 프레임에서 추방된 페이지가 $n+1$ 프레임 상주 집합에 *없을 수도* 있음. 다른 추방 패턴 → Belady 변칙 가능.

7. **Reference bit + clock:** 원형 큐에 5 페이지, reference bit (1,1,0,1,0). 손이 페이지 1 가리킴. 추방 필요. Clock 포인터를 따라가고 어떤 페이지가 추방되는가?

   > **정답:** 페이지 1에서 시작, bit=1 → 0으로 클리어, 진행. 페이지 2, bit=1 → 0으로 클리어, 진행. 페이지 3, bit=0 → **VICTIM**: 페이지 3 추방. (페이지 1, 2는 "두 번째 기회" 받고 살아남았고 비트가 이제 0 — 추가 참조 없으면 다음 victim. 페이지 4, 5는 이번 패스에 안 건드림.) 삽입 직전 비트 최종 상태: (0, 0, ?, 1, 0); 페이지 3이 새 페이지로 교체됨(보통 bit=0).

8. **Enhanced second-chance:** 프레임이 (R=0, M=1). 이 페이지에 대해 무엇을 말하는가, 그리고 (R=0, M=0)과 비교한 추방 비용은?

   > **정답:** R=0은 이번 사이클에 페이지가 참조되지 않았음 ("최근 사용 안 됨"); M=1은 적재 후 쓰임("dirty"). 따라서 **오래되었지만 수정됨**. (R=0, M=0) — 역시 오래됐지만 **clean** — 와 비교하면 (0, 1) 추방이 더 비쌈: 프레임 재사용 전에 dirty 내용을 **swap/디스크로 write-back** 해야 함. Clean 페이지는 그냥 폐기 가능(디스크에 같은 바이트가 이미 있음). Enhanced-second-chance는 write-back을 피하려고 정확히 (0, 0)을 먼저 추방하고 clean stale 페이지가 없을 때만 (0, 1) 추방.

9. **Working-set:** $\Delta = 8$, 참조 문자열 `1 2 1 3 4 5 1 2 3 4 5 6`. 마지막 참조 후 working set은?

   > **정답:** 마지막 8 참조는 위치 5–12, `4, 5, 1, 2, 3, 4, 5, 6`. 따라서 $WS = \{1, 2, 3, 4, 5, 6\}$ — 마지막 8 참조의 서로 다른 6 페이지. 현재 locality에서 thrash 안 하려면 적어도 6 프레임 필요.

10. **Thrashing 진단:** 서버 CPU 사용률이 OS가 프로세스 하나 더 admit 한 뒤 80% 에서 15% 로 급락. 페이지 폴트 율 폭증. 어떤 일이 벌어지고 있으며, (a) global 교체, (b) local 교체, (c) PFF 하에서 해법은?

    > **정답:** 전형적 thrashing 발생: 총 working-set 수요 $D = \sum WSS_i$ 가 물리 프레임 $m$ 을 초과해 모든 프로세스가 페이지 부족, 폴트 I/O가 지배하고 CPU는 디스크 대기에 idle. **(a) Global 교체:** OS가 *suspend* 한 프로세스(보통 가장 최근 admit, 또는 최대 폴트 율) 골라 $D \leq m$ 까지; suspend로 그 프로세스 프레임이 다른 것들로 풀림. **(b) Local 교체:** local 자체는 한 나쁜 프로세스가 다른 프로세스를 오염시키는 것을 막음 — 하지만 admission control 실패. 해법: $\sum WSS_i + WSS_\text{new} > m$ 이면 새 프로세스 admit 안 함. **(c) PFF:** 한 개 이상 프로세스의 폴트 율이 상한 초과; 알고리즘이 더 많은 프레임을 주려 하지만 실패(free frame 없음), 그래서 프로세스 suspend. 같은 결과 — *신호* 가 working-set 크기에서 유추되는 대신 직접 측정될 뿐.

11. **Buddy vs slab:** 커널이 *둘 다* 필요한 이유? 각각 무엇에 좋고 무엇에 나쁜가?

    > **정답:** **Buddy** 는 **거시 입자 연속 물리 메모리** 처리 — 페이지와 다중 페이지 블록. 해제된 buddy를 $O(\log n)$ 에 합쳐 큰 블록 단편화가 깔끔히 회복. 하지만 페이지 이하 객체에 **내부 단편화** 큼(17 바이트 할당이 32 바이트 블록, 더 나쁘게는 페이지 최소를 차지), 수백만 개의 작은 커널 구조체(PCB, 파일 객체, inode)에 받아들일 수 없음. **Slab** 은 buddy *위에* 앉아 보완: slab은 (buddy에서 받은) 하나 이상의 연속 페이지를 고정 크기 객체 슬롯으로 분할. 각 slab 캐시가 한 타입 전용이므로 슬롯 크기가 객체 크기와 정확히 일치 — 해당 객체에 대해 **내부 단편화 0**, 할당/해제는 O(1) free-list pop/push. Slab은 가변 크기 큰 할당에 안 좋음(크기별 별도 캐시 필요), 이는 buddy가 자연스럽게 처리. 따라서 커널은 raw 페이지-프레임 할당과 큰 연속 블록(예: DMA 버퍼)에 **buddy**, 고빈도 타입 객체에 **slab**.

12. **프로그램 구조:** 4 KB 페이지 시스템, `double M[1024][1024]` (각 행 = 8 KB = 2 페이지). `for i, for j: M[i][j] = 0` vs `for j, for i: M[i][j] = 0` 의 페이지 폴트 동작 비교.

    > **정답:** **행-우선 접근 (`for i, for j`):** M[i][0], M[i][1], …, M[i][1023] 순서로 씀. 각 행이 2 페이지; 4 KB(512 double) 쓰면 다음 페이지로 넘어감. 전체 행렬: $1024 \text{ 행} \times 2 \text{ 페이지} = 2048$ 페이지 — 캐싱 효과 무시 시 **2048 페이지 폴트**. **열-우선 접근 (`for j, for i`):** M[0][j], M[1][j], …, M[1023][j] 씀 — 각 접근이 다른 행 → 다른 페이지(또는 페이지들). $j$ 의 한 외부 반복마다 2048 페이지 건드림(행당 하나, $j$가 페이지 경계 가로지르면 둘). 2048+ 프레임 없으면 모든 접근이 폴트: **최대 1024 × 1024 = ~10⁶ 폴트**. 속도 향상 인자: 열-우선이 **~500배 많은 폴트**. 정확히 이것이 C에서 행-우선 접근이 큰 배열에 비-타협적인 이유 — OS는 행 vs 열 불일치에서 회복 불가.
