## 1. 암묵적 스레딩

### 1.1 암묵적 스레딩 개요

**암묵적 스레딩(Implicit Threading)** 은 스레드의 생성과 관리를 **컴파일러와 런타임 라이브러리**에 위임하는 전략이다.

> 명시적 스레딩(W04)이 셰프가 주방의 모든 작업자를 직접 관리하는 것 — 업무 배정, 타이밍 조절, 충돌 해결 — 이라면, 암묵적 스레딩은 레스토랑 매니저를 고용해서 인력 관리와 스케줄링은 모두 맡기고 셰프는 어떤 요리를 만들지만 결정하는 것과 같다.

**왜 필요한가?**
- 수백에서 수천 개의 스레드를 사용하는 애플리케이션이 등장했다
- 멀티스레드 프로그래밍의 **정확성(correctness)** 을 보장하기가 극히 어렵다
- 프로그래머가 직접 스레드를 관리하면 복잡도가 폭발적으로 증가한다

**핵심 아이디어**: 개발자는 **병렬화 가능한 태스크만 식별** → 라이브러리가 스레드 매핑을 처리

**다섯 가지 주요 기법**:
1. 스레드 풀(Thread Pools)
2. Fork-Join 병렬처리
3. OpenMP
4. Grand Central Dispatch (GCD)
5. Intel Thread Building Blocks (TBB)

> **핵심:** 명시적 스레딩에서 암묵적 스레딩으로의 전환은 시스템 프로그래밍의 더 넓은 흐름을 반영한다: **저수준 세부사항을 추상화**하여 개발자가 *어떻게* 스레드를 관리할지가 아니라 *무엇을* 병렬화할지에 집중할 수 있게 하는 것이다. 이는 가비지 컬렉션 언어가 개발자를 수동 메모리 관리로부터 해방시킨 것과 유사하다.

### 1.2 태스크 vs 스레드

**태스크 기반 접근법**:
- 태스크는 일반적으로 **함수**로 작성된다
- 런타임 라이브러리가 태스크를 개별 스레드에 매핑한다
- 일반적으로 **다대다(Many-to-Many) 모델**을 활용한다
- 개발자는 스레드 수, 생성 시점, 스케줄링에 대해 신경 쓸 필요가 없다

```text
┌──────────────────────────────────────┐
│       애플리케이션 코드                │
│    (병렬 태스크 식별)                  │
├──────────────────────────────────────┤
│     컴파일러 / 런타임 라이브러리        │
│  (스레드 생성, 태스크 배정,             │
│   스케줄링 관리)                       │
├──────────────────────────────────────┤
│          운영체제                      │
│    (커널 스레드 관리)                   │
└──────────────────────────────────────┘
```

> 이 계층 구조에서 애플리케이션 프로그래머는 가장 높은 추상화 수준에서 작업한다. 컴파일러/런타임 계층은 "똑똑한 번역기" 역할을 한다 — 사용 가능한 코어 수, 현재 시스템 부하를 알고 있으며, 프로그래머가 매 애플리케이션마다 수동으로 관리하기에는 비현실적인 스레드 생성과 스케줄링에 대한 결정을 내린다.

## 2. 스레드 풀

### 2.1 스레드 풀 개념

**문제 시나리오** (웹 서버 예시):
- 매 요청마다 새 스레드를 생성하면?
  - 스레드 생성에 시간이 걸린다 → 완료 후 파괴됨 (낭비)
  - 동시 스레드 수에 **제한이 없다** → 시스템 자원 고갈 위험

**스레드 풀 해결책**:
- 시작 시 **고정된 수의 스레드를 미리 생성**하여 풀에서 대기시킨다
- 요청이 들어오면 풀에서 **유휴 스레드를 깨워** 태스크를 처리한다
- 유휴 스레드가 없으면 → 태스크는 **큐에서 대기**한다
- 태스크 완료 후 스레드는 재사용을 위해 풀로 **반환**된다

```text
요청 ──→ [ 태스크 큐 ] ──→ ┌─ Thread 1 (작업중) ─┐
요청 ──→ [ 태스크 큐 ] ──→ │  Thread 2 (유휴)    │ ──→ 실행
요청 ──→ [ 태스크 큐 ] ──→ │  Thread 3 (작업중)  │
                           │  Thread 4 (유휴)    │
                           └─ 스레드 풀 ──────────┘
```

> 스레드 풀은 공항의 택시 승강장과 같다. 승객마다 새 차를 제조하고 한 번 태운 뒤 폐차하는 대신(스레드 생성/파괴), 택시 무리가 승강장에서 대기한다. 승객이 오면 가용 택시가 태우고, 끝나면 승강장으로 돌아온다. 모든 택시가 바쁘면 승객은 줄을 선다.

### 2.2 스레드 풀의 장점

**교재의 세 가지 장점** (Silberschatz, p.178):

1. **속도(Speed)**: 기존 스레드를 재사용 → 새 스레드 생성 대기 시간 제거
2. **자원 제한(Resource bound)**: 동시 스레드 수를 **제한** → 시스템 과부하 방지
3. **실행과 생성의 분리(Separation of execution and creation)**: 태스크 실행과 생성 메커니즘을 분리
   - 지연 실행, 주기적 실행 등 다양한 전략 가능

**풀 크기 결정**:
- CPU 수, 물리 메모리, 예상 동시 요청 수를 기반으로 **경험적(heuristic)** 으로 설정
- 동적으로 조정 가능 (예: GCD는 시스템 부하에 따라 자동 조정)

> **시험 팁:** 스레드 풀의 세 가지 장점(속도, 자원 제한, 실행과 생성의 분리)은 W04의 스레드의 네 가지 장점(응답성, 자원 공유, 경제성, 확장성)과 밀접하게 관련된다. 스레드 풀은 반복적인 생성/파괴 오버헤드를 제거함으로써 경제성(Economy) 이점을 더욱 증폭시킨다.

### 2.3 스레드 풀 — Windows API

Windows는 `Thread_Create()`와 유사하게 사용할 수 있는 스레드 풀 API를 제공한다:

```c
// 별도의 스레드로 실행될 함수 정의
DWORD WINAPI PoolFunction(PVOID Param) {
    /* this function runs as a separate thread. */
}

// 스레드 풀에 작업 제출
QueueUserWorkItem(&PoolFunction, NULL, 0);
```

**`QueueUserWorkItem()`의 세 가지 매개변수**:
1. `LPTHREAD_START_ROUTINE Function` — 별도 스레드로 실행할 함수 포인터
2. `PVOID Param` — 함수에 전달할 매개변수
3. `ULONG Flags` — 스레드 풀이 스레드를 생성/관리하는 방법을 지정하는 플래그

추가 유틸리티: **주기적 실행**, **비동기 I/O 완료 시 콜백** 등

### 2.4 스레드 풀 — Java ExecutorService

`java.util.concurrent` 패키지가 제공하는 세 가지 스레드 풀 아키텍처:

| 팩토리 메서드 | 설명 |
|--------------|------|
| `newSingleThreadExecutor()` | 크기 1의 풀 생성 |
| `newFixedThreadPool(int size)` | 고정 크기 풀 생성 |
| `newCachedThreadPool()` | 무제한 풀, 유휴 스레드 재사용 |

```java
import java.util.concurrent.*;

public class ThreadPoolExample {
    public static void main(String[] args) {
        int numTasks = Integer.parseInt(args[0].trim());

        /* 스레드 풀 생성 */
        ExecutorService pool = Executors.newCachedThreadPool();

        /* 풀의 스레드를 사용하여 각 태스크 실행 */
        for (int i = 0; i < numTasks; i++)
            pool.execute(new Task());

        /* 모든 스레드가 완료된 후 풀 종료 */
        pool.shutdown();
    }
}
```

*Figure 4.15 — Java에서 스레드 풀 생성 (교재 p.180)*

> 이 예제는 W04의 Java Executor 프레임워크 섹션에서 소개되었다. 여기서 핵심적인 차이는 **관점**이다: W04에서는 Executor API를 스레드 라이브러리로서 배웠고, 지금은 스레드 풀을 라이브러리가 스레드를 대신 관리해주는 **암묵적 스레딩 기법**으로 보고 있다.

### 2.5 Java ExecutorService — 상세

**ExecutorService 인터페이스 구조**:

```text
      Executor (인터페이스)
          │
          ▼
    ExecutorService (인터페이스)
          │
    ┌─────┴─────┐
    │            │
execute()    submit()    shutdown()
(Runnable)   (Callable)  (종료 요청)
```

**execute() vs submit()**:
- `execute(Runnable)` — 반환 값 없이 태스크 실행
- `submit(Callable<T>)` — `Future<T>`를 반환, 나중에 결과 조회 가능

**shutdown() 호출 시**:
- 추가 태스크 제출을 거부한다
- 현재 실행 중인 모든 태스크가 완료될 때까지 기다린 후 종료한다

**참고 — Android AIDL**: 원격 서비스에 스레드 풀을 제공하여, 풀의 개별 스레드로 다수의 동시 요청을 처리한다

> **[모바일 개발]** Android에서 AIDL(Android Interface Definition Language)은 프로세스 간 통신을 가능하게 한다. 클라이언트가 원격 서비스 메서드를 호출하면, AIDL 프레임워크가 서비스의 스레드 풀에서 스레드를 하나 꺼내 해당 호출을 처리한다 — 개발자가 직접 스레드를 만들 필요가 없다. 이것은 프레임워크 수준의 암묵적 스레딩의 또 다른 예이다.

## 3. Fork-Join 병렬처리

### 3.1 Fork-Join 개념

**Fork-Join 모델**:
- 메인(부모) 스레드가 하나 이상의 자식 태스크를 **fork**(생성)한다
- 자식 태스크들이 병렬로 실행된다
- 모든 자식이 완료되면 **join**(합류)하여 결과를 결합한다

**암묵적 스레딩 관점에서**:
- 스레드를 직접 생성하는 대신, **병렬 태스크를 명시**한다
- 라이브러리가 스레드 수를 결정하고 태스크를 배정한다
- 동기 스레드 풀의 변형이다

*Silberschatz, Figure 4.16 — Fork-join 병렬처리*

> Fork-Join 모델은 팀 리더가 하위 업무를 위임하는 것과 같다. 리더(부모 스레드)가 팀원들에게 작업을 "fork"하고, 모든 사람이 결과를 가지고 "join"할 만남의 장소에서 기다린다. Fork-Join 라이브러리는 몇 명을 고용하고 어떻게 업무를 효율적으로 배분할지 결정하는 프로젝트 매니저이다.

### 3.2 Fork-Join과 분할 정복

Fork-Join은 **재귀적 분할 정복(divide-and-conquer)** 알고리즘에 특히 적합하다:

```text
Task(problem):
    if problem이 충분히 작으면:
        문제를 직접 해결
    else:
        subtask1 = fork(new Task(problem의 부분집합))
        subtask2 = fork(new Task(problem의 부분집합))
        result1 = join(subtask1)
        result2 = join(subtask2)
        return 결합된 결과
```

*Silberschatz, Figure 4.17 — Java에서의 Fork-join*

> **[알고리즘]** 이것은 병합 정렬(merge sort)이나 퀵 정렬(quicksort)과 동일한 패턴이다. 핵심 통찰은 분할 정복이 자연스럽게 fork-join에 매핑된다는 것이다: 각 재귀 호출이 fork된 태스크가 되고, 병합 단계가 join이 된다. 차이점은 fork-join이 실제로 **하위 태스크를 여러 스레드에서 병렬로 실행**한다는 것이며, 일반 재귀 호출은 스택에서 순차적으로 실행된다.

### 3.3 Java에서의 Fork-Join — ForkJoinPool

Java 1.7에서 도입된 Fork-Join 프레임워크:
- **ForkJoinPool**: 워커 스레드 풀
- **ForkJoinTask\<V\>**: 추상 기본 클래스

```java
ForkJoinPool pool = new ForkJoinPool();
// array는 합산할 정수를 담고 있다
int[] array = new int[SIZE];
SumTask task = new SumTask(0, SIZE - 1, array);
int sum = pool.invoke(task);
```

**클래스 계층** (Figure 4.19):

```text
      ForkJoinTask<V> (추상)
              │
      ┌───────┴────────┐
      │                 │
RecursiveTask<V>   RecursiveAction
  (V를 반환)        (반환값 없음)
```

*Silberschatz, Figure 4.19 — Fork-Join UML 클래스 다이어그램*

> `RecursiveTask<V>`는 계산된 결과를 반환해야 할 때(합계 등), `RecursiveAction`은 부수 효과만 있는 태스크(제자리 정렬 등)에 사용된다.

### 3.4 Java에서의 Fork-Join — SumTask 구현

*Figure 4.18 — Java API를 사용한 Fork-join 계산 (교재 p.183)*

```java
import java.util.concurrent.*;

public class SumTask extends RecursiveTask<Integer> {
    static final int THRESHOLD = 1000;
    private int begin;
    private int end;
    private int[] array;

    public SumTask(int begin, int end, int[] array) {
        this.begin = begin;
        this.end = end;
        this.array = array;
    }

    protected Integer compute() {
        if (end - begin < THRESHOLD) {
            int sum = 0;
            for (int i = begin; i <= end; i++)
                sum += array[i];
            return sum;
        } else {
            int mid = (begin + end) / 2;
            SumTask leftTask = new SumTask(begin, mid, array);
            SumTask rightTask = new SumTask(mid + 1, end, array);
            leftTask.fork();
            rightTask.fork();
            return rightTask.join() + leftTask.join();
        }
    }
}
```

> **코드 분석:**
> 1. 범위가 THRESHOLD(1000)보다 작으면 직접 계산한다 — 이것이 **기저 조건(base case)** 이다
> 2. 그렇지 않으면 범위를 반으로 나누고, 양쪽을 하위 태스크로 fork한다
> 3. `fork()`는 하위 태스크를 ForkJoinPool에 제출하여 병렬 실행시킨다
> 4. `join()`은 하위 태스크가 완료되어 결과를 반환할 때까지 블로킹한다
> 5. 결과는 `+`로 결합된다
>
> 이것은 재귀적 병합 정렬과 정확히 같은 모양인데, 각 재귀 호출이 풀이 다른 스레드에서 실행할 수 있는 **별도의 태스크**로 실행된다는 점이 다르다.

### 3.5 작업 훔치기 (Work Stealing)

**작업 훔치기(Work Stealing) 알고리즘**:
- 각 스레드가 자신만의 **태스크 큐**를 유지한다
- 자기 큐가 비면 → **다른 스레드의 큐에서 태스크를 훔친다**
- 소수의 워커 스레드로 수천 개의 태스크를 효율적으로 처리한다

```text
Thread 1: [Task A] [Task B] [Task C]
Thread 2: [Task D]  (비어있음)  (비어있음) ← Thread 1에서 Task C를 훔침
Thread 3: [Task E] [Task F]  (비어있음)
Thread 4:  (비어있음)  (비어있음)  (비어있음) ← Thread 3에서 Task F를 훔침
```

**THRESHOLD 결정**:
- 문제가 "충분히 작은지" 판단하는 기준
- 교재 예제에서 THRESHOLD = 1000
- 실무에서는 **타이밍 실험**을 통해 최적값을 결정해야 한다

> **핵심:** 작업 훔치기가 fork-join을 실제로 효율적으로 만드는 핵심이다. 작업 훔치기가 없으면 태스크가 불균등하게 분배될 때 일부 스레드는 놀고 다른 스레드는 과부하에 걸린다. 작업 훔치기는 **양방향 큐(deque)** 를 사용한다 — 소유 스레드는 한쪽 끝에서 push/pop하고, 도둑은 반대쪽 끝에서 훔쳐가서 경합을 최소화한다.

> **[알고리즘]** 작업 훔치기 전략은 중앙 스케줄러 없이 **부하 분산(load balancing)** 을 달성한다. 각 스레드가 독립적으로 행동한다: 먼저 자기 일을 하고, 끝나면 남의 일을 훔친다. 이 분산 접근법은 모든 스레드가 공유하는 단일 태스크 큐의 병목을 피하므로 잘 확장된다.

## 4. OpenMP

### 4.1 OpenMP 개요

**OpenMP (Open Multi-Processing)**:
- C, C++, Fortran을 위한 **컴파일러 지시어** 기반 병렬 프로그래밍 API
- **공유 메모리** 환경에서의 병렬 프로그래밍을 지원한다
- 개발자가 **병렬 영역(parallel region)** 을 지시어로 표시한다
- OpenMP 런타임 라이브러리가 해당 영역을 병렬로 실행한다

**기본 사용법**:

```c
#include <omp.h>
#include <stdio.h>

int main(int argc, char *argv[]) {
    /* 순차 코드 */

    #pragma omp parallel
    {
        printf("I am a parallel region.");
    }

    /* 순차 코드 */
    return 0;
}
```

`#pragma omp parallel` → **자동으로 처리 코어 수만큼 스레드를 생성**한다
- 듀얼코어 → 2개 스레드, 쿼드코어 → 4개 스레드

> **`#pragma`** 는 구현체 특정 지침을 제공하는 C/C++ 컴파일러 지시어이다. OpenMP 지시어(`#pragma omp ...`)는 OpenMP를 지원하지 않는 컴파일러에서는 무시되므로 코드는 여전히 유효한 C이다 — 단지 순차적으로 실행될 뿐이다. 이것이 큰 장점이다: **동일한 코드가 OpenMP 유무에 관계없이 동작**하므로, 기존 순차 코드에 병렬성을 쉽게 추가할 수 있다.

### 4.2 parallel for

**for 루프의 자동 병렬화**: `#pragma omp parallel for`

```c
#pragma omp parallel for
for (i = 0; i < N; i++) {
    c[i] = a[i] + b[i];
}
```

- 배열 `a`와 `b`의 원소를 `c`에 더하는 연산을 **자동으로 분할**한다
- OpenMP가 반복 공간을 사용 가능한 스레드 수만큼 분할한다
- 예시: N=1000, 4개 스레드 → 각 스레드가 250번 반복 처리

**실행 모델**:

```text
순차 코드    #pragma omp parallel for      순차 코드
          ─→  ┌─ Thread 0: i=0..249  ─┐ ─→
              │  Thread 1: i=250..499  │
              │  Thread 2: i=500..749  │
              └─ Thread 3: i=750..999 ─┘
                   (암묵적 배리어)
```

> **암묵적 배리어(Implicit barrier)**: `parallel for` 영역의 끝에서 모든 스레드가 동기화된다 — 모든 스레드가 자기 반복을 끝낼 때까지 어떤 스레드도 배리어를 지나 진행하지 않는다. 이는 병렬 영역 이후의 순차 코드가 완전한 결과를 볼 수 있게 보장한다.

> **[컴퓨터구조]** `parallel for` 패턴은 W04의 **데이터 병렬성(Data Parallelism)** 에 직접 매핑된다. 각 스레드가 *동일한 연산*(`c[i] = a[i] + b[i]`)을 *다른 데이터 부분집합*에 수행한다. 이는 GPU 컴퓨팅에서도 사용되는 모델이다(CUDA의 `<<<blocks, threads>>>` 구문이 유사한 분할을 달성한다).

### 4.3 추가 기능

**스레드 수 수동 설정**:

```c
#include <omp.h>
#include <stdio.h>

int main() {
    omp_set_num_threads(4);  // 스레드 수를 4로 설정

    #pragma omp parallel
    {
        printf("Thread %d of %d\n",
            omp_get_thread_num(),
            omp_get_num_threads());
    }

    return 0;
}
```

**데이터 공유 속성**:
- `shared(var)` — 모든 스레드가 변수를 공유
- `private(var)` — 각 스레드가 변수의 자체 사본을 가짐
- `reduction(+:sum)` — 지정된 연산으로 각 스레드의 결과를 결합

**컴파일**: `gcc -fopenmp program.c -o program`

> **데이터 공유 속성은 정확성에 매우 중요하다.** 이것들이 없으면 `parallel` 영역에서 모든 변수는 기본적으로 `shared`이다(`parallel for`의 루프 카운터는 기본적으로 `private`인 것이 예외). 흔한 버그는 `private`이어야 할 변수를 실수로 공유하는 것이다 — 이는 W04 실습의 `global_sum` 예제처럼 경쟁 조건을 유발한다.

### 4.4 reduction 예제

**reduction을 사용한 배열 합산 병렬화**:

```c
#include <omp.h>
#include <stdio.h>

int main() {
    int sum = 0;
    int a[1000];

    // 배열 초기화
    for (int i = 0; i < 1000; i++)
        a[i] = i + 1;

    // reduction: 각 스레드의 부분 결과를 자동으로 합산
    #pragma omp parallel for reduction(+:sum)
    for (int i = 0; i < 1000; i++) {
        sum += a[i];
    }

    printf("Sum = %d\n", sum);  // 500500
    return 0;
}
```

- 각 스레드가 `sum`의 **개인 복사본**을 유지한다
- 병렬 영역 끝에서 모든 스레드의 합이 **+ 연산으로 결합**된다
- 경쟁 조건 없이 안전한 합산이 가능하다

> W04 실습의 `global_sum` 경쟁 조건과 비교해보자. 그 실습에서 `global_sum += array[i]`는 여러 스레드가 같은 변수에 비원자적 읽기-수정-쓰기를 수행하여 데이터 손실을 일으켰다. OpenMP의 `reduction` 절은 이것을 우아하게 해결한다: 각 스레드가 자신만의 개인 복사본을 가지고, 런타임이 마지막에 안전하게 합친다. 락도 없고, 경쟁 조건도 없고, 코드 한 줄이면 된다.

## 5. Grand Central Dispatch (GCD)

### 5.1 GCD 개요

**GCD (Grand Central Dispatch)**:
- Apple이 macOS/iOS용으로 개발한 기술
- **런타임 라이브러리 + API + 언어 확장**의 조합
- OpenMP와 유사하게 스레딩 세부사항을 자동으로 관리한다

**핵심 메커니즘**: **디스패치 큐(Dispatch Queue)**
- 디스패치 큐에 놓인 태스크는 GCD가 **스레드 풀에서 스레드를 배정**하여 실행한다

**두 가지 유형의 디스패치 큐**:

| 유형 | 설명 |
|------|------|
| **직렬 큐(Serial Queue)** | FIFO 순서로 디큐, 한 번에 하나의 태스크만 실행 |
| **동시 큐(Concurrent Queue)** | FIFO 순서로 디큐, 여러 태스크를 동시에 실행 |

> **핵심 구분**: 두 큐 유형 모두 FIFO 순서로 디큐하지만, 직렬 큐는 현재 태스크가 완료될 때까지 다음 태스크를 시작하지 않고, 동시 큐는 이전 태스크가 완료되지 않았어도 새 태스크를 시작할 수 있다. 비유하자면: 직렬 = 계산대 한 줄, 동시 = 같은 대기 공간에서 여러 계산대가 고객을 받는 것이다.

### 5.2 직렬 큐 vs 동시 큐

**직렬 큐** (= 프라이빗 디스패치 큐):
- 각 프로세스는 자체 직렬 큐(**메인 큐**)를 가진다
- 개발자가 추가 직렬 큐를 생성할 수 있다
- **순차 실행이 필요한 태스크**에 유용하다

**동시 큐** (= 글로벌 디스패치 큐):
- 시스템 전체 큐
- 네 가지 **QoS (Quality of Service)** 클래스:

| QoS 클래스 | 설명 | 예시 |
|------------|------|------|
| `QOS_CLASS_USER_INTERACTIVE` | UI/이벤트 처리, 작은 태스크 | 애니메이션 |
| `QOS_CLASS_USER_INITIATED` | 사용자가 결과를 기다리는 중 | 파일/URL 열기 |
| `QOS_CLASS_UTILITY` | 장시간 실행 태스크, 즉각적 결과 불필요 | 데이터 가져오기 |
| `QOS_CLASS_BACKGROUND` | 사용자에게 보이지 않는 태스크 | 인덱싱, 백업 |

> **[모바일 개발]** QoS 클래스는 iOS/macOS 개발에서 매우 중요하다. 시스템은 이 우선순위를 사용하여 CPU 시간을 할당하고 전력 소비를 관리한다. 흔한 실수는 네트워크 요청을 `USER_INTERACTIVE`에서 실행하는 것이다 — 이는 시스템이 애니메이션과 터치 처리를 희생하고 네트워크 I/O를 과도하게 우선시하므로 UI 반응성이 저하된다. 사용자 트리거 네트워크 요청에는 `USER_INITIATED`, 프리페칭에는 `UTILITY`나 `BACKGROUND`를 사용하라.

### 5.3 블록과 클로저

**C/C++/Objective-C에서: 블록(Block)** — `^{ }` 구문으로 독립된 작업 단위를 정의

```c
// 블록 정의
^{ printf("I am a block"); }

// 직렬 큐 생성 및 작업 제출
dispatch_queue_t q =
    dispatch_queue_create("my.queue", NULL);

// 동시 글로벌 큐 사용
dispatch_queue_t gq =
    dispatch_get_global_queue(
        DISPATCH_QUEUE_PRIORITY_DEFAULT, 0);

// 비동기 작업 제출
dispatch_async(gq, ^{
    printf("Running in background\n");
});
```

**Swift에서: 클로저(Closure)** — 블록과 같은 개념, `^` 없이 사용

```text
let queue = dispatch_get_global_queue(QOS_CLASS_USER_INITIATED, 0)
dispatch_async(queue, { print("I am a closure.") })
```

> 블록과 클로저는 GCD에서의 "태스크" 단위이다. OpenMP가 `#pragma` 지시어로 병렬 영역을 표시하는 것처럼, GCD는 블록/클로저로 작업 단위를 정의한다. 핵심 차이점은 GCD 태스크가 **큐**에 제출(비동기 또는 동기)된다는 것으로, 개발자에게 실행 순서와 동시성에 대한 더 많은 제어권을 준다.

### 5.4 GCD 내부 아키텍처

**GCD의 스레드 풀 관리**:
- 내부적으로 **POSIX 스레드**로 구성된다
- GCD가 풀 크기를 **동적으로 조정**한다 (확장/축소)
- 애플리케이션 수요와 시스템 용량에 따라 자동 관리된다

**libdispatch 라이브러리**:
- GCD의 구현체
- Apple이 **Apache Commons 라이선스**로 오픈소스화
- FreeBSD 등 다른 운영체제에도 포팅됨

```text
┌──────────────────────────────────────────┐
│  애플리케이션  (블록 / 클로저)              │
├──────────────────────────────────────────┤
│  디스패치 큐 (직렬 / 동시)                 │
├──────────────────────────────────────────┤
│  libdispatch (GCD 런타임)                 │
│  - 동적 스레드 풀 관리                     │
│  - QoS 기반 스케줄링                       │
├──────────────────────────────────────────┤
│  POSIX 스레드 (pthreads)                  │
├──────────────────────────────────────────┤
│  커널 스레드                               │
└──────────────────────────────────────────┘
```

> 이 계층 아키텍처는 GCD가 애플리케이션 코드에서 커널 스레드까지 스레딩을 완전히 추상화하는 모습을 보여준다. 개발자는 최상위에서 블록/클로저를 작성하고, 4단계의 추상화가 나머지를 모두 처리한다. GCD가 궁극적으로 POSIX 스레드 위에 놓인다는 점에 주목하라 — Apple의 "마법의" 스레딩도 W04에서 배운 것과 같은 Pthreads 위에 구축되어 있다.

## 6. Intel Thread Building Blocks (TBB)

### 6.1 Intel TBB 개요

**Intel TBB (Threading Building Blocks)**:
- 병렬 애플리케이션 설계를 위한 **C++ 템플릿 라이브러리**
- **특별한 컴파일러나 언어 확장이 필요 없다** (라이브러리만으로 동작)
- 개발자가 **병렬화 가능한 태스크를 식별**하면, TBB 태스크 스케줄러가 스레드에 매핑한다

**TBB 태스크 스케줄러 특성**:
- **부하 분산(Load balancing)**: 스레드 간 작업량을 균등하게 분배
- **캐시 인지(Cache aware)**: 캐시에 데이터가 있을 가능성이 높은 태스크를 우선 처리

**제공하는 주요 기능**:
- 병렬 루프 구조 (`parallel_for`, `parallel_reduce`)
- 원자적 연산
- 상호 배제 잠금
- 동시 자료 구조 (해시 맵, 큐, 벡터)

> **OpenMP와의 비교**: OpenMP는 **컴파일러 지시어**(`#pragma`)를 사용하므로 OpenMP를 지원하는 컴파일러가 필요하다. TBB는 순수 **C++ 템플릿과 라이브러리**를 사용하므로 어떤 표준 C++ 컴파일러에서든 동작한다. 이는 TBB를 일부 환경에서 더 이식성 있게 만들지만, OpenMP가 구문적으로 더 간단하다.

### 6.2 parallel_for 예제

**순차 코드**:

```cpp
for (int i = 0; i < n; i++) {
    apply(v[i]);
}
```

**TBB `parallel_for`로 변환**:

```cpp
#include <tbb/tbb.h>
using namespace tbb;

parallel_for(size_t(0), n, [=](size_t i) {
    apply(v[i]);
});
```

**작동 방식**:
- 처음 두 매개변수: 반복 공간 (0부터 n-1까지)
- 세 번째 매개변수: **C++ 람다 함수** `[=](size_t i)` → 각 `i`에 대해 실행
- TBB가 루프 반복을 **청크(chunk)** 로 분할
- 여러 태스크를 생성하고 가용 스레드에 배정
- Java의 Fork-Join과 유사한 접근법

> 람다의 `[=]`은 모든 변수를 값으로 캡처한다. 이는 스레드 안전성에 중요하다 — 각 스레드가 캡처된 변수의 자체 복사본으로 작업한다. 공유 상태를 수정해야 한다면 `[&]`(참조 캡처)를 적절한 동기화와 함께 사용해야 한다.

### 6.3 parallel_reduce 예제

**`parallel_reduce`를 사용한 배열 합산 구현**:

```cpp
#include <tbb/tbb.h>
using namespace tbb;

// parallel_reduce 사용
int sum = parallel_reduce(
    blocked_range<int>(0, n),  // 반복 공간
    0,                          // 초기값
    [&](const blocked_range<int>& r, int local_sum) {
        for (int i = r.begin(); i < r.end(); i++)
            local_sum += data[i];
        return local_sum;
    },
    [](int x, int y) { return x + y; }  // 결합 함수
);
```

- 태스크 분할과 스케줄링은 **라이브러리가 자동으로 처리**한다
- **작업 훔치기(work-stealing)** 알고리즘으로 부하 분산을 유지한다
- 상용 및 오픈소스 버전 모두 사용 가능 (Windows, Linux, macOS)

> **OpenMP reduction과의 비교**: 둘 다 동일한 목표(병렬 배열 합산)를 달성하지만, 접근법이 다르다:
> - OpenMP: `#pragma omp parallel for reduction(+:sum)` — 지시어 한 줄
> - TBB: `parallel_reduce(range, init, body, combine)` — 람다를 포함한 명시적 함수 호출
>
> OpenMP가 더 간결하고, TBB가 더 유연하다(내장 연산자뿐 아니라 임의의 결합 연산을 정의할 수 있다).

## 7. 암묵적 스레딩 — 5가지 기법 비교

| 기법 | 언어/플랫폼 | 접근법 | 핵심 특징 |
|------|------------|--------|-----------|
| **스레드 풀** | Java, Windows, Android | 스레드를 풀에 미리 생성 | 범용, 재사용 |
| **Fork-Join** | Java 1.7+ | 분할 정복 | 작업 훔치기 |
| **OpenMP** | C/C++/Fortran | 컴파일러 지시어 | 가장 간단한 구문 |
| **GCD** | macOS/iOS (C, Swift) | 디스패치 큐 | QoS 기반 |
| **Intel TBB** | C++ | 템플릿 라이브러리 | 캐시 인지 |

**공통 특성**:
- 개발자는 **태스크(병렬 작업 단위)만 식별**한다
- 스레드 생성/관리/스케줄링은 **라이브러리/런타임이 처리**한다
- 프로그램의 정확성을 향상시키고 개발 복잡도를 줄인다

> **시험 팁:** 다섯 가지 기법과 각각의 구별되는 특징을 말할 수 있어야 한다. 도움이 되는 약어: **T-F-O-G-I** (Thread pool, Fork-join, OpenMP, GCD, Intel TBB). 다섯 가지 모두 핵심 아이디어를 공유한다는 것을 기억하자: **개발자는 태스크를 정의하고, 시스템이 스레드를 관리한다**.

## 8. 스레딩 이슈

### 8.1 스레딩 이슈 개요

멀티스레드 프로그래밍에서 고려해야 할 다섯 가지 핵심 이슈:

1. **fork()와 exec()의 의미론** — 멀티스레드 프로세스의 복제
2. **시그널 처리** — 멀티스레드 환경에서의 시그널 전달
3. **스레드 취소** — 실행 중인 스레드의 취소
4. **스레드-로컬 저장소 (TLS)** — 스레드별 고유 데이터
5. **스케줄러 활성화** — 커널과 스레드 라이브러리 간의 통신

> 이러한 이슈들은 멀티스레딩의 근본적인 긴장에서 비롯된다: 스레드는 프로세스의 자원을 공유하고(이것이 스레드를 효율적으로 만든다), 때로는 독립적인 동작이 필요하다(이것이 복잡성을 만든다). 아래의 각 이슈는 공유-독립 경계가 문제를 일으키는 구체적인 사례이다.

### 8.2 fork()와 exec()의 의미론

**문제**: 멀티스레드 프로세스에서 한 스레드가 `fork()`를 호출하면 어떻게 되는가?

**두 가지 가능한 동작**:
1. **모든 스레드를 복제** — 새 프로세스가 부모의 모든 스레드를 갖는다
2. **호출한 스레드만 복제** — 새 프로세스는 단일 스레드이다

```text
  부모 프로세스                     자식 프로세스 (옵션 1)
  ┌─────────────┐                ┌─────────────┐
  │ Thread 1    │   fork()       │ Thread 1'   │
  │ Thread 2 ───┼──────────→    │ Thread 2'   │  모든 스레드 복제
  │ Thread 3    │                │ Thread 3'   │
  └─────────────┘                └─────────────┘

  부모 프로세스                     자식 프로세스 (옵션 2)
  ┌─────────────┐                ┌─────────────┐
  │ Thread 1    │   fork()       │             │
  │ Thread 2 ───┼──────────→    │ Thread 2'   │  호출 스레드만 복제
  │ Thread 3    │                │             │
  └─────────────┘                └─────────────┘
```

**선택 기준:**

| 시나리오 | fork() 동작 | 이유 |
|---------|-------------|------|
| fork() 직후 exec() 호출 | 호출 스레드만 복제 | exec()가 전체 프로세스를 교체하므로 다른 스레드 복제는 낭비 |
| fork() 후 exec() 없이 실행 | 모든 스레드 복제 | 자식 프로세스도 모든 스레드가 필요 |

**exec() 호출 시**:
- 전체 프로세스를 (**모든 스레드 포함**) 새 프로그램으로 **교체**한다
- 따라서 fork() + exec() 패턴에서는 호출 스레드만 복제하면 충분하다

**일부 UNIX 시스템**: 두 가지 버전의 fork()를 모두 제공한다

> **핵심:** 실제로 `fork()` 직후 `exec()`를 호출하는 것이 가장 흔한 패턴이다(W02-W03). 이 경우 모든 스레드를 복제하는 것은 `exec()`가 어차피 전체 주소 공간을 버리기 때문에 낭비이다. POSIX의 `fork()`는 호출 스레드만 복제하고, 일부 시스템은 모든 스레드를 복제하는 `forkall()`을 제공한다.

> **주의:** fork()가 호출 스레드만 복제할 때, 다른 스레드가 보유한 뮤텍스는 자식 프로세스에서 잠긴 채 남아있지만, 그것을 해제할 스레드가 존재하지 않는다. 이는 자식 프로세스에서 **데드락**으로 이어질 수 있다 — fork()+exec()가 선호되는 또 다른 이유이다.

### 8.3 시그널 처리 — 개념과 패턴

**시그널(Signal)**: UNIX 시스템에서 특정 이벤트가 발생했음을 프로세스에 알리는 메커니즘

**모든 시그널이 따르는 3단계 패턴**:
1. 특정 이벤트에 의해 시그널이 **생성**된다
2. 시그널이 프로세스에 **전달**된다
3. 전달된 시그널이 **처리**된다

**시그널 핸들러의 유형**:
1. **기본 시그널 핸들러(Default signal handler)**: 커널에 의해 실행되는 기본 핸들러
2. **사용자 정의 시그널 핸들러(User-defined signal handler)**: 프로그래머가 정의한 커스텀 핸들러

| 범주 | 동기 시그널 | 비동기 시그널 |
|------|-----------|-------------|
| **원인** | 프로세스 자신이 유발 | 외부에서 생성 |
| **전달 대상** | 해당 행동을 수행한 프로세스 | 다른 프로세스에 전달 |
| **예시** | 잘못된 메모리 접근, 0으로 나누기 | `Ctrl+C`, 타이머 만료 |

> **동기 시그널**은 스레드 자신의 행동의 결과이다 — 0으로 나누거나 잘못된 메모리에 접근하는 것처럼. 스레드가 시그널을 "유발"했으므로 전달이 간단하다. **비동기 시그널**은 프로세스 외부에서 온다(사용자가 Ctrl+C를 누르거나, 타이머가 만료되거나, 다른 프로세스가 시그널을 보내거나). 멀티스레딩의 복잡성이 발생하는 곳이 바로 여기이다: *어떤* 스레드가 받아야 하는가?

### 8.4 시그널 처리 — 멀티스레드 환경

**단일 스레드**: 시그널은 항상 해당 프로세스에 전달된다 (간단)

**멀티스레드**: 시그널을 어디에 전달해야 하는가?

**네 가지 옵션** (교재 p.189):
1. **시그널을 유발한 스레드**에 전달 (동기 시그널에 적합)
2. **프로세스의 모든 스레드**에 전달 (예: `Ctrl+C`)
3. **특정 스레드에만** 전달 (시그널을 차단하지 않은 스레드)
4. **전담 시그널 처리 스레드**를 지정

```c
// UNIX: 특정 프로세스에 시그널 전달
kill(pid_t pid, int signal);

// Pthreads: 특정 스레드에 시그널 전달
pthread_kill(pthread_t tid, int signal);
```

**POSIX**: 각 스레드가 어떤 시그널을 수락/차단할지 지정할 수 있다
- 비동기 시그널 → 시그널을 차단하지 않은 **첫 번째 스레드**에 전달
- **Windows**: 시그널 대신 **APC (Asynchronous Procedure Call)** 를 사용 — 특정 스레드에 직접 전달

> **시험 팁:** 네 가지 시그널 전달 옵션은 자주 출제된다. (1) 유발자, (2) 모두, (3) 일부(차단하지 않은), (4) 전담 스레드 하나로 기억하자. 실무에서는 옵션 4(전담 시그널 처리 스레드)가 서버 애플리케이션에서 흔한 패턴이다 — 다른 모든 스레드는 시그널을 차단하고, 지정된 한 스레드가 제어된 루프에서 시그널을 처리한다.

### 8.5 스레드 취소 — 개념

**스레드 취소(Thread Cancellation)**: 스레드가 **완료되기 전에** 종료시키는 것

**사용 예시**:
- 여러 스레드가 DB를 검색 → 하나가 결과를 찾으면 **나머지 스레드를 취소**
- 웹 브라우저에서 중지 버튼 누르기 → 이미지 로딩 스레드를 **모두 취소**

취소 대상 스레드를 **대상 스레드(target thread)** 라 한다.

**두 가지 취소 접근법**:

| 접근법 | 설명 | 안전성 |
|--------|------|--------|
| **비동기 취소(Asynchronous cancellation)** | 즉시 종료 | 위험 |
| **지연 취소(Deferred cancellation)** | 취소 지점에서 종료 | 안전 |

### 8.6 비동기 취소 vs 지연 취소

**비동기 취소의 문제점**:
- 자원이 할당된 상태에서 즉시 종료 → **자원 누수(resource leak)**
- 공유 데이터 업데이트 중 취소 → **데이터 불일치**
- OS가 시스템 자원은 회수하지만 **모든 자원을 회수할 수는 없다**
- Pthreads 문서에서 **사용을 권장하지 않는다**

**지연 취소**:
- 스레드에 "취소 요청"만 보낸다
- 대상 스레드가 **취소 지점(cancellation point)** 에 도달할 때만 실제 종료
- 안전한 자원 정리가 가능하다

```text
비동기:  취소 ──→ [즉시 종료!] (자원 정리 없음)

지연:    취소 ──→ [계속 실행] ──→ [취소 지점] ──→ [정리 후 종료]
```

> 비동기 취소는 컴퓨터의 전원 플러그를 뽑는 것과 같다 — 빠르지만 데이터 손상과 자원 누수의 위험이 있다. 지연 취소는 "종료" 버튼을 클릭하는 것과 같다 — 시스템이 중요한 작업을 마치고, 상태를 저장하고, 자원을 해제한 뒤 깔끔하게 종료한다.

### 8.7 스레드 취소 — Pthreads API

**Pthreads의 세 가지 취소 모드**:

| 모드 | 상태 | 유형 |
|------|------|------|
| **Off** | 비활성 | -- |
| **Deferred** (기본값) | 활성 | 지연 |
| **Asynchronous** | 활성 | 비동기 |

```c
pthread_t tid;

/* 스레드 생성 */
pthread_create(&tid, 0, worker, NULL);

/* 스레드 취소 */
pthread_cancel(tid);

/* 스레드 종료 대기 */
pthread_join(tid, NULL);
```

- `pthread_cancel(tid)` — 취소 **요청**만 보낸다 (즉시 종료하지 않는다!)
- 실제 취소는 대상 스레드의 취소 모드에 의해 결정된다
- 취소가 **비활성**이면 → 요청은 **보류(pending)** 상태로 남는다

> **중요한 구분**: `pthread_cancel()`은 **요청**이지 명령이 아니다. 대상 스레드가 언제(또는 할지) 이행할지를 결정한다. 이는 일반적으로 프로세스를 즉시 종료시키는 `kill()`과 근본적으로 다르다. 지연 모델은 대상 스레드에게 자신의 종료에 대한 제어권을 준다.

### 8.8 취소 지점과 정리 핸들러

**취소 지점(Cancellation Point)**:
- 지연 취소에서, 스레드가 "취소될 수 있는" 지점
- POSIX/표준 C 라이브러리의 대부분의 **블로킹 시스템 호출**이 취소 지점이다
  - 예시: `read()`, `write()`, `sleep()`, `pthread_join()` 등
- 전체 목록은 `man pthreads`로 확인 가능

```c
while (1) {
    /* 잠시 작업 수행 */
    ...

    /* 취소 요청이 있는지 확인 */
    pthread_testcancel();
}
```

**`pthread_testcancel()`**:
- 명시적으로 취소 지점을 생성한다
- 보류 중인 취소 요청이 있으면 → 스레드 종료
- 없으면 → 정상 반환, 실행 계속

**정리 핸들러(Cleanup Handler)**: 스레드가 취소될 때 자원(메모리, 파일, 락 등)을 안전하게 해제하는 함수

> **실무 패턴**: 블로킹 시스템 호출을 하지 않는 장시간 실행 워커 스레드는 주기적으로 `pthread_testcancel()`을 호출해야 한다. 이것이 없으면 취소 지점이 없어서 지연 취소 요청이 영원히 보류 상태로 남게 된다. 스레드가 취소 요청을 "무시"하는 것처럼 보일 것이다.

### 8.9 스레드 취소 — Java

Java는 **Pthreads의 지연 취소와 유사한** 정책을 사용한다:

```java
Thread worker;
...

/* 스레드의 인터럽트 상태 설정 */
worker.interrupt();
```

`interrupt()`: 대상 스레드의 **인터럽트 상태를 true로** 설정한다

```java
while (!Thread.currentThread().isInterrupted()) {
    // ... 작업 수행 ...
}
```

**`isInterrupted()`**: 스레드의 인터럽트 상태를 boolean으로 반환한다

**Pthreads와의 비교**:

| | Pthreads | Java |
|--|---------|------|
| 취소 요청 | `pthread_cancel(tid)` | `worker.interrupt()` |
| 상태 확인 | `pthread_testcancel()` | `isInterrupted()` |
| 자동 취소 지점 | 블로킹 시스템 호출 | 블로킹 메서드 (sleep, wait, join) |

**참고**: Linux에서 Pthreads 취소는 내부적으로 **시그널**을 사용하여 구현된다

> Java의 접근법은 Pthreads보다 더 안전하다: `interrupt()`는 boolean 플래그만 설정한다. 스레드가 명시적으로 `isInterrupted()`를 확인하거나, `InterruptedException`을 던지는 블로킹 메서드에 있어야 한다. 이로 인해 안전하지 않은 지점에서 스레드를 실수로 종료시키는 것이 불가능하다.

### 8.10 스레드-로컬 저장소 (TLS)

**TLS (Thread-Local Storage)**: 각 스레드가 데이터의 자신만의 **복사본을 유지**하는 메커니즘

**왜 필요한가?**
- 스레드는 근본적으로 프로세스 데이터를 **공유**한다
- 그러나 때로는 스레드가 **자신만의 고유한 데이터**가 필요하다
- 예시: 트랜잭션 처리 시스템에서 각 스레드에 **고유한 트랜잭션 ID**를 배정

**TLS vs 지역 변수**:

| | 지역 변수 | TLS |
|--|----------|-----|
| 수명 | 함수 호출 중에만 | **스레드의 전체 수명 동안** |
| 가시성 | 함수 범위 내 | 모든 함수에서 접근 가능 |
| 유사성 | 스택에 할당 | **정적 데이터**와 유사 (단, 스레드별로 고유) |

> TLS는 각 스레드가 사무실에 자기만의 "사물함"을 갖는 것과 같다. 모든 사람이 같은 개방형 업무 공간(공유 데이터)을 사용하지만, 각자 근무 기간(스레드 수명) 동안 유지되는 개인 사물함(TLS)을 가지고 있으며 본인만 접근할 수 있다. 지역 변수는 회의 중에 사용하는 포스트잇에 가깝다 — 회의가 끝나면 버려진다.

### 8.11 TLS — 언어별 구현

**언어/컴파일러별 TLS 지원**:

```c
// GCC: __thread 키워드
static __thread int threadID;

// C11 표준: _Thread_local 키워드
_Thread_local int var = 0;
```

```java
// Java: ThreadLocal<T> 클래스
ThreadLocal<Integer> value =
    ThreadLocal.withInitial(() -> 0);

value.set(42);           // 현재 스레드의 값 설정
int v = value.get();     // 현재 스레드의 값 조회
```

```csharp
// C#: [ThreadStatic] 어트리뷰트
[ThreadStatic]
static int threadLocalVar;
```

**특히 스레드 풀 환경에서 유용**: 스레드 생성을 개발자가 직접 제어하지 않더라도 각 워커 스레드가 고유한 상태를 유지할 수 있다
- Pthreads: `pthread_key_t` 타입을 사용하여 TLS 키를 생성/조회/삭제한다

> **[프로그래밍 언어]** TLS는 플랫폼에 따라 내부적으로 다르게 구현된다. Linux/x86-64에서 TLS 데이터는 `fs` 세그먼트 레지스터를 통해 접근되며, 이 레지스터는 각 스레드 자신의 TLS 블록을 가리킨다. 이는 TLS 접근이 일반 전역 변수 접근과 거의 같은 속도라는 것을 의미한다 — 락이나 해시 테이블 조회가 관여하지 않는다.

### 8.12 스케줄러 활성화 — LWP와 업콜

**문제**: 다대다(Many-to-Many) / 두 수준(Two-Level) 모델에서 커널과 스레드 라이브러리 간의 **통신**이 필요하다

**LWP (Lightweight Process)**:
- 사용자 스레드와 커널 스레드 사이의 **중간 자료 구조**
- 사용자 수준 스레드 라이브러리에게 **가상 프로세서**로 보인다
- 각 LWP는 **하나의 커널 스레드에 부착**된다

*Silberschatz, Figure 4.20 — 경량 프로세스 (LWP)*

**LWP 수**: CPU 바운드 애플리케이션은 1개만 필요하고, I/O 집약적 애플리케이션은 동시 블로킹 I/O 연산 수만큼 필요하다

> LWP 개념은 W04에서 두 수준 모델의 일부로 소개되었다. 여기서는 운용 맥락에서 본다: LWP는 커널과 사용자 수준 스레드 라이브러리 간의 "통신 채널"이다. 커널이 스레드 라이브러리에 이벤트(스레드 블로킹 등)를 알려야 할 때 LWP를 통해 수행한다.

### 8.13 스케줄러 활성화 — 업콜 메커니즘

**업콜(Upcall)**: 커널에서 사용자 수준 스레드 라이브러리로의 **이벤트 알림**

**작업 흐름** (스레드가 블로킹될 때):

```text
1. Thread A가 I/O 요청에서 블로킹
   ┌──────────────────────────────────────┐
   │ 커널: "Thread A가 블로킹됨"           │ ──→ 업콜
   └──────────────────────────────────────┘

2. 커널이 새 가상 프로세서(LWP)를 할당

3. 스레드 라이브러리가 업콜 핸들러를 실행:
   - 블로킹 스레드의 상태를 저장
   - 새 LWP에 다른 적격 스레드를 스케줄링

4. 블로킹 이벤트가 완료될 때:
   ┌──────────────────────────────────────┐
   │ 커널: "Thread A가 준비됨"             │ ──→ 업콜
   └──────────────────────────────────────┘

5. 스레드 라이브러리가 Thread A를 실행 가능 상태로 전환
```

> **업콜 vs 시스템 호출**: 시스템 호출은 **하향** 요청(사용자 → 커널)이다: "I/O를 수행해주세요." 업콜은 **상향** 알림(커널 → 사용자 스레드 라이브러리)이다: "당신의 스레드가 방금 블로킹되었습니다." 이 양방향 통신이 다대다 모델을 작동하게 하는 것이다 — 업콜이 없으면 스레드 라이브러리는 사용자 스레드가 커널에서 블로킹되었는지 알 수 없고, 그 자리에 다른 스레드를 스케줄링할 수 없다.

> **참고:** 스케줄러 활성화는 주로 다대다 모델과 관련된다. 대부분의 현대 시스템은 일대일 모델을 사용하므로(Linux, Windows), 스케줄러 활성화는 실무에서 덜 자주 접하게 된다. 그러나 하이브리드 스레딩 모델의 작동 방식을 이해하는 데 이 개념은 여전히 중요하다.

## 9. 운영체제 사례

### 9.1 Windows 스레드 — 개요

**Windows 스레드 구현**:
- **일대일(One-to-One) 모델** 사용 (각 사용자 수준 스레드 → 커널 스레드)
- Windows API의 `CreateThread()`로 스레드를 생성한다

**스레드의 일반적 구성요소** (스레드의 **컨텍스트**):
- 스레드 ID (고유 식별자), 레지스터 집합 (프로세서 상태), 프로그램 카운터
- **사용자 스택** (사용자 모드용) + **커널 스택** (커널 모드용)
- 프라이빗 저장 영역 (런타임 라이브러리, DLL이 사용)

**세 가지 핵심 자료 구조** (Figure 4.21):

| 구조 | 위치 | 주요 내용 |
|------|------|-----------|
| **ETHREAD** | 커널 공간 | 스레드 시작 주소, 부모 프로세스 포인터, KTHREAD 포인터 |
| **KTHREAD** | 커널 공간 | 스케줄링/동기화 정보, 커널 스택, TEB 포인터 |
| **TEB** | 사용자 공간 | 스레드 ID, 사용자 모드 스택, TLS 배열 |

> **ETHREAD** = Executive Thread, **KTHREAD** = Kernel Thread, **TEB** = Thread Environment Block. 이름은 Windows의 계층화된 커널 아키텍처를 따른다: Executive 계층(ETHREAD)은 상위 수준 스레드 관리를 담당하고, Kernel 계층(KTHREAD)은 스케줄링을 담당하며, TEB는 사용자 공간에서 접근 가능한 정보를 제공한다.

### 9.2 Windows 스레드 — 자료 구조

*Silberschatz, Figure 4.21 — Windows 스레드 자료 구조*

- ETHREAD, KTHREAD → **커널만 접근 가능** (커널 공간)
- TEB → **사용자 모드에서 접근 가능** (사용자 공간)
- 연결: ETHREAD → KTHREAD → TEB

```text
  커널 공간                              사용자 공간
  ┌─────────────┐                      ┌─────────────┐
  │  ETHREAD    │                      │    TEB      │
  │ - 시작 주소 │    ┌─────────────┐   │ - 스레드 ID │
  │ - 프로세스  │──→ │  KTHREAD    │──→│ - 사용자    │
  │   포인터    │    │ - 스케줄링  │   │   스택      │
  └─────────────┘    │ - 커널     │   │ - TLS 배열  │
                     │   스택     │   └─────────────┘
                     └─────────────┘
```

> **시험 팁:** "ETHREAD → KTHREAD → TEB"는 기억해야 할 핵심 연결 고리이다. ETHREAD와 KTHREAD는 커널 공간에(보호됨), TEB는 사용자 공간에(애플리케이션 접근 가능) 있다. TEB의 TLS 배열은 Windows가 OS 수준에서 스레드-로컬 저장소를 구현하는 방법이다.

### 9.3 Linux 스레드 — 태스크 개념

**Linux의 독특한 접근법**:
- 프로세스와 스레드를 **구분하지 않는다**
- 모든 실행 흐름을 **"태스크(task)"** 라 부른다
- 각 태스크는 자신만의 고유한 **struct task_struct**를 갖는다

**핵심**: `task_struct`는 데이터를 직접 저장하지 않고 다른 자료 구조에 대한 **포인터**를 포함한다

```text
  ┌─ task_struct ──────────────────────────┐
  │                                         │
  │  *files  ──→ [열린 파일 테이블]          │
  │  *signal ──→ [시그널 처리 정보]          │
  │  *mm     ──→ [메모리 디스크립터]          │
  │  *fs     ──→ [파일시스템 정보]            │
  │  pid, state, priority, ...              │
  │                                         │
  └─────────────────────────────────────────┘
```

> **핵심:** Linux의 설계는 우아할 정도로 단순하다: "프로세스"와 "스레드"는 모두 그저 `task_struct`이다. 둘의 차이는 **얼마나 공유하는가**이다. 새 프로세스(`fork()`)는 모든 것의 자체 복사본을 갖는다. 새 스레드(`clone()` + 공유 플래그)는 부모의 자료 구조에 대한 포인터를 공유한다. 이 통합 모델은 프로세스와 스레드 관리를 위한 별도의 코드 경로를 유지할 필요를 없앤다.

### 9.4 Linux — clone() 시스템 호출

**`clone()`**: Linux에서 태스크를 생성하는 시스템 호출
- **플래그**가 부모와 자식 간의 **공유 수준**을 결정한다

**사용 패턴**:

```text
fork() = clone(플래그 없음)
  → 아무것도 공유하지 않음 (완전한 복사)

thread = clone(CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND)
  → 거의 모든 것을 공유 (= 스레드 생성)
```

*Silberschatz, Figure 4.22 — clone() 플래그*

주요 clone 플래그:

| 플래그 | 공유 대상 |
|--------|----------|
| `CLONE_VM` | 가상 메모리 (주소 공간) |
| `CLONE_FS` | 파일시스템 정보 (루트 디렉터리, 현재 디렉터리) |
| `CLONE_FILES` | 열린 파일 디스크립터 |
| `CLONE_SIGHAND` | 시그널 핸들러 |

> Linux에서 `pthread_create()`가 호출되면 내부적으로 모든 공유 플래그가 설정된 `clone()`을 호출한다. 이것이 Linux에서 Pthreads가 주소 공간, 파일 디스크립터, 시그널 핸들러를 공유하는 스레드를 생성하는 이유이다 — 모두 `clone()` 플래그에 의해 제어된다.

### 9.5 Linux — fork() vs clone() 비교

**fork() 호출 시**: 새 태스크 + 부모의 모든 자료 구조를 **복사**

**clone() 호출 시 (공유 플래그 포함)**: 새 태스크 + 포인터를 통해 부모의 자료 구조를 **공유**

```text
  fork() — 공유 없음               clone() — 공유 플래그 있음
  ┌──────────┐  ┌──────────┐        ┌──────────┐  ┌──────────┐
  │ 부모     │  │ 자식     │        │ 부모     │  │ 자식     │
  │ task_    │  │ task_    │        │ task_    │  │ task_    │
  │ struct   │  │ struct   │        │ struct   │  │ struct   │
  │          │  │          │        │          │  │          │
  │ *mm ─→[A]│  │ *mm ─→[A']│      │ *mm ──┐  │  │ *mm ──┐  │
  │ *files→[B]│  │*files→[B']│      │ *files─┤  │  │*files─┤  │
  └──────────┘  └──────────┘        └───────┼──┘  └──────┼──┘
   서로 다른 복사본                         └──→ [공유됨] ←┘
                                           같은 자료 구조를 공유
```

**clone()의 확장 — 컨테이너**: 특정 플래그로 네임스페이스를 격리하면 컨테이너(Docker, LXC)를 생성할 수 있다 (Ch 18)

> **[시스템 프로그래밍]** Linux의 `clone()`은 다음의 기본 빌딩 블록이다:
> - **스레드**: 모든 것을 공유 (`CLONE_VM | CLONE_FILES | ...`)
> - **프로세스**: 아무것도 공유하지 않음 (일반 `fork()`)
> - **컨테이너**: 선택적 공유 + 네임스페이스 격리 (`CLONE_NEWNS`, `CLONE_NEWPID` 등)
>
> 이 스펙트럼 — 완전 격리(프로세스)에서 완전 공유(스레드)를 거쳐 선택적 격리(컨테이너)까지 — 모두 같은 시스템 호출로 제어된다. Linux의 가장 우아한 설계 결정 중 하나이다.

## 10. 실습 — OpenMP 병렬 프로그래밍

### 10.1 실습: OpenMP 행렬 곱셈 병렬화

```c
#include <omp.h>
#include <stdio.h>
#include <stdlib.h>
#include <time.h>
#define N 1000

double A[N][N], B[N][N], C[N][N];

void multiply_parallel(int num_threads) {
    omp_set_num_threads(num_threads);
    double start = omp_get_wtime();

    #pragma omp parallel for collapse(2)
    for (int i = 0; i < N; i++)
        for (int j = 0; j < N; j++) {
            C[i][j] = 0.0;
            for (int k = 0; k < N; k++)
                C[i][j] += A[i][k] * B[k][j];
        }

    double end = omp_get_wtime();
    printf("Threads: %d, Time: %.3f sec\n",
           num_threads, end - start);
}
```

컴파일: `gcc -fopenmp matrix_mul.c -o matrix_mul`

> **`collapse(2)`** 는 바깥 두 루프(`i`와 `j`)를 N*N = 1,000,000회 반복의 단일 반복 공간으로 합친 뒤 스레드 간에 분배한다. `collapse` 없이는 가장 바깥 루프(`i`)만 병렬화되어 1,000회 반복이 된다. 스레드당 반복 수가 많을수록 일반적으로 부하 분산이 개선된다.

> **참고**: 가장 안쪽 `k` 루프는 `C[i][j]`가 합계를 누적하므로 병렬화되지 **않는다** — 병렬화하려면 reduction이 필요하다. 바깥 루프들은 독립적이다(서로 다른 `(i,j)` 쌍이 서로 다른 `C[i][j]` 원소에 쓴다) — 동기화 없이도 안전하게 병렬화할 수 있다.

### 10.2 실습: 스레드 수별 성능 비교

```c
int main() {
    srand(time(NULL));
    for (int i = 0; i < N; i++)
        for (int j = 0; j < N; j++) {
            A[i][j] = (double)rand() / RAND_MAX;
            B[i][j] = (double)rand() / RAND_MAX;
        }

    multiply_parallel(1);
    multiply_parallel(2);
    multiply_parallel(4);
    multiply_parallel(8);
    return 0;
}
```

**실험에서 관찰할 사항**:
- 스레드 수가 증가함에 따른 실행 시간 변화
- 암달의 법칙과 실제 속도 향상의 비교
- 코어 수보다 많은 스레드를 생성할 때의 성능 영향
- 캐시 효과와 메모리 대역폭 병목

> **[컴퓨터구조]** 행렬 곱셈은 이론적 속도 향상과 실제 속도 향상 간의 격차를 관찰하는 고전적 벤치마크이다. 순수 병렬 워크로드(S ≈ 0)라 해도 선형 속도 향상을 볼 수 없는 이유:
> 1. **메모리 대역폭**: 모든 스레드가 같은 메모리 버스를 놓고 경쟁한다
> 2. **캐시 효과**: `B[k][j]` 접근 패턴의 공간 지역성이 나쁘다 (행 우선 배열에서 열 우선 접근)
> 3. **스레드 오버헤드**: 스레드 생성과 동기화에는 비용이 든다
>
> 물리적 코어 수보다 많은 스레드를 생성하면 컨텍스트 스위칭 오버헤드로 인해 성능이 오히려 **저하**될 수 있다.

### 10.3 실습: OpenMP reduction 실습

```c
#include <omp.h>
#include <stdio.h>
#define SIZE 10000000

int main() {
    double *arr = malloc(SIZE * sizeof(double));
    double sum = 0.0;

    // 배열 초기화
    for (int i = 0; i < SIZE; i++)
        arr[i] = 1.0 / (i + 1);

    // 순차 실행
    double start = omp_get_wtime();
    for (int i = 0; i < SIZE; i++)
        sum += arr[i];
    printf("Sequential sum: %.6f, Time: %.4f sec\n",
           sum, omp_get_wtime() - start);

    // 병렬 실행 (reduction)
    sum = 0.0;
    start = omp_get_wtime();
    #pragma omp parallel for reduction(+:sum)
    for (int i = 0; i < SIZE; i++)
        sum += arr[i];
    printf("Parallel   sum: %.6f, Time: %.4f sec\n",
           sum, omp_get_wtime() - start);

    free(arr);
    return 0;
}
```

> 이 실습은 조화급수의 부분합을 계산한다: 1/1 + 1/2 + 1/3 + ... + 1/10000000. 순차 버전과 병렬 버전 모두 **같은 결과**를 생성해야 한다 (또는 매우 근사한 값 — 부동소수점 덧셈은 완벽하게 결합법칙을 만족하지 않으므로 연산 순서에 따라 미세한 차이가 발생할 수 있다). 핵심 관찰 포인트는 **시간 차이**이다: 멀티코어 머신에서 병렬 버전이 상당히 빨라야 한다.

> **W04 실습과의 비교**: W04 실습에서는 배열을 Pthreads로 수동 분할하고, 경쟁 조건을 피하기 위해 `partial_sum[]` 배열을 사용했다. 여기서는 OpenMP의 `reduction` 절이 이 모든 것을 하나의 지시어로 자동 처리한다. 이것이 암묵적 스레딩의 힘이다 — Pthreads 코드로 약 50줄이 걸렸던 같은 병렬화가 OpenMP로는 1줄이면 된다.
